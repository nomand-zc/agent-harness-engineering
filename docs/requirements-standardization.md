# 模块1：需求标准化转化

## 1. 目标定位

本模块核心解决「口语化需求→可落地标准化需求文档」的转化问题，确保后续所有开发环节（架构设计、代码实现、验收测试）有一致、无歧义的需求基准。主要解决以下问题：

- 需求表述模糊，导致开发方向偏差；
- 需求边界不清，导致功能过度扩展或遗漏关键功能；
- 非功能指标缺失（性能、安全、兼容），导致上线后问题频发；
- 需求文档格式不统一，无法支撑多Agent协同执行。

## 2. 涉及的 Agent

| Agent | 角色 | 职责 |
|-------|------|------|
| Agent-0（需求校准Agent） | 执行主体 | 读取《需求校准指南》，将口语化需求转化为标准化PRD文档 |
| 人类（技术负责人/产品经理） | 触发方 + 校验方 | 提供口语化需求，校验并确认PRD文档，绑定需求ID |

## 3. 触发前置条件

- **手动触发**：人类输出口语化需求（文字形式），显式触发Agent-0；
- 前置不要求任何其他模块完成，是流程起点；
- 若为需求变更场景：变更幅度超过20%时需重新执行本模块。

## 4. 输入与输出

### 输入

| 输入项 | 来源 | 格式 |
|--------|------|------|
| 口语化需求描述 | 人类（文字/语音转文字） | 自由文本，如"开发一个用户登录接口，支持手机号+验证码登录，需校验验证码有效期，支持每秒100次并发" |
| 项目架构规范 | `docs/ARCHITECTURE.md` | Markdown |
| 现有需求产出物 | `docs/demand/` 目录 | Markdown |
| 现有代码库 | 项目根目录 | 源代码 |
| 《需求校准指南》 | `docs/guide/requirement-calibration-guide.md` | Markdown文档 |

### 输出

| 输出物 | 格式 | 归档路径 |
|--------|------|----------|
| 标准化PRD文档 | Markdown | `docs/demand/{需求ID}/PRD.md` |
| 《需求校准报告》 | Markdown | `docs/demand/{需求ID}/requirement-calibration-report.md` |
| 校准问题跟踪表 | Markdown | `docs/demand/{需求ID}/calibration-issues.md` |

> **校准问题跟踪表说明**：跟踪表按**轮次**组织，每次驳回重新校准时追加新轮次记录，不覆盖历史数据。它承担两个作用：①驱动每轮PRD修改（问题→修改→闭环）；②作为模块自我迭代的数据依据（跨需求的问题模式分析）。详细格式见《需求校准指南》第十二节。

### 产出物版本管理与保留规范

本模块所有产出物均遵循**追加不覆盖（Append-Only）** 原则，目的是保留完整的迭代过程记录，为后续模块对标与当前模块自我迭代优化提供可追溯的数据依据。

**关键原则**：

1. **版本映射不可丢失**：每个产出物版本必须与以下信息绑定：
   - 版本号（如 V1.0.0 / V1.0.1）
   - 生成/修改时间（`YYYY-MM-DD HH:mm:ss`）
   - 对应的 Git Commit（如 `abc1234`，代码提交后填写）
   - 对应的PRD版本

2. **历史记录永久保留**：
   - PRD 旧版本文件应保存至 `docs/demand/{需求ID}/history/PRD-V{版本号}.md`
   - 校准报告、问题跟踪表采用追加模式，通过轮次分隔保留历史
   - 所有产出物均建立"产出物版本对应矩阵"，作为查询索引

3. **跨轮次关联完整性**：
   - 校准报告的每一轮补充都需明确关联问题跟踪表的对应轮次
   - 问题编号（`R1-Q001` / `R2-Q001`）与报告中的澄清问题清单编号必须一致
   - 绝不允许因版本覆盖导致"问题→修改→闭环"的追溯链条断裂

**产出物版本对应矩阵模板**（应生成至 `docs/demand/{需求ID}/version-mapping.md`）：

```markdown
# 产出物版本对应矩阵

| 轮次 | 生成时间 | PRD版本 | Commit | 校准报告章节 | 问题跟踪轮次 | 状态 | 备注 |
|------|---------|--------|--------|------------|------------|------|------|
| Round-1 | 2026-03-30 10:00 | V1.0.0-draft | N/A | 一、二、三...七 | Round-1 | ✅ 已完成 | 首次校准 |
| Round-2 | 2026-03-31 14:30 | V1.0.1-draft | abc1234 | 第2轮校准补充 | Round-2 | ✅ 已完成 | 驳回后重校 |
```

### 与其他模块的标准化对接

- **下游（模块2）**：标准化PRD文档确认通过后，自动触发Agent-1启动架构设计；
- **接口约定**：PRD文档头部必须包含需求ID、版本号、生效时间等锚点信息（格式见第6节指南）。

## 5. 处理流程、校验标准与异常处理

### 5.1 操作步骤（共3步）

**第1步（人类执行）**：输出口语化需求，手动触发Agent-0。

> 示例：「开发一个用户登录接口，支持手机号+验证码登录，需校验验证码有效期，支持每秒100次并发」

**第2步（Agent-0执行）**：

1. 读取《需求校准指南》；
2. **强制执行前置探索**（不可跳过），分三层完成：

   **第一层——理解项目背景与现有功能**：
   - 阅读 `README.md` 或项目顶层文档，理解项目核心目标、使用场景和用户群体；
   - 阅读 `docs/overview.md`，理解 Agent Harness Engineering 全流程和模块分工；
   - 遍历 `docs/demand/` 下已有需求的PRD，识别与新需求的功能重叠、逻辑冲突、可复用资产；
   - 明确新需求在整体功能版图中的位置（全新功能、扩展现有功能还是与现有功能存在冲突）。

   **第二层——阅读项目规范文档**：
   - 阅读 `AGENTS.md`（项目索引与核心规范）；
   - 阅读 `docs/ARCHITECTURE.md`（设计约束、模块职责边界、禁止事项）；
   - 阅读 `docs/CONVENTIONS.md`（命名规范、架构原则）。

   **第三层——探索现有代码库**：
   - 搜索代码库中与新需求相关的现有实现；
   - 判断新需求与现有代码的关联关系，识别可复用模块；
   - 评估新需求的技术可行性风险。

   在校准报告中输出"仓库现状快照"（详见《需求校准指南》第二节）；

3. 拆解口语化需求，结合前置探索结论补充：
   - 业务边界（如"不支持第三方登录"）；
   - 异常场景（如"验证码错误/过期的提示"）；
   - 非功能指标（并发、安全、兼容要求）；
   - 与现有功能的关系说明（交集处理策略、依赖的现有模块）；
4. 执行需求完整性校验、合理性校验（含与现有功能的冲突/重复校验）；
5. 生成《需求校准报告》和标准化PRD文档。

**第3步（人类执行）**：

1. 校验标准化PRD文档（仅校验，不修改）；
2. 确认无偏差后，标注"需求确认通过"；
3. 绑定唯一需求ID（格式：`REQ-年份-序号`，例：`REQ-2026-001`）；
4. 上传至产出物仓库"需求文档"目录；
5. 触发下一步流程（模块2）。

### 5.2 校验标准（刚性约束，必须满足）

- 标准化PRD文档必须包含：需求ID、需求描述、业务边界、异常场景、非功能指标（性能/安全/兼容），缺一不可；
- 需求ID格式统一，无重复、无遗漏，与后续所有关联文档/代码绑定；
- 无模糊表述（如"并发尽量高"需明确为"每秒并发≥100次"）；
- PRD文档头部必须包含锚点字段（见指南第7节模板）；
- **《需求校准报告》中必须包含"仓库现状快照"章节**，明确说明与现有需求/功能的重叠、冲突或交集关系，不可缺失；
- **PRD文档中必须包含"与现有功能的关系"章节**，说明新需求与现有模块的边界及处理策略。

### 5.3 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| PRD文档不符合要求（缺字段/有模糊表述） | 人类手动驳回给Agent-0，附带具体修改建议，Agent-0重新生成，直至通过校验 |
| 需求存在重大缺陷（逻辑矛盾/技术不可行） | Agent-0在校准报告中标注"需重构"，人类重新梳理需求后再触发 |
| 需求变更（变更幅度<20%） | 人类直接修改PRD文档，绑定新版本号，同步更新关联文档 |
| 需求变更（变更幅度≥20%） | 重新走完整需求校准流程，生成新版PRD |

## 7. 《需求校准指南》编写规范

### 7.1 目标

本指南为Agent-0执行需求校准时的核心执行依据，确保所有需求均经过系统性的完整性与合理性校验，输出格式统一、无歧义的标准化PRD文档。

### 7.2 前置探索规范

本节为《需求校准指南》的核心前置步骤，Agent-0在执行任何校准动作之前必须完成以下三层探索。

**第一层——理解项目背景与现有功能**

- 阅读 `README.md` 或项目顶层文档，理解项目核心目标、使用场景和用户群体；
- 阅读 `docs/overview.md`，理解 Agent Harness Engineering 全流程和模块分工；
- 遍历 `docs/demand/` 下已有需求的PRD，识别与新需求的功能重叠、逻辑冲突、可复用资产；
- 明确新需求在整体功能版图中的位置（全新功能、扩展现有功能还是与现有功能存在冲突）。

**第二层——阅读项目规范文档**

| 文档 | 必读内容 |
|------|---------|
| `AGENTS.md` | 项目索引、模块分工、核心规范 |
| `docs/overview.md` | 全局架构、Agent角色分工 |
| `docs/ARCHITECTURE.md` | 设计约束、模块职责边界、禁止事项 |
| `docs/CONVENTIONS.md` | 命名规范、架构原则 |

**第三层——探索现有代码库**

- 搜索代码库中与新需求相关的现有实现；
- 判断新需求与现有代码的关联关系，识别可复用模块；
- 评估新需求的技术可行性风险（是否超出当前技术栈能力范围）。

> **强制约束**：跳过前置探索直接校准需求，等于在不了解全局的情况下评估局部，极易遗漏与现有功能的冲突和重叠，必须将探索结论记录在"仓库现状快照"章节中。

### 7.3 指南核心目录（必含模块，不可遗漏）

**1. 文档基础信息**

- 需求ID（格式：`REQ-年份-序号`）；
- 需求来源（人类姓名/角色）；
- 校准时间（格式：`YYYY-MM-DD`）；
- 校准Agent（`Agent-0`）。

**2. 原始需求摘要**

- 用户原始需求描述（完整引用，不可摘要）；
- 业务背景；
- 期望目标。

**3. 需求完整性校验**

检查需求是否涵盖以下要素（缺失项标注）：

- 业务目标与成功指标；
- 用户场景与使用流程；
- 功能边界（做什么/不做什么）；
- 非功能需求（性能、安全、兼容性）；
- 依赖与约束条件；
- 验收标准雏形。

**4. 需求合理性校验**

检查需求是否存在以下问题：

- 逻辑矛盾（同一需求内存在冲突表述）；
- 技术可行性风险（超出当前技术栈能力范围）；
- 资源匹配风险（人力、时间、预算是否充足）；
- 优先级冲突（与现有需求或规划存在冲突）。

**5. 需求澄清问题清单**

针对模糊、缺失、矛盾点，生成待确认问题列表（格式：问题编号 + 问题描述 + 期望答复方向）。

**6. 校准结论**

- **通过**：需求完整、清晰、可落地，可直接进入设计阶段；
- **需补充**：需求主体清晰但存在缺失项，需补充后进入设计阶段；
- **需重构**：需求存在重大缺陷，需重新梳理后再进入设计阶段。

**7. 标准化PRD输出模板**

校准通过后，按以下结构输出标准化PRD文档：

```markdown
---
需求ID: REQ-YYYY-NNN
版本号: V1.0.0
生效时间: YYYY-MM-DD
基准Commit: （代码提交后填写）
上版迭代号: 无
---

# 标准化PRD：{需求标题}

## 文档基础信息
- 需求ID：REQ-YYYY-NNN
- 版本号：V1.0.0
- 生效时间：YYYY-MM-DD
- 校准Agent：Agent-0

## 业务背景与目标
（业务背景描述，以及本需求需要达成的业务目标）

## 用户故事与场景描述
（用 "作为...，我希望...，以便..." 格式描述）

## 功能需求清单
（按模块/优先级组织，每条需求有唯一编号）

## 非功能需求
- 性能：（如"每秒并发≥100次，接口响应时间≤500ms"）
- 安全：（如"验证码有效期10分钟，支持防暴力破解"）
- 兼容性：（如"兼容iOS 14+、Android 10+"）
- 可观测性：（如"接口调用量、错误率须接入监控"）

## 边界定义
（明确不做的内容，如"不支持第三方登录"）

## 依赖与约束
（外部依赖、技术约束、时间约束等）

## 验收标准摘要
（核心验收条件，后续由Agent-1细化为完整验收标准）

## 附录
- 术语表
- 参考文档
```

### 7.3 校准触发时机

- **新需求接入时**：必须执行需求校准，确保需求可落地；
- **需求变更时**：变更幅度超过20%时，需重新执行校准；
- **定期巡检时**：每日巡检发现需求与实现偏差时，触发校准。

### 7.4 校准输出物

| 输出物 | 用途 |
|--------|------|
| 《需求校准报告》 | 按轮次记录校准过程、仓库探索快照、问题清单、校准结论；驳回重校时追加新轮次节，不覆盖历史（格式见《需求校准指南》第九节） |
| 《标准化PRD》 | 校准通过后输出的标准化需求文档，作为后续所有模块的输入基准 |
| 校准问题跟踪表 | 按轮次记录每次PRD修改迭代中的所有问题（含 Human 与 Agent-0 双方提出的问题，通过"提出方"字段区分）与闭环结论；兼作模块自我迭代的数据依据 |

## 8. 模块输出验收标准

本模块完成的验收标准（由人类或Agent-2评审时复核）：

- [ ] PRD文档存在于 `docs/demand/{需求ID}/PRD.md`；
- [ ] PRD文档头部包含所有必填锚点字段（需求ID、版本号、生效时间）；
- [ ] PRD文档包含所有必含模块（见7.2节），无缺失项；
- [ ] 所有需求描述均无模糊表述，非功能指标均已量化；
- [ ] 需求ID已绑定且格式符合规范（`REQ-年份-序号`）；
- [ ] 《需求校准报告》已归档至指定路径；
- [ ] 《需求校准报告》包含"仓库现状快照"章节，已说明与现有需求/功能的重叠、冲突或交集关系；
- [ ] PRD文档包含"与现有功能的关系"章节，已明确与现有模块的边界及处理策略；
- [ ] 校准问题跟踪表已归档至指定路径，包含完整的轮次记录（每次驳回重校均追加新轮次，不覆盖历史）；
- [ ] 跟踪表中 Human 与 Agent-0 双方提出的所有问题均已记录，每条问题的"提出方"字段已正确填写（`Human` 或 `Agent-0`）；
- [ ] 跟踪表中所有P0级问题已闭环，P1及P2问题已填写处置结论；
- [ ] 跟踪表顶部"轮次总览"与实际轮次节内容一致；
- [ ] 产出物版本对应矩阵已生成至 `docs/demand/{需求ID}/version-mapping.md`，完整记录各轮次的生成时间、版本号、Commit、对应报告与跟踪轮次；
- [ ] 若存在多轮迭代，旧版PRD文件已保存至 `docs/demand/{需求ID}/history/PRD-V{版本号}.md`，确保版本映射不丢失。
