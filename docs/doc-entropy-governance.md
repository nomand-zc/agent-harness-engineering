# 模块5：文档全链路版本联动与熵收敛治理

## 1. 目标定位

本模块核心解决「项目代码实现与架构设计文档脱节漂移」问题，通过双机制保障（Git Hook强卡点 + 专属巡检Agent兜底）实现代码Commit版本与设计文档版本的持续同步联动，收敛长期累积的熵增。主要解决以下问题：

- 代码实现变更后，关联的架构设计文档未同步更新，导致文档过时；
- 架构设计文档修改后，代码基于旧版本继续开发，新功能与设计不符；
- 设计文档与代码无版本锚点绑定（Commit Hash），无法追溯变更根因；
- 代码与架构文档偏差静默扩散，发现时整改成本极高。

> **与模块6区分**：本模块专注于"代码与架构文档的版本同步"，而模块6《指南与规约文档的自我迭代》专注于"规约体系版本管理与优化"。两者问题域不同，应各司其职。

## 2. 涉及的 Agent

| Agent/机制 | 角色 | 职责 |
|------------|------|------|
| Agent-6（文档熵收敛巡检Agent） | 执行主体 | 执行全量/增量文档熵收敛巡检，生成巡检报告，预警和处置代码与架构文档的版本偏差 |
| Git Hook（自动校验机制） | 自动门禁 | 在pre-push阶段自动触发 Agent-6 执行代码-文档版本一致性校验，阻断不合规推送 |
| 人类（技术负责人） | 规则制定 + 例外审批 | 制定校验规则，审批例外申请，复核致命/严重问题 |

> 规约指南的维护与迭代职责由 Agent-5（规约指南Agent）承担，归属于模块6《指南与规约文档的自我迭代》，本模块不涉及。

## 3. 触发前置条件

本模块是长效保障机制，贯穿全流程，触发时机包括：

- **pre-push触发（主要卡点）**：代码或文档推送至分支时，Git Hook 自动触发 Agent-6 执行全链路代码-文档版本一致性校验，不合规则阻断推送；
- **主干合并触发**：文件合并至主干时，触发 Agent-6 增量联动校验；
- **定时触发**：每日22:00，Agent-6 触发全量基线巡检。

### 产出物版本管理与保留规范

本模块产出物遵循**全量保留、按日期归档** 原则——巡检报告是驱动代码与文档持续对齐的核心数据，同时也是模块6指南自我迭代的重要依据，任何历史巡检报告均不得删除或覆盖。

**关键原则**：

1. **巡检报告永久保留（禁止覆盖）**：
   - 每次巡检输出的报告均作为独立文件归档，文件名包含日期和类型（全量/增量），天然防止覆盖
   - 同一天多次巡检需在文件名中加序号（如 `报告-20260330-增量-001.md`、`报告-20260330-增量-002.md`）
   - 任何历史巡检报告均不得删除，是全流程问题模式分析的原始数据

2. **多轮整改过程完整保留**：
   - 当巡检发现问题→对应Agent整改→再次巡检验证，多轮整改的每次巡检报告均须保留
   - 报告中须标注"本次巡检关联的上次报告路径"，形成可追溯的整改闭环链条
   - 致命/严重问题从发现到闭环的完整过程必须可通过报告序列完整还原

3. **例外审批记录永久留痕**：
   - 人类的每次例外审批记录须归档至 `docs/demand/inspection/exceptions/exception-YYYYMMDD-{需求ID}.md`
   - 记录包含：申请时间、申请原因、关联产出物、批准人、有效期
   - 例外审批记录是审计留痕的核心证据，不可删除

4. **版本映射表的变更历史**：
   - `版本映射表.xlsx` 每次更新须同步提交至版本控制，通过 Git 历史保留变更记录
   - 建议每次更新时在 Commit message 中注明更新原因（如 `【版本映射表】新增REQ-2026-002产出物绑定`）

### 与其他模块的标准化对接

- **跨模块**：本模块不依赖单一上游，监控所有模块产出物的一致性；
- **输出使用方**：所有Agent和人类均可查看巡检报告，对应Agent负责整改；
- **对模块6的输入**：历史巡检报告中的高频问题模式是模块6指南自我迭代的重要依据，须永久保留。

## 5. 前置条件与前置探索

### 5.0 Agent-6执行前置探索

在执行全量/增量文档巡检之前，Agent-6必须完成以下项目探索，建立对全链路产出物的完整认知：

1. **理解项目背景与全链路工程流程**：
   - 阅读 `README.md` 和 `docs/overview.md`，理解项目核心目标和Agent Harness Engineering的全流程；
   - 梳理 `docs/demand/` 下的全部历史需求及产出物，了解文档版本映射规律；
   - 理解需求→设计→代码→验收的完整链路及各环节的产出物关系。

2. **阅读项目规范文档与治理指南**：`docs/ARCHITECTURE.md`（架构约束）、《文档熵收敛巡检检查指南》（巡检规则）、版本映射表（版本绑定规则）；

3. **分析全链路产出物与代码的锚点关系**：
   - 检查已有需求的PRD、设计文档、验收标准、验收报告的绑定关系；
   - 梳理Commit Hash与架构版本的对应关系；
   - 识别已有的版本映射规律和可能的偏差点。

4. 将探索结论作为后续巡检的基线参考。

## 6. 核心落地机制

### 6.1 机制1：Git Hook（刚性拦截）

**触发时机**：任何Agent/人类修改代码或设计文档（架构设计方案、PRD、验收标准）后执行推送/合并时触发。

**校验内容**：当前修改文件绑定的需求ID、基准Commit，是否与全链路关联文档中的版本锚点一致；代码实现是否与关联的架构设计文档描述吻合。

**处置逻辑**：若发现关联文档未同步、Commit锚点不一致 → 直接阻断推送，弹窗提示"需同步修改关联文档（例：修改核心模块代码后，需同步更新架构设计方案V1.0.1）"，强制完成对齐后，方可推送。

### 6.2 机制2：Agent-6兜底巡检（文档熵收敛巡检Agent）

**触发方式**：
- Git pre-push 钩子自动触发（主要卡点）：代码或文档推送时，Agent-6 执行全链路一致性校验，不合规则阻断推送；
- 每日固定时间（如22:00）执行全量扫描；
- 任何文档/代码合并至主干时，自动触发增量扫描。

**校验内容**：严格遵循《文档熵收敛巡检检查指南》，校验全链路产出物的锚点一致性、版本递进、变更联动。

**处置逻辑**：生成《文档熵偏离巡检报告》，标红差异项、关联责任人（对应Agent）、给出具体修改建议；致命/严重项自动预警，进入人工复核池；一般/优化项仅预警，纳入下次迭代整改。

## 7. 巡检核心校验标准

### 7.1 刚性校验基线（必须满足，否则触发预警/拦截）

- **ID唯一绑定规则**：所有产出物（文档/代码/测试用例）必须绑定1个有效需求ID，禁止无ID、多ID、过期ID、伪造ID；
- **Commit锚点规则**：设计文档标注的基准Commit，必须与主干当前生效Commit一致；代码内注释、接口契约，必须能回溯到对应架构文档段落；
- **代码-文档变更联动规则**：修改以下任意一项，全链路关联的架构设计文档必须同步复核修改：
  - 核心业务逻辑调整；
  - 数据结构/接口字段变更；
  - 非功能指标（性能、容量等）调整；
  - 模块边界与职责变更。

### 7.2 问题分级标准

| 问题级别 | 判定标准 | 整改时限 | 责任主体 |
|----------|----------|----------|----------|
| 致命 | 需求ID错位、核心变更未同步、Commit锚点篡改 | 24小时内 | 对应Agent + 人类复核 |
| 严重 | Commit锚点不一致、关联架构文档漏更新 | 48小时内 | 对应执行Agent |
| 一般 | 备注/说明类文案不同步 | 下次迭代前 | 对应执行Agent |
| 优化 | 格式、索引、引用排版不规范 | 无强制时限 | 对应执行Agent |


### 7.3 触发校验时机规范

| 触发时机 | 校验范围 | 校验类型 |
|----------|----------|----------|
| pre-push（主要卡点） | 当前代码/文档及所有关联设计文档 | 代码-文档 Commit 锚点一致性校验 |
| 主干合并 | 变更部分及关联文档 | 增量联动校验 |
| 每日22:00 | 所有产出物 | 全量基线巡检 |

### 7.4 逐条校验规则明细（Agent可直接解析）

| 规则编号 | 规则内容 | 问题分级 |
|----------|----------|----------|
| 规则001 | 所有产出物必须绑定唯一有效需求ID，无ID、多ID、过期ID、伪造ID，均判定为致命问题 | 致命 |
| 规则002 | 设计文档标注的基准Commit，与主干当前生效Commit不一致 | 严重 |
| 规则003 | 修改核心业务逻辑、数据结构/接口字段、非功能指标、模块边界后，未同步修改关联架构设计文档 | 致命 |
| 规则004 | 代码内注释、接口契约，无法回溯到对应架构文档段落 | 一般 |
| 规则005 | 文档格式、索引、引用排版不规范（如字段缺失、排版混乱） | 优化 |
| 规则006 | 版本映射表未及时更新（新增需求产物后未同步） | 一般 |

### 7.5 问题分级与处置流程

| 问题分级 | 处置方式 | 整改时限 | 责任主体 |
|----------|----------|----------|----------|
| 致命 | 阻断提交/合并，自动预警，进入人工复核池，强制整改 | 24小时内 | 对应执行Agent + 人类复核 |
| 严重 | 预警提示，禁止基于旧文档新增功能，限期整改 | 48小时内 | 对应执行Agent |
| 一般 | 仅预警，不阻断，纳入下次迭代统一整改 | 下次迭代前 | 对应执行Agent |
| 优化 | 仅预警，不阻断，按需整改 | 无强制时限 | 对应执行Agent |

### 7.6 巡检报告输出规范

**6.1 报告模板（固定格式，不可修改）**：

- 标题：`《文档熵收敛巡检报告-巡检时间-需求ID（全量/增量）》`（例：`《文档熵收敛巡检报告-20260328-全量》`）；
- 核心内容：
  1. 巡检概况（巡检范围、巡检时长、异常项总数）；
  2. 差异清单（问题分级、问题描述、关联产出物路径、关联需求ID）；
  3. 整改建议（具体修改操作、责任Agent）；
  4. 整改时限。

**6.2 报告归档**：巡检报告绑定对应需求ID（全量报告绑定"全量"标识），上传至产出物仓库"巡检报告"目录，可追溯、可查询；

**6.3 预警方式**：致命/严重问题通过系统弹窗+邮件通知人类，一般/优化问题仅在报告中标注。

### 7.7 例外审批规则（仅人类可操作）

**7.1 例外场景**（仅以下场景可申请例外，其余场景不可豁免）：

- 极小文案优化（如修改文档错别字），不影响核心内容与关联关系；
- 非关联注释调整（如代码内注释优化，不涉及逻辑与接口）。

**7.2 审批流程**：人类提交例外申请（注明申请理由、关联产出物、修改内容），记录豁免记录，留痕审计，豁免有效期最长为7天，过期自动失效。

## 8. 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| 巡检发现致命问题 | Agent-6自动阻断相关推送/合并，预警推送至企微群，对应Agent在24小时内整改，人类复核通过后解除阻断 |
| 关联文档同步不及时 | Agent-6自动提醒对应Agent，逾期未整改的升级为严重问题，触发人类介入 |
| 例外申请审批 | 人类需在24小时内完成审批，审批结果同步至Agent-6，纳入巡检记录 |

> 规约指南文档的修正与迭代，由模块6《指南与规约文档的自我迭代》负责，本模块不涉及。

## 9. 模块输出验收标准

本模块持续运行的验收标准：

- [ ] Git Hook已配置并生效（pre-push阶段可自动触发代码-文档一致性校验）；
- [ ] Agent-6已配置定时触发（每日22:00），全量巡检正常运行；
- [ ] 巡检报告按规范格式归档，无悬空链接；
- [ ] 致命/严重问题均在规定时限内完成整改；
- [ ] 巡检报告均以独立文件归档（不覆盖历史报告），同日多次巡检以序号区分（如 `-001`、`-002`）；
- [ ] 多轮整改场景中，每次巡检报告已标注关联的上次报告路径，整改闭环链条可追溯；
- [ ] 所有例外审批记录已归档至 `docs/demand/inspection/exceptions/`，包含申请原因、批准人、有效期；
- [ ] 版本映射表每次更新均已提交至版本控制，Commit message注明更新原因，历史变更可追溯。
