# 模块7：统一交互入口集成方案

## 1. 目标定位

本模块核心解决「多场景下人类与Agent的统一交互入口」问题，以**企微群 + 数字人员工**为核心载体。各 Agent 始终以数字人员工身份常驻群内，人可随时 @单个 Agent 执行单步任务；当各 Agent 输出质量稳定达标后，可新增 Workflow-Agent 入群，直接 @Workflow-Agent 触发全流程自动化。两种方式长期并存，使用哪种完全取决于个人。主要解决以下问题：

- 需求提交入口分散，无统一渠道接入Agent流程；
- 口语化会议需求难以直接转化为标准化输入，人工整理成本高；
- 前期规约指南质量不稳定，需要人类深度Review与持续修正；
- 多Agent协作流程割裂，缺乏统一调度与全自动化串联能力；
- 审批事项推送不及时，流程阻塞感知滞后。

## 2. 核心交互模式

### 2.1 方式A：@单个 Agent（单步精细控制）

各个 Agent 始终以**数字人员工**的身份常驻企微群，每个 Agent 对应一个独立的企微账号/机器人身份，人通过 @Agent 的方式触发对应的单步任务。此方式长期可用，适合需要深度介入和逐步Review的场景。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    企微群（方式A：@单个Agent单步执行）                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   群成员（数字人员工，长期常驻）：                                   │
│   @Agent-0（需求校准）  @Agent-1（方案设计）  @Agent-2（方案评审）  │
│   @Agent-3（对齐确认）  @Agent-4（验收）      @Agent-5（规约指南）  │
│   @安全评审  @性能评审  @合规评审（专项评审，按需触发）             │
│   @Agent-6（文档熵巡检，pre-push自动触发，无需手动@）              │
│   @Agent-7（Backlog整理，人类手动/事件驱动/定时触发）              │
│   @会议助手（腾讯会议预约+纪要整理+需求转化）                       │
│   @Workflow-Agent（全自动化，质量达标后新增入群）                    │
│                                                                     │
│   交互方式：                                                        │
│   人类 → @Agent-X "执行任务描述" → Agent产出物发回群               │
│   人类 → 深度Review产出物 → 符合预期✓ / 不符合预期✗               │
│                            ↓                                        │
│                    @Agent-5（规约指南Agent）                        │
│                    "请根据以下反馈修改指南：..."                     │
│                            ↓                                        │
│                    指南更新 → 下次执行质量提升                       │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 方式B：@Workflow-Agent（全流程自动化）

当各 Agent 的单次输出产物质量稳定达到预期后，可将 Workflow-Agent 新增入群。此后人可直接 @Workflow-Agent 触发全流程，Workflow-Agent 自动按序调度各 Agent 完成任务。**原有各 Agent 数字人员工不会退出，两种方式长期并存，使用哪种完全取决于个人。**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    企微群（方式B：@Workflow-Agent全流程）             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   人类 → @Workflow-Agent "新需求：xxx"                              │
│              ↓                                                      │
│   Workflow-Agent 自动编排：                                         │
│   Agent-0 → Agent-1 → Agent-2 → Agent-3 → Agent-4                 │
│              ↓（每个节点产出物自动推送群内）                        │
│   人类按需介入 @某Agent 进行局部调整（仍然有效）                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 4. 初期建设阶段（人类深度介入与指南收敛）

### 4.1 核心逻辑

初期建设阶段的核心目标是**通过人类深度Review + 规约指南持续修正，将各Agent单次输出质量提升至稳定预期水平**。此阶段不影响原有@单个Agent的方式继续可用，两种方式长期并存。

```
人类 @Agent-X 执行任务
        ↓
   Agent产出物发回群内
        ↓
   人类深度Review产出物
        ↓
┌───────────────────────┐
│  符合预期？           │
│  是 → 流转下一步      │
│  否 → @Agent-5        │
│       "根据以下反馈   │
│        修改规约指南：  │
│        [具体问题]"    │
└───────────────────────┘
        ↓
   Agent-5更新指南文档
        ↓
   下次该Agent执行质量提升
```

### 4.2 各 Agent 触发方式

| Agent | 企微身份 | 触发示例 |
|-------|----------|----------|
| Agent-0（需求校准） | @需求校准 | `@需求校准 请对以下需求进行标准化：[需求描述]` |
| Agent-1（方案设计） | @方案设计 | `@方案设计 请根据 REQ-2026-018 输出架构设计方案` |
| Agent-2（方案评审） | @方案评审 | `@方案评审 请对 REQ-2026-018 的设计方案进行评审` |
| Agent-3（对齐确认） | @对齐确认 | `@对齐确认 请对 REQ-2026-018 进行开发对齐确认` |
| Agent-4（验收） | @验收 | `@验收 请对 REQ-2026-018 执行验收流程` |
| 安全专家Agent | @安全评审 | `@安全评审 请对 REQ-2026-018 的设计方案执行安全专项评审` |
| 性能专家Agent | @性能评审 | `@性能评审 请对 REQ-2026-018 的设计方案执行性能专项评审` |
| 合规专家Agent | @合规评审 | `@合规评审 请对 REQ-2026-018 的设计方案执行合规专项评审` |
| 会议助手Agent | @会议助手 | `@会议助手 请预约明天14:00需求评审会，参会人：[人员列表]` |
| Agent-5（规约指南） | @规约指南 | `@规约指南 请根据以下反馈修改[指南名称]：[具体问题]` |
| Agent-7（Backlog整理） | @Backlog整理 | `@Backlog整理 请整理当前 Backlog，输出排期建议` |

### 4.3 规约指南迭代闭环

```
人类Review发现问题
        ↓
@规约指南 "问题描述 + 期望改进方向"
        ↓
Agent-5 分析问题 → 更新对应指南文档
        ↓
Agent-5 将变更内容发回群内确认
        ↓
人类确认 → 指南正式生效
        ↓
后续该Agent执行时加载新指南
```

**质量达标判断标准**：连续 N 次（建议≥3次）执行，产出物无需指南修正，则认为该 Agent 的输出质量已稳定达标。

## 5. 会议助手 Agent

### 5.1 角色定位

会议助手 Agent 以数字人员工身份常驻企微群，作为**需求收集的前置入口**，负责将口语化的会议讨论内容自动转化为可直接输入 Agent-0 的标准化需求草稿，打通「会议室 → 企微群 → Agent流程」的全链路。

### 5.2 核心职责

| 职责 | 说明 |
|------|------|
| 预约腾讯会议 | 根据群内 @ 指令，调用腾讯会议 API 创建会议，自动生成会议链接并推送至群内 |
| 会议纪要记录 | 接入腾讯会议录音/实时字幕流，自动生成结构化会议纪要（含决策项、待办项、遗留问题） |
| 需求转化输出 | 从会议纪要中提取需求相关内容，整理为符合 Agent-0 输入格式的需求草稿，直接推送群内供人类确认后流转 |

### 5.3 工作流程

```
人类 @会议助手 预约会议
        ↓
会议助手 调用腾讯会议API → 创建会议室 → 群内推送会议链接
        ↓
   会议进行中（自动录制/字幕）
        ↓
会议结束 → 会议助手 自动拉取录音/字幕
        ↓
   生成结构化会议纪要
   ├── 会议基本信息（时间、参会人、议题）
   ├── 讨论要点摘要
   ├── 决策事项列表
   ├── 待办事项（含责任人与 deadline）
   └── 遗留问题
        ↓
   提取需求相关内容
        ↓
   整理为 Agent-0 输入格式的需求草稿
   （含：需求背景、核心诉求、约束条件、验收预期）
        ↓
   推送至群内：
   「【会议助手】本次会议纪要已生成，以下为提取的需求草稿，
     请确认后 @需求校准 进行标准化处理」
        ↓
人类确认草稿 → @需求校准 启动 Agent-0 流程
```

### 5.4 触发示例

```
# 预约会议
人类：@会议助手 请预约明天下午2点需求评审会，参会人：张三、李四、王五，预计1小时

会议助手：【会议助手】已创建腾讯会议
  📅 会议主题：需求评审会
  🕑 时间：2026-03-31 14:00-15:00
  👥 参会人：张三、李四、王五
  🔗 会议链接：https://meeting.tencent.com/xxx
  📝 会议结束后将自动生成纪要

# 会议结束后自动输出
会议助手：【会议助手】需求评审会纪要已生成

  📋 会议纪要摘要：
  - 决策：用户登录模块需支持微信扫码登录
  - 待办：李四负责整理详细需求，本周五前完成
  - 遗留：第三方登录安全审计需进一步讨论

  📝 需求草稿（可直接提交 Agent-0）：
  ---
  需求背景：现有登录方式单一，用户体验较差
  核心诉求：支持微信扫码登录，降低登录门槛
  约束条件：需通过安全审计，兼容现有账号体系
  验收预期：登录成功率≥99.5%，扫码响应时间≤3s
  ---
  @需求校准 请对以上需求草稿进行标准化处理
```

### 5.5 与 Agent-0 的衔接规范

会议助手输出的需求草稿需满足以下格式，确保 Agent-0 可直接处理：

| 字段 | 说明 | 是否必填 |
|------|------|----------|
| 需求背景 | 来源于会议讨论的问题或痛点描述 | 必填 |
| 核心诉求 | 本次需求的核心目标，一句话描述 | 必填 |
| 约束条件 | 技术、合规、时间等限制 | 选填 |
| 验收预期 | 可量化的成功标准（来自会议决策项） | 选填 |
| 会议纪要链接 | 原始纪要文档链接，供 Agent-0 参考原始上下文 | 必填 |

人类确认需求草稿无误后，直接 `@需求校准 [草稿内容]` 启动 Agent-0 标准化流程。

## 6. Workflow-Agent 新增阶段

### 5.1 新增条件

Workflow-Agent 新增入群的触发条件（两种方式并存后，原有各 Agent 数字人员工不受影响）：

- [ ] Agent-0 至 Agent-4 每个 Agent 单次输出质量均已稳定达标（连续≥3次无需指南修正）；
- [ ] 各 Agent 对应的指南文档已通过人工Review确认稳定；
- [ ] Workflow-Agent 已完成各 Agent 串联逻辑的配置。

### 5.2 Workflow-Agent 职责

Workflow-Agent 负责全流程的自动化编排，接收人类触发指令后，按序调度各 Agent 执行，并将关键节点产出物推送至企微群，供人类按需查阅与介入。

| 职责 | 说明 |
|------|------|
| 全流程编排 | 按 Agent-0 → Agent-1 → Agent-2 → Agent-3 → Agent-4 顺序调度 |
| 节点产物推送 | 每个 Agent 执行完毕后，将产出物摘要推送至群内 |
| 异常感知 | 任意节点执行异常时，@人类审批人并暂停流程 |
| 局部重跑 | 支持 @Workflow-Agent 从指定节点重新执行 |

### 5.3 触发示例

```
人类：@Workflow-Agent 新需求：用户登录优化，详见 REQ-2026-020

Workflow-Agent：收到，开始全流程执行 REQ-2026-020
  ▶ Step 1/5：Agent-0 需求校准中...
  ✅ Step 1/5：需求校准完成 → [查看产物]
  ▶ Step 2/5：Agent-1 方案设计中...
  ✅ Step 2/5：方案设计完成 → [查看产物]
  ...
  ✅ 全流程完成，REQ-2026-020 验收通过
```

## 7. 分阶段实施路径

| 阶段 | 实施内容 | 启动条件 | 预期成果 |
|------|----------|----------|----------|
| 第一阶段 | 各Agent以数字人员工形式入企微群，人工@触发执行（方式A上线） | 完成Agent基础开发 | 建立人机协作基础交互链路 |
| 第二阶段 | 人类深度Review产出物，@规约指南Agent持续修正指南 | 第一阶段完成 | 各Agent指南文档趋于稳定 |
| 第三阶段 | 新增Workflow-Agent入群（方式B上线），两种方式并存 | 各Agent单次输出质量稳定达标 | 全流程自动化可选，灵活度最高 |
| 第四阶段 | 新增会议助手Agent入群，支持腾讯会议预约、纪要生成与需求转化 | 第三阶段稳定运行，腾讯会议API接入完成 | 需求收集全自动化，会议纪要→需求草稿闭环打通 |

## 8. 企微群配置规范

### 7.1 群成员配置

| 成员类型 | 说明 |
|----------|------|
| 人类成员 | 需求方、架构师、审批人等相关干系人 |
| 数字人员工（核心链路） | Agent-0 至 Agent-5，每个 Agent 对应独立企微账号，长期常驻群内 |
| 数字人员工（专项评审） | 安全专家Agent（@安全评审）、性能专家Agent（@性能评审）、合规专家Agent（@合规评审），按需入群，人类手动@触发；三类可合并为一个账号（@专项评审）用于规模较小的项目 |
| Agent-6（文档熵巡检） | 由 Git pre-push 钩子自动触发，不占用企微群交互席位，无需入群 |
| Agent-7（Backlog整理） | 长期常驻群内，支持人类手动 @ 触发、需求验收完成后事件驱动触发、每周一定时触发三种方式；汇总整理跟进事项，推送整理报告与排期建议 |
| 会议助手Agent | 腾讯会议API接入后新增入群，负责会议预约、纪要生成与需求草稿转化；长期常驻 |
| Workflow-Agent | 各Agent质量稳定达标后新增入群，负责全流程自动化调度；与数字人员工模式长期并存 |

### 7.2 群消息规范

- Agent 每次响应需在消息头部标注身份，如：`【需求校准 Agent-0】`；
- 产出物过长时，Agent 推送摘要 + 文档链接，完整产物存储至 `docs/demand/` 目录；
- 规约指南变更需 @所有人 通知，并附变更摘要。

## 9. 成功关键因素

- **人类Review深度**：初期建设阶段需要架构师深度参与产物Review与指南修正，确保规约收敛质量；
- **规约迭代耐心**：指南稳定需要多轮迭代，但不影响原有@单个Agent方式的继续使用；
- **质量达标标准**：明确定义"稳定达标"的量化判断标准，达标后可选择新增Workflow-Agent作为补充方案，但@单个Agent方式始终保留；
- **Workflow配置严谨**：若选择引入Workflow-Agent，其全自动化串联逻辑需经过充分测试，避免流程中断。

## 10. 风险与应对

| 风险点 | 风险描述 | 应对措施 |
|--------|----------|----------|
| 指南质量不稳定 | Agent生成的指南初稿质量参差不齐 | 强化人工Review环节，@规约指南Agent持续修正 |
| 过早引入Workflow-Agent | 单个Agent输出质量未达标就新增Workflow-Agent，导致全流程质量问题 | 严格执行质量达标判断标准，达标后方可选择新增Workflow-Agent |
| 数字人员工身份混淆 | 多Agent同时响应，群内消息混乱 | 严格消息头标注规范，每次响应标明Agent身份 |
| 自主进化失控 | 规约指南Agent提案方向偏离 | 所有指南变更需人类确认后方可生效 |
| 会议纪要质量不稳定 | 录音质量差或字幕识别率低导致纪要失真 | 会议结束后人类核查纪要关键内容，确认后方可流转需求草稿 |
| 需求草稿过度发散 | 会议讨论范围广，助手提取的需求粒度不当 | 明确草稿字段格式规范，人类确认草稿后再 @需求校准 触发 Agent-0 |
