# 模块6：指南与规约文档的自我迭代（Agent自主学习进化）

## 1. 目标定位

本模块核心解决「规约体系随项目演进的动态优化」问题，通过Agent自主学习进化机制和规约减重/增重双向机制，实现指南文档的持续迭代优化，使规约体系保持"足够精准、不过度冗余"的最佳状态。主要解决以下问题：

- 指南文档编写门槛高，团队缺乏经验难以快速建立规约体系；
- 规约随项目推进逐渐失效或冗余，但无机制精简；
- 需求交付质量不足时，无体系化路径将问题根因转化为规约改进；
- 规约变更缺乏控制，改动失控导致执行混乱。

## 2. 涉及的 Agent

| Agent | 角色 | 职责 |
|-------|------|------|
| 各执行Agent（Agent-0至Agent-4） | 提案发起方 | 在执行需求任务过程中识别规约优化点，发起迭代提案 |
| Agent-5（规约指南Agent） | 规约迭代 + 巡检监控 | **双重职责**：① 接收人类Review反馈，分析根因后修正对应指南文档，变更需人类确认后生效；② 定期巡检发现高频问题，触发增重提案建议；校验slim-index.md一致性 |
| 人类（技术负责人/指南Owner） | 审批决策方 | 审批所有规约变更提案，执行减重/增重操作，最终确认 |

## 3. 触发前置条件

- **增重触发信号**（满足任意一项即可发起提案）：
  - 验收报告或评审文档中出现致命/严重问题，且根因可归因于规约未覆盖；
  - 同一类型的人工介入在 ≥3 个需求中重复出现；
  - Agent主动识别规约歧义或覆盖盲区；
  - 项目引入新技术栈、新合规要求或新业务类型。

- **减重触发信号**（须同时满足三维评估标准）：
  - 行为稳定性：Agent在连续 ≥10 个需求中均自主达标；
  - 泛化覆盖性：达标表现覆盖规则的全部约束场景；
  - 失效可感知：偏差可被现有校验体系及时捕获。

## 4. 输入与输出

### 输入

| 输入项 | 来源 | 说明 |
|--------|------|------|
| 验收报告/评审文档/巡检报告 | 各模块产出物 | 作为增重提案的佐证材料 |
| 历史需求执行记录 | 全链路产出物 | 作为减重提案的稳定性证明 |
| 各类指南文档（当前版本） | `docs/guide/` 目录 | 待迭代的目标文档 |

### 输出

| 输出物 | 格式 | 归档路径 |
|--------|------|----------|
| 减重提案文件 | Markdown | `docs/slim/proposals/{提案ID}.md` |
| 增重提案文件 | Markdown | `docs/slim/proposals/{提案ID}.md` |
| 归档规则文件（减重） | Markdown | `docs/slim/archived-rules/{提案ID}-rule.md` |
| 生效规则快照（增重） | Markdown | `docs/slim/enacted-rules/{提案ID}-rule.md` |
| 规约提案统一索引 | Markdown | `docs/slim/slim-index.md` |
| 更新后的指南文档 | Markdown | `docs/guide/{指南名}.md` |

### 与其他模块的标准化对接

- 本模块的产出（更新后的指南文档）直接影响所有其他模块的执行依据；
- 本模块自行管理 slim-index.md 的一致性校验，Git Hook（pre-commit 阶段）自动执行 S-001 至 S-006 规则，阻断不合规提交；Agent-5 在巡检时做兜底校验；
- 所有提案提交时，须同步更新 slim-index.md，并触发 S-001 至 S-006 校验；
- 模块5《文档全链路版本联动与熵收敛治理》专注于代码与架构文档的版本同步，不介入本模块的规约索引校验。

## 5. 处理流程、校验标准与异常处理

### 5.1 指南文档生成策略

#### 策略一：Agent生成 + 人工Review模式

适用场景：团队有一定文档规范经验，可快速建立指南框架。

```
人类提供指南目标与约束 → Agent生成初稿（基于模板） → 人工Review修改（优化细节）
        ↑                                                        ↓
  持续迭代优化（版本递进） ← 试行验证反馈（实战检验） ← 版本发布归档（正式生效）
```

执行步骤：

1. **需求分析**：人类明确指南的目标Agent、适用场景、核心约束条件；
2. **模板选择**：基于各模块的指南结构模板，选择合适的文档框架；
3. **Agent生成**：提供详细的Prompt，让Agent生成符合结构要求的初稿；
4. **人工Review**：检查内容的准确性、完整性、可执行性，补充业务细节；
5. **试行验证**：在1-2个实际需求中试用，收集问题与改进点；
6. **版本发布**：修正问题后发布正式版本，纳入指南文档库。

质量保障规则：

- 每份指南必须经过至少2个实际需求的试行验证；
- Review修改记录必须保留，作为指南迭代的历史依据；
- 指南版本号遵循语义化版本规范，迭代历史可追溯。

#### 策略二：反向生成策略（TUI交互先行）

适用场景：团队缺乏编写指南经验，推荐"先实践、后提炼"的反向生成策略。

```
TUI交互式执行完整需求（Agent-0→Agent-1→...→Agent-4）
                        ↓
              收集所有中间产物文档
                        ↓
   分析文档结构/内容模式/决策逻辑（识别通用模式、规范格式）
                        ↓
              反向生成对应指南初稿
                        ↓
              人工优化 → 正式发布
```

| 对比维度 | 传统编写方式 | 反向生成方式 |
|----------|--------------|--------------|
| 启动成本 | 高（需提前设计完整框架） | 低（边做边总结） |
| 适用场景 | 团队有成熟的文档规范 | 团队缺乏文档编写经验 |
| 准确性 | 取决于设计者的经验 | 基于实际产出物，更贴近实战 |
| 迭代效率 | 需人工维护更新 | 可持续从新需求中学习优化 |

### 5.2 Agent自主学习进化机制

#### 自我反向迭代原理

```
指南文档（当前版本） ──执行任务──> Agent执行（遵循指南）
        ↑                                    ↓
指南文档（迭代版本） <──经验提取── 产出物 & 执行日志
        ↓                    ↑
  版本发布（正式生效） <──反馈优化── 问题分析 & 改进建议
```

#### 进化触发场景

| 触发场景 | 触发条件 | 学习内容 | 迭代目标 |
|----------|----------|----------|----------|
| 问题修复 | 执行过程中发现指南规则不合理 | 调整规则表述、补充例外情况 | 提升指南可执行性 |
| 新模式发现 | 多个需求中出现相似的处理模式 | 提取通用模式，补充到指南 | 扩展指南覆盖范围 |
| 效率优化 | 发现可简化的冗余步骤 | 优化流程、合并步骤 | 提升执行效率 |
| 错误预防 | 出现指南未覆盖的错误场景 | 补充错误预防规则 | 降低错误率 |
| 最佳实践 | 人类Review时给出优秀建议 | 纳入最佳实践章节 | 提升产出质量 |

#### 进化流程与审批机制（4步）

**第1步（变更提案阶段）**：Agent识别到可优化点后，生成《指南迭代提案》：

```markdown
## 指南迭代提案

### 基本信息
- 提案ID：PROPOSAL-{年份}-{序号}
- 关联需求ID：REQ-xxxx-xxx
- 提案Agent：{Agent名称}
- 提案时间：YYYY-MM-DD

### 变更内容
- 目标指南：{指南名称}
- 变更类型：新增章节 / 修改规则 / 删除冗余项
- 变更描述：{具体描述}

### 变更理由
- 触发场景：{触发变更的需求ID及具体情况}
- 现有指南缺陷：{当前指南未覆盖或不合理之处}
- 预期收益：{变更后的预期改善}

### 验证依据
- 历史案例：{已有实际案例的需求ID及相关文档路径}
- 人类Review反馈：{架构师或技术负责人的确认意见}
```

**第2步（人类审批阶段）**：

- 审批责任人：对应指南的Owner（通常是架构师或技术负责人）；
- 审批时限：提案提交后48小时内完成；
- 审批结果：通过（进入实施）/ 驳回（说明理由）/ 延期（纳入下一迭代周期）。

**第3步（变更实施阶段）**：

- 审批通过后，Agent更新指南文档，生成新版本；
- 版本递进规则：
  - 新增章节/规则：次版本号+1（如V1.0.0 → V1.1.0）；
  - 修改表述/优化细节：修订号+1（如V1.0.0 → V1.0.1）；
  - 重大架构调整：主版本号+1（如V1.0.0 → V2.0.0）；
- 在指南文档末尾维护变更历史，记录变更内容、原因、审批人。

**第4步（效果验证阶段）**：

- 验证周期：变更生效后的3个需求任务；
- 验证指标：变更规则的执行覆盖率 + 相关问题的发生率变化；
- 验证结论：若效果达标，变更正式固化；若效果不佳，回滚变更并重新评估。

#### 进化边界与安全约束

- **人类审批一票否决**：任何指南变更必须经过人类审批，Agent不可自行生效；
- **变更可追溯**：所有变更提案、审批记录、变更内容必须完整留存，支持回溯审计；
- **回滚机制**：变更生效后若出现问题，支持一键回滚至上一版本；
- **敏感规则保护**：涉及安全、合规、核心架构的规则，需更高级别审批（如技术委员会）；
- **变更频率限制**：单份指南的变更频率不超过每周2次，避免频繁变动导致执行混乱。

### 5.3 规约减重流程

规约减重是指：Agent能力内化后，将不再需要显式约束的规则降级归档，精简规则库冗余度。

#### 减重判断标准（三维，缺一不可）

| 评估维度 | 达标条件 |
|----------|----------|
| 行为稳定性 | Agent在连续 ≥10 个需求（含不同类型）中均自主达标，无需规则提示、无人工干预、无回退 |
| 泛化覆盖性 | 达标表现覆盖规则的全部约束场景（正常/边界/异常），不存在某类场景的稳定偏差 |
| 失效可感知 | 即便移除规则约束，偏差可被现有校验体系（门禁、巡检、验收）及时捕获，不会静默扩散 |

#### 减重操作流程

1. **规则归档，不删除**：将待减重规则降级为"历史参考"，保留在规则库归档区，注明减重时间与触发原因；
2. **灰度退出**：先在单个需求中移除规则观察一个完整交付周期，确认无异常后再全量退出；
3. **人类终审**：减重决策须由人类（技术负责人）最终确认，Agent可发起减重提案，但无权自行执行。

#### 减重提案标准模板

```markdown
## 规约减重提案

### 基本信息
- 提案ID：SLIM-{年份}-{序号}（例：SLIM-2026-001）
- 提案时间：YYYY-MM-DD
- 提案发起方：{Agent名称 / 人类姓名}
- 目标规则：{规则所在指南名称} - {规则编号/名称}
- 规则当前状态：执行约束

### 规则原文
（完整引用待减重规则的原始表述，不可摘要，不可改写）

### 三维评估结果

#### 1. 行为稳定性
- 评估周期内需求总数：{N} 个（需 ≥10，列出需求ID）
- 需求类型覆盖：{列出已覆盖的需求类型}
- 达标情况：全部自主达标 / 存在以下例外（列出例外）
- 是否有人工干预记录：有 / 无

#### 2. 泛化覆盖性
- 正常场景覆盖：已覆盖 / 未覆盖（说明缺口）
- 边界场景覆盖：已覆盖 / 未覆盖（说明缺口）
- 异常场景覆盖：已覆盖 / 未覆盖（说明缺口）

#### 3. 失效可感知
- 偏差捕获机制：{列出可兜底的校验环节}
- 静默扩散风险评估：低 / 中 / 高（高风险不可减重）

### 减重方式建议
- [ ] 完全退出：从执行约束移至归档区
- [ ] 降级保留：校验级别从"阻断"降为"预警"
- [ ] 范围收窄：仅在特定场景下保留约束

### 审批记录
- 审批人：
- 审批时间：
- 审批结论：通过 / 驳回 / 延期
- 生效时间：
```

#### 减重归档路径规范

| 文件类型 | 归档路径 | 命名规则 |
|----------|----------|----------|
| 减重提案文件 | `docs/slim/proposals/` | `{提案ID}.md` |
| 归档规则原文 | `docs/slim/archived-rules/` | `{提案ID}-rule.md` |

#### 减重恢复触发机制

以下任意一种情形出现时，须立即触发恢复流程：

- **偏差复现**：已减重规则所覆盖的行为维度出现可观测的质量偏差，首次出现即触发；
- **模型/Agent版本切换**：底层模型或工具链发生版本升级，须重新执行三维评估；
- **业务场景扩展**：引入新业务类型、技术栈或合规要求；
- **巡检异常告警**：Agent-5巡检报告连续2次（含）以上在同一维度发现问题。

恢复操作规则：

1. **即时生效**：恢复触发后，归档规则须在24小时内重新纳入执行约束；
2. **根因定位前置**：恢复的同时，须同步启动根因分析，结论纳入规则备注；
3. **恢复记录留存**：在规则归档区追加恢复记录（恢复时间、触发情形、根因结论、操作人）。

### 5.4 规约增重流程（需求完成度不足时）

规约增重是指：通过向规约体系补充新规则或强化已有规则，修复质量不足的根因。

#### 增重触发信号

| 信号来源 | 识别主体 | 典型表现 | 根因类型 |
|----------|----------|----------|----------|
| 验收报告中的致命/严重问题 | Agent-4 | 核心场景未覆盖、接口行为不符预期 | 规约覆盖盲区 |
| 评审文档中的反复修改项 | Agent-2 | 同一类问题在多轮评审中持续出现 | 规约约束不明确 |
| 人类多次介入同类场景 | 人类 | 相同类型的干预在 ≥3 个需求中重复触发 | 规约未覆盖该场景 |
| Agent对齐确认中的模糊点 | Agent-3 | 多个需求中均出现同类设计歧义 | 规约表述存在歧义 |
| Agent-5巡检的高频问题 | Agent-5 | 同一类偏差在巡检报告中反复出现 | 规约校验规则缺失 |

#### 增重评估维度（四维，缺一不可）

| 维度 | 达标条件 |
|------|----------|
| 可复现性 | 问题在 ≥2 个需求中独立复现，或在单次需求中影响范围足够系统性 |
| 可规则化 | 问题根因可被抽象为明确的执行规则，不依赖主观判断 |
| 可验证性 | 新规则的遵守情况可被 Agent 或自动化工具客观校验 |
| 无冗余 | 确认现有规约中无语义重叠的规则 |

#### 增重操作步骤（5步）

**第1步：问题溯源记录**

```markdown
## 问题溯源记录

- 首次出现需求：{REQ-xxxx-xxx}
- 复现需求列表：{REQ-xxxx-xxx, ...}
- 信号来源：{验收报告 / 评审文档 / 人工干预记录 / 巡检报告}（附文档路径）
- 问题描述：{Agent的实际行为与预期行为的偏差}
- 根因分类：{覆盖盲区 / 约束不明确 / 表述歧义 / 校验规则缺失 / 结构性缺失}
- 目标指南：{指南名称及路径}
- 补充方向：{新增规则 / 强化规则 / 重写规则 / 新增校验 / 新增章节}
```

**第2步：规则草稿起草**

新规则必须包含以下要素，缺一不可：

- 规则编号（延续目标指南现有编号序列）；
- 规则表述（使用"必须/禁止/须"等强制性动词）；
- 适用范围（说明规则在何种场景下生效）；
- 校验方式（pre-commit / pre-push / Agent-5巡检 / 验收Agent）；
- 问题分级（违反时的问题等级及处置方式）；
- 正例/反例（各至少1个）。

**第3步：提案提交（使用增重提案模板）**

```markdown
## 规约增重提案

### 基本信息
- 提案ID：ENRICH-{年份}-{序号}
- 提案时间：YYYY-MM-DD
- 提案发起方：{Agent名称 / 人类姓名}
- 目标指南：{指南名称及文档路径}
- 补充方向：新增规则 / 强化规则 / 重写规则 / 新增校验 / 新增章节

### 问题溯源
（引用第1步溯源记录内容）

### 规则草稿
（引用第2步规则草稿内容，含正例/反例）

### 四维评估结果
（可复现性/可规则化/可验证性/无冗余各维度评估结论）

### 审批记录
- 审批人：
- 审批结论：通过 / 驳回 / 退回重新起草
- 灰度试行需求：
- 全量生效时间：
```

**第4步：灰度试行**

- 审批通过后，在正式纳入指南前，须在1-2个需求中试行；
- 观察指标：新规则是否被Agent正确理解 + 是否产生新的歧义 + 是否存在误判；
- 无问题则全量生效，有表述问题则修正后再试行一轮，有根本性问题则退回重新起草。

**第5步：指南版本递进**

- 新增/修改单条规则：修订号+1（如V1.0.0 → V1.0.1）；
- 新增章节或多条规则：次版本号+1（如V1.0.0 → V1.1.0）；
- 规约结构性重构：主版本号+1（如V1.0.0 → V2.0.0）。

#### 增重归档路径规范

| 文件类型 | 归档路径 | 命名规则 |
|----------|----------|----------|
| 增重提案文件 | `docs/slim/proposals/` | `{提案ID}.md` |
| 生效规则快照 | `docs/slim/enacted-rules/` | `{提案ID}-rule.md` |

### 5.5 slim-index.md 维护规范

`docs/slim/slim-index.md` 统一记录所有减重提案（`SLIM-`）与增重提案（`ENRICH-`）的状态汇总：

```markdown
## 规约提案索引

| 提案ID | 提案类型 | 目标指南/规则 | 提案时间 | 当前状态 | 生效时间 | 最近恢复时间 |
|--------|----------|---------------|----------|----------|----------|--------------|
| SLIM-2026-001 | 减重 | 《需求校准指南》- 规则003 | 2026-03-28 | 已减重 | 2026-04-01 | - |
| ENRICH-2026-001 | 增重 | 《方案设计文档编写指南》- 规则009（新增） | 2026-04-20 | 已生效 | 2026-04-25 | - |
```

`slim-index.md` 由操作人在每次提案提交、审批完成、生效或恢复触发后同步更新；Git Hook 在 pre-commit 阶段自动执行 S-001 至 S-006 校验（脚本见本节附录），阻断不合规提交；Agent-5 巡检时做全量兜底校验。

> **关于 Agent-5 的规约迭代职责**：人类在企微群中 @规约指南（Agent-5）并附上产出物Review反馈后，Agent-5 分析问题根因 → 修正对应指南文档 → 将变更摘要发回群内 → 人类确认后正式生效。此闭环是规约体系持续收敛的核心驱动力，与本节减重/增重提案流程并行运作，互为补充。

### 5.6 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| 提案被驳回 | Agent补充材料后可重新提案，须在原提案基础上说明改进方向 |
| 灰度试行发现根本性问题 | 退回重新起草，重走审批流程 |
| 变更生效后效果不佳 | 触发回滚，恢复到上一版本，同步启动根因分析 |
| 频率超限（单份指南每周>2次变更） | 合并提案，统一处理，避免频繁变动 |

## 6. 依赖的指南文档

本模块自身即为指南迭代的管控机制，不依赖单一指南文档，而是管理所有指南文档的生命周期。

## 7. 模块输出验收标准

本模块运行健康的验收标准：

- [ ] `docs/slim/` 目录结构完整（proposals/、archived-rules/、enacted-rules/、slim-index.md）；
- [ ] slim-index.md 通过 S-001 至 S-006 全部校验规则（由 Git Hook pre-commit 自动执行，脚本见 5.5 节）；
- [ ] 所有提案文件按规范格式归档，审批记录完整；
- [ ] 所有指南文档末尾包含变更历史，版本号符合语义化版本规范；
- [ ] 未经人类审批，无指南文档被直接修改（可通过Git提交记录追溯）。
