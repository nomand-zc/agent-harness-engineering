# Backlog 管理规范

**文档定位**：定义整个项目的跟进处理事项（Backlog）管理机制，包括事项来源、文件结构、字段规范、优先级体系、排期转化流程。  
**核心目标**：将研发流程中发现的「不阻碍当前需求但值得后续处理」的事项统一沉淀，避免这些事项被遗忘或无序堆积，同时避免它们侵入当前需求流程导致复杂度失控。  
**管理者**：Agent-7（Backlog 整理 Agent）

---

## 一、核心原则

### 1.1 聚焦当前需求，不让外围事项侵入

当任何 Agent 在处理当前需求时，发现以下类型的事项：

- 当前代码存在但**不影响本次需求功能**的重构点
- 设计方案中识别出的**增强型需求**（当前不做，未来值得做）
- 评审发现的**可优化点**（「优化」级别问题，非必须修改）
- 为完成本次需求而做出的技术妥协（**技术债务**）
- 对齐确认中发现的**改进建议**（不影响编码启动）
- 人类提出的**补充性需求**或改进想法

**这类事项一律不得塞入当前需求的设计方案、评审文档或对齐确认文档中作为阻塞项**，而应写入 Backlog，由 Agent-7 统一管理。

### 1.2 两类来源，统一入口

| 来源类型 | 典型场景 | 写入方式 |
|---------|---------|---------|
| **Agent 自动写入** | Agent-1 设计时发现重构机会；Agent-2/专家 Agent 评审时输出「优化」建议；Agent-3 对齐时记录改进建议 | 写入当前需求目录下的 `backlog.md`，由 Agent-7 定期汇总 |
| **人类 / 第三方接入** | 人类在企微群提出改进想法；需求管理平台导入 Backlog 条目；复盘会议产出 | 直接写入全局 `docs/backlog/backlog.md`，或 @ Backlog整理 触发 Agent-7 整理 |

### 1.3 Backlog 不等于需求

Backlog 条目是「待排期候选」，不是已立项需求。转化为需求需经 Agent-7 整理评估后，由人类确认排期，再触发 Agent-0 标准化转化流程。

## 二、文件结构

### 2.1 需求级 Backlog（各需求目录下）

每个需求处理过程中产生的 Backlog 条目，写入该需求目录：

```
docs/demand/{需求ID}/backlog.md
```

**写入时机**：
- Agent-1 完成方案设计时，输出「不在本次范围内但值得记录」的事项
- Agent-2/专家 Agent 完成评审时，输出「优化」级建议
- Agent-3 完成对齐确认时，输出改进建议

### 2.2 全局 Backlog 汇总（项目级）

由 Agent-7 定期将各需求级 Backlog 汇总、去重、分类后维护：

```
docs/backlog/backlog.md
```

**内容来源**：
- 各需求目录下的 `backlog.md`（Agent 自动写入后由 Agent-7 汇总）
- 人类直接写入
- 第三方系统接入（预留扩展，如需求管理平台 Webhook）

## 三、Backlog 条目字段规范

### 3.1 需求级 backlog.md 单条格式

```markdown
### BL-{需求ID}-{序号} {事项标题（一句话）}

| 字段 | 内容 |
|------|------|
| **来源** | Agent-1 方案设计 / Agent-2 平级评审 / 安全专家 / 性能专家 / 合规专家 / Agent-3 对齐确认 / 人类 |
| **来源需求** | {当前需求ID，如 REQ-2026-001} |
| **来源文档** | {产生该条目的原始文档路径，如 `docs/demand/REQ-2026-001/review/peer-review-V1.0.0.md` 中 P-005} |
| **类型** | 重构 / 增强 / 技术债务 / 可优化点 / 补充性需求 / 其他 |
| **优先级** | P0（紧急）/ P1（高）/ P2（中）/ P3（低） |
| **描述** | {具体描述，说明「是什么」「为什么值得做」「不做的风险」} |
| **影响范围** | {涉及的模块/文件/功能，便于后续评估工作量} |
| **建议处理方式** | 独立需求 / 附加到某个已有需求 / 技术任务（不对外） |
| **状态** | 待汇总 / 已汇总 / 已排期 / 处理中 / 已完成 / 已关闭 |
| **写入时间** | YYYY-MM-DD |
```

### 3.2 全局 backlog.md 单条格式

全局 Backlog 在需求级基础上增加：

```markdown
| **汇总时间** | YYYY-MM-DD（Agent-7 汇总时填写） |
| **排期建议** | {Agent-7 给出的排期建议，如「建议在 REQ-2026-003 之后处理」} |
| **转化为需求** | 未转化 / 已转化为 {需求ID} |
```

## 四、事项类型与优先级定义

### 4.1 类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| **重构** | 现有代码可以更好地组织，不改变外部行为 | 将某个包内的类型提取到公共包，消除重复定义 |
| **增强** | 现有功能的扩展，当前已有基础但未完全实现 | 为现有接口增加批量操作能力 |
| **技术债务** | 为了赶进度做出的妥协，后续需归还 | 某处硬编码配置，后续需改为可配置 |
| **可优化点** | 性能、安全、可维护性上的改进空间，非必须 | 某查询路径存在 N+1 问题，当前数据量小暂不影响 |
| **补充性需求** | 当前需求完成后，自然衍生出的新需求 | 完成了数据采集，后续可以增加数据导出功能 |
| **其他** | 不属于上述类型的其他跟进事项 | 需要补充某段文档说明 |

### 4.2 优先级定义

| 优先级 | 说明 | 触发排期条件 |
|--------|------|------------|
| **P0（紧急）** | 不处理可能影响线上稳定性或数据安全 | 发现后立即安排，不等当前需求完成 |
| **P1（高）** | 影响后续多个需求的推进效率 | 当前需求完成后优先处理 |
| **P2（中）** | 有明确改善效果，但不紧迫 | 纳入下个迭代计划 |
| **P3（低）** | 有想法但价值不明确，待进一步评估 | 积累后定期复盘，按需处理 |

> **注意**：P0 级别的 Backlog 条目应立即通知人类，不走常规排期流程。

## 五、Agent-7 的整理与分发流程

### 5.1 整理流程

Agent-7 执行整理时，按以下步骤处理：

1. **扫描收集**：遍历 `docs/demand/` 目录，收集所有需求目录下状态为「待汇总」的 `backlog.md` 条目
2. **去重合并**：识别内容相似的条目，合并为同一条，保留所有来源引用
3. **分类整理**：按类型和优先级重新组织
4. **优先级评估**：结合当前项目阶段、已有需求计划，调整优先级建议
5. **排期建议**：为 P1/P2 条目给出排期建议（建议在某需求之前/之后处理）
6. **更新状态**：将已汇总的需求级条目状态更新为「已汇总」
7. **更新全局 Backlog**：将整理结果写入 `docs/backlog/backlog.md`
8. **通知人类**（企微群）：推送整理摘要，高亮 P0/P1 条目，等待人类确认排期

### 5.2 排期转化流程

当人类确认某条 Backlog 条目可以排期处理时：

```
人类确认排期
    ↓
Agent-7 将条目状态更新为「已排期」
    ↓
Agent-7 将条目转化为需求草稿（或由人类直接提供）
    ↓
触发 Agent-0（需求校准 Agent）标准化 PRD 转化
    ↓
进入标准研发流程（Agent-0→1→2→3→4）
```

也可由人类直接 @ Agent-0 触发，Agent-7 负责更新该条目的状态为「处理中」。

### 5.3 全自动排期模式（高级扩展，条件具备时启用）

满足以下条件时，Agent-7 可不经人类确认，自动将 Backlog 条目转化为需求并触发 Agent-0：

- 人类已在配置中预设了「自动排期策略」（如「P1 条目在当前需求完成后自动排期」）
- 条目类型为「技术债务」且优先级为 P1
- 该条目无跨模块影响（影响范围仅限单一模块/包）

> **当前默认模式为「人类手动确认排期」**，全自动模式为可选扩展，需人类显式启用。

## 六、触发方式

### 6.1 人类手动触发

在企微群中发送：
```
@Backlog整理 请整理当前 Backlog，输出排期建议
```

Agent-7 执行一次完整的整理流程（第5.1节），输出整理报告至企微群。

### 6.2 需求完成事件驱动

每次需求验收通过（模块4 验收 Agent 输出「验收通过」结论）后，自动触发 Agent-7 执行一次整理流程，检查是否有与该需求关联的 Backlog 条目可以联动排期。

### 6.3 定时自动触发

每周一 09:00 自动执行一次全量整理，输出上周新增 Backlog 条目摘要和当前待处理事项看板至企微群。

## 七、与第三方系统的扩展接入（预留）

以下为预留扩展接口，当具备条件时可接入：

| 接入方式 | 说明 | 接入条件 |
|---------|------|---------|
| 需求管理平台 Webhook | 外部系统推送新条目到全局 backlog.md | 需要 Webhook 接收服务 |
| 人工导入 CSV/JSON | 从其他系统批量导出条目后人工放入 `docs/backlog/imports/` | 由 Agent-7 解析并标准化格式 |
| 复盘会议纪要解析 | 会议助手 Agent 将复盘纪要中的改进建议自动提取为 Backlog 条目 | 需会议助手 Agent 支持该能力 |

## 九、与现有模块的关联

| 模块/Agent | 关联说明 |
|-----------|---------|
| Agent-1（方案设计） | 设计时将「不在本次范围内但值得记录」的事项写入需求级 backlog.md |
| Agent-2（评审） | 将「优化」级评审意见中值得后续处理的条目写入需求级 backlog.md |
| 专家 Agent（安全/性能/合规） | 将非当前版本必须解决的专项问题写入需求级 backlog.md |
| Agent-3（对齐确认） | 将「改进建议」章节中不影响编码启动的建议写入需求级 backlog.md |
| Agent-4（验收） | 验收通过后触发 Agent-7 执行整理流程 |
| Agent-7（Backlog 整理） | 汇总、整理、优先级评估、排期建议、通知人类 |
| 会议助手 Agent | 可将复盘会议纪要中的改进建议提取为 Backlog 条目（扩展能力） |
