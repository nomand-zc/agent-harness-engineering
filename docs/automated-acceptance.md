# 模块4：自动化验收与全流程闭环

## 1. 目标定位

本模块核心解决「代码实现→质量达标可交付」的闭环问题，通过全维度自动化验收确保交付质量，并通过版本打标完成全流程闭环。主要解决以下问题：

- 验收维度不全，遗漏性能、安全、兼容等关键场景；
- 验收标准不可量化，导致验收结论主观不一；
- 验收不通过时整改流程不规范，反复循环耗时；
- 交付物无版本标记，无法支撑版本追溯。

## 2. 涉及的 Agent

| Agent | 角色 | 职责 |
|-------|------|------|
| Agent-4（验收Agent） | 执行主体 | 依据验收标准，执行全维度自动化验收测试，生成验收报告 |
| 性能专家Agent | 辅助验收 | 协助执行性能验收（并发、吞吐量、响应时间测试） |
| 安全专家Agent | 辅助验收 | 协助执行安全验收（漏洞扫描、权限校验、数据脱敏） |
| Agent-3（高级开发Agent） | 整改执行 | 根据验收报告中的整改建议修改代码，重新提交验收 |
| Agent-7（Backlog整理Agent） | 跟进事项承接 | 验收通过后自动触发，汇总本需求全流程产生的 Backlog 条目，检查是否有关联条目可联动排期 |
| 人类（技术负责人） | 兜底决策 | 验收出现无法自动整改的问题时介入，调整验收标准或指导整改 |

## 3. 触发前置条件

- 模块3（开发对齐与代码实现）完成：代码已推送至主干，pre-commit和pre-push门禁均已通过；
- **自动触发**：代码推送至主干后，自动触发Agent-4。

| 现有代码库 | 项目根目录 | 源代码 |

| 《验收报告编写指南》 | `docs/guide/acceptance-report-guide.md` | Markdown |

### 输出

| 输出物 | 格式 | 归档路径 |
|--------|------|----------|
| 验收报告 | Markdown | `docs/demand/{需求ID}/acceptance/report-V{版本号}.md` |
| CI构建结果 | 系统记录 | CI平台（绑定需求ID、版本标记） |
| 版本标记 | Git Tag | 格式：`V{版本号}-{需求ID}`，如 `V1.0.0-REQ-2026-001` |

### 产出物版本管理与保留规范

本模块产出物遵循**多轮验收版本递进** 与 **变更追踪** 原则：

**关键原则**：

1. **验收报告版本递进规则**：
   - 初始验收报告版本：与验收标准版本号一致（如验收标准V1.0.0对应报告V1.0.0）
   - 验收失败重试：每次重试输出新版本（V1.0.0→V1.0.1→V1.0.2...）而非覆盖
   - 验收标准变更后：验收标准升级版本（如V1.0.0→V1.1.0）时，对应验收报告也需升级版本
   - 最多允许3次自动重试，超过则触发人类介入

2. **多次验收的完整保留**：
   - 验收失败的报告应保存至 `docs/demand/{需求ID}/acceptance/history/`
   - 保留完整的"第1次验收→失败→整改→第2次验收→失败→整改→第3次验收→通过"的过程记录
   - 每份验收报告都应记录本轮失败原因、整改建议、对应代码Commit

3. **验收标准变更的版本管理**：
   - 当人类修改验收标准文档时，标准版本号应升级（如V1.0.0→V1.0.1）
   - 变更记录应单独保存至 `docs/demand/{需求ID}/acceptance/criteria-changelog.md`
   - 记录"何时变更、变更原因、变更内容、变更前后版本"
   - 对应的验收报告版本号也应同步升级

4. **Git Tag版本策略**：
   - 首次验收通过时打Tag：`V1.0.0-REQ-2026-001`
   - 若因标准变更重新验收，新Tag应采用递进版本：`V1.0.1-REQ-2026-001`（替换旧Tag）
   - 保留旧Tag的归档记录在 `git tag -l` 的备注中

5. **版本对应矩阵**（新增必需产出物）：
   - 生成至 `docs/demand/{需求ID}/acceptance/version-mapping.md`
   - 记录验收标准版本、验收报告版本、验收结论、整改次数、最终Commit、Git Tag

### 与其他模块的标准化对接

- **上游（模块3）**：接收代码仓库Commit和验收标准文档；
- **下游（全流程闭环）**：验收全量通过后，执行CI构建、版本打标，完成交付；
- **接口约定**：验收报告与验收标准一一对应，版本号保持关联；所有版本变更、验收重试的完整历史均需保留；验收标准与验收报告的版本映射矩阵必须完整记录。

## 5. 处理流程、校验标准与异常处理

### 5.1 操作步骤（共5步，全自动化为主）

**第1步（自动触发）**：代码推送至主干后，自动触发Agent-4。

**第2步（Agent-4执行前置探索）**：

在执行验收之前，Agent-4必须完成以下项目探索，理解业务背景和验收标准的约束：

1. **理解项目背景与现有功能**：
   - 阅读 `README.md` 或项目顶层文档，理解项目核心目标和用户场景；
   - 检查 `docs/demand/` 下是否有相关的历史需求与验收记录，了解该功能的验收历史；
   - 梳理本次需求与现有功能的关联与验收策略。

2. **阅读项目规范文档**：`docs/ARCHITECTURE.md`，了解项目的设计约束和验收约束；

3. **深度探索现有代码库**，回答以下关键问题：
   - 代码实现的关键模块、核心接口、主要数据流是什么？
   - 已有类似功能的验收方式是什么？（可复用现有验收策略）
   - 代码中是否有性能敏感点、安全敏感点值得重点验收？

4. 将探索结论记录在验收报告的"背景理解"章节中。

**第3步（Agent-4执行全维度验收）**：

1. 读取验收标准文档、代码、单元测试用例；
2. 按验收标准文档逐项执行验收测试（业务场景、接口性能、安全合规、兼容性）；
3. 生成验收报告（含执行记录、问题清单、验收结论）。

**第4步（结果判断）**：

- **全量通过** → 自动执行CI构建、版本打标（格式：`V1.0.0-REQ-2026-001`），完成交付；同时触发 Agent-7 执行 Backlog 整理，汇总本需求全流程产生的跟进事项，检查是否有关联条目可联动排期；
- **存在不通过项** → 自动反馈给Agent-3，Agent-3根据整改建议修改代码，重新提交验收（最多3次）；
- **3次重试仍不通过** → 触发人类介入。

**第5步（人工兜底）**：

- 若验收出现无法自动整改的问题（如验收标准不合理）：Agent-4自动预警，人类介入调整验收标准或指导Agent-3整改，直至验收通过。

### 5.2 验收维度明细（无遗漏，可直接复用）

| 验收维度 | 校验内容 | 量化标准（示例） | 执行主体 |
|----------|----------|-----------------|----------|
| 业务验收 | 正向场景、异常场景、边界场景 | 所有场景执行成功，无报错 | Agent-4 |
| 接口验收 | 入参校验、出参格式、错误码、接口稳定性 | 接口响应时间≤500ms，错误码符合规范 | Agent-4 |
| 性能验收 | 并发量、吞吐量、资源占用 | 并发≥100QPS，CPU占用≤70%，内存占用≤800M | Agent-4 + 性能专家Agent |
| 安全验收 | 权限校验、数据脱敏、防注入、漏洞扫描 | 无安全漏洞，敏感数据（如手机号）脱敏显示 | Agent-4 + 安全专家Agent |
| 兼容性验收 | 不同环境（开发/测试/预生产）、不同版本依赖 | 所有环境均可正常运行，无依赖冲突 | Agent-4 |

### 5.3 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| 验收不通过（代码问题） | Agent-3自动读取整改建议，修改代码后重新提交验收，最多重试3次，3次仍不通过触发人类介入 |
| 验收不通过（验收标准问题） | 人类修改验收标准文档，重新触发Agent-4验收，修改记录绑定需求ID，纳入版本追溯 |
| 无法自动整改（根因不明） | Agent-4预警，人类全程介入排查根因，明确整改方向后恢复自动化流程 |

## 7. 《验收标准文档编写指南》编写规范

本指南为Agent-4执行验收测试时的核心依据，确保验收覆盖全维度、所有验收项均可量化执行。

### 7.1 指南核心目录（必含模块，不可遗漏）

1. **文档基础信息**：需求ID、关联架构版本、基准Commit；
2. **验收总则**：验收范围、验收标准、验收工具、验收责任人；
3. **业务验收**：正向场景、异常场景、边界场景（每条均有可执行步骤）；
4. **接口验收**：入参校验规则、出参格式、错误码规范、接口稳定性要求；
5. **性能验收**：并发量、吞吐量、响应时间、资源占用的量化阈值；
6. **安全验收**：权限校验、数据脱敏、防注入、漏洞扫描标准；
7. **兼容性验收**：环境兼容、版本兼容、依赖兼容要求；
8. **自动化用例**：可直接生成测试脚本，明确执行逻辑；
9. **验收结果判定标准**：全量通过条件、不通过整改要求。

## 8. 《验收报告编写指南》编写规范

本指南为Agent-4输出验收报告时的结构依据，确保验收报告记录完整、问题描述清晰可整改。

### 8.1 指南核心目录（必含模块，不可遗漏）

1. **文档基础信息**：需求ID、关联验收标准版本、验收时间、验收Agent；
2. **验收概况**：验收范围、验收环境、验收耗时、参与人员；
3. **验收执行记录**（按验收标准逐项执行，记录执行结果）：
   - 业务验收执行记录（正向场景、异常场景、边界场景执行结果）；
   - 接口验收执行记录（入参校验、出参格式、错误码测试结果）；
   - 性能验收执行记录（并发测试、压力测试、响应时间测试结果）；
   - 安全验收执行记录（权限校验、数据脱敏、漏洞扫描结果）；
   - 兼容性验收执行记录（环境兼容、版本兼容测试结果）。
4. **问题清单**（按严重程度分级）：
   - 致命问题：阻断验收通过，必须立即修复；
   - 严重问题：影响核心功能，需限期修复；
   - 一般问题：影响体验或边缘功能，可纳入后续迭代；
   - 优化建议：非问题类改进建议，写入需求目录下的 `backlog.md`，交由 Agent-7 统一排期，不阻断验收通过。
5. **验收结论**：
   - **通过**：所有验收项全部达标，无致命/严重问题；
   - **有条件通过**：核心功能达标，存在一般问题需限期整改，整改后无需重新验收；
   - **不通过**：存在致命/严重问题，需修复后重新验收。
6. **遗留问题跟踪**：未闭环问题的跟踪计划、责任人、整改时限；
7. **验收签名**：验收Agent签名、开发负责人确认、产品负责人确认。

### 8.2 验收报告与验收标准的关系

- **验收标准**（`criteria-V{版本号}.md`）：验收前制定，明确验收项与通过条件；
- **验收报告**（`report-V{版本号}.md`）：验收后输出，记录执行过程与结果；
- **一一对应**：每个验收标准版本对应一个验收报告版本，版本号保持一致。

### 8.3 验收报告归档要求

- 归档路径：`docs/demand/{需求ID}/acceptance/report-V{版本号}.md`；
- 归档时机：验收完成后24小时内归档；
- 版本映射：同步更新版本映射表，记录验收报告与验收标准的关联关系。

## 9. 模块输出验收标准

本模块完成的验收标准：

- [ ] 验收报告存在于 `docs/demand/{需求ID}/acceptance/report-V{版本号}.md`；
- [ ] 验收结论为"通过"或"有条件通过"（无致命/严重遗留问题）；
- [ ] CI构建成功，版本打标完成（格式：`V{版本号}-{需求ID}`）；
- [ ] 验收报告头部包含完整锚点字段，与验收标准版本号一致；
- [ ] 所有遗留问题已录入跟踪表，责任人和整改时限明确；
- [ ] 产出物版本对应矩阵已生成至 `docs/demand/{需求ID}/acceptance/version-mapping.md`，记录验收标准版本、验收报告版本、验收结论、整改次数、最终Commit、Git Tag；
- [ ] 若存在多轮验收重试（≤3次），所有失败的验收报告已保存至 history/ 目录，版本号逐次递增（V1.0.0 → V1.0.1 → V1.0.2）；
- [ ] 若存在验收标准变更，变更记录已保存至 `criteria-changelog.md`，记录变更时间、原因、前后版本；
- [ ] 最终Git Tag与最新验收报告版本号保持一致，历史Tag的变更记录已备案。
