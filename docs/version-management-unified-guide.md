# 版本管理与产出物保留统一指南

本文档统一规范模块1-7的产出物版本管理原则，确保全链路执行的一致性。

## 零、涉及的 Agent

| Agent | 角色 | 与本指南的关系 |
|-------|------|--------------|
| Agent-0 | 需求校准Agent | 生成 PRD 产出物，遵循轮次追加规则，维护模块1版本映射 |
| Agent-1 | 高级架构师Agent | 生成设计/验收标准产出物，遵循版本递进规则，维护模块2版本映射 |
| Agent-2 | 平级评审Agent | 生成评审文档，版本号与设计方案同步递进，写入模块2版本映射 |
| Agent-3 | 高级开发Agent | 生成对齐文档，遵循版本联动规则，维护模块3版本映射 |
| Agent-4 | 验收Agent | 生成验收报告，遵循多轮迭代规则，维护模块4版本映射；验收通过后触发 Agent-7 |
| Agent-5 | 规约指南Agent | 生成规约提案，遵循提案归档规则，维护 slim-index.md |

## 一、全局统一原则

所有产出物必须遵循以下三层原则体系：

### L1：全局核心原则（所有模块强制）

**产出物全生命周期追加不覆盖（Append-Only）原则**

- **不覆盖**：旧版本文件永久保留，禁止删除、禁止修改、禁止用新版本内容覆盖旧文件
- **追加**：新版本通过版本号递进标识，在独立文件中存储（如 `PRD-V1.0.0.md` → `PRD-V1.0.1.md`）
- **可追溯**：任意历史版本均可通过 `history/` 目录或 Git 历史恢复

### L2：模块专项原则（各模块特殊约束）

| 模块 | 专项原则 | 核心规则 |
|------|--------|--------|
| 模块1（需求校准） | 轮次追加规则 | 校准报告/问题跟踪表按Round-N轮次追加，每轮独立章节 |
| 模块2（架构设计） | 版本递进规则 | 设计/验收/评审文档版本号同步递进（V1.0.0→V1.0.1→...) |
| 模块3（开发对齐） | 版本联动规则 | 对齐文档版本与设计方案版本号保持一致，上游升级→下游同步 |
| 模块4（自动化验收） | 多轮迭代规则 | 验收失败重试→版本号递增（V1.0.0→V1.0.1→V1.0.2），最多3次 |
| 模块5（文档熵收敛） | 日期序号规则 | 巡检报告按日期+序号组织（报告-YYYYMMDD-001.md），便于追踪巡检时间 |
| 模块6（指南自我迭代） | 提案归档规则 | 减重/增重提案永不删除，所有审批/回滚记录追加在提案末尾 |
| Backlog管理（Agent-7） | 报告归档规则 | 整理报告按日期序号归档至 `docs/backlog/reports/`，需求草稿归档至 `docs/backlog/pending-requirements/`，条目状态流转遵循 Append-Only 原则 |

### L3：统一措辞（所有文档必用）

**在所有6个模块"产出物版本管理与保留规范"章节的第一段，使用以下标准表述：**

> 本模块所有产出物遵循**产出物全生命周期追加不覆盖（Append-Only）原则**。历史版本永久保留，新版本通过版本号递进标识。在此基础上，根据模块特点，实施【模块名称】专项规则。

## 二、统一的 history/ 归档规范

### 文件命名格式（所有模块统一）

**标准格式**：`{文档类型}-V{版本号}.md`

示例：
```
PRD-V1.0.0.md
architecture-V1.0.1.md
criteria-V1.0.0.md
alignment-V1.0.2.md
report-V1.0.1.md
{指南名}-V1.1.0.md
```

**模块5特例**（巡检报告）：`报告-YYYYMMDD-{类型}-{序号}.md`
```
报告-20260330-全量.md
报告-20260331-增量-001.md
报告-20260331-增量-002.md
```

### 保存路径结构（统一）

**基础规则**：`docs/demand/{需求ID}/{功能区}/history/`

示例：
```
docs/demand/REQ-2026-001/design/history/
docs/demand/REQ-2026-001/review/history/
docs/demand/REQ-2026-001/alignment/history/
docs/demand/REQ-2026-001/acceptance/history/
docs/demand/REQ-2026-001/inspection/history/
docs/guide/history/（模块6特例：指南文档）
```

### 版本递进规则（统一）

| 修改类型 | 版本号变化 | 示例 |
|---------|----------|------|
| 修改/补充/优化细节 | 修订号+1 | V1.0.0 → V1.0.1 |
| 新增章节/规则/评审维度 | 次版本号+1 | V1.0.0 → V1.1.0 |
| 重大架构/结构调整 | 主版本号+1 | V1.0.0 → V2.0.0 |
| 回滚后重新修复 | 修订号+1（不退回旧版本号） | V1.0.1回滚后重新为V1.0.2 |

### 触发频率（各模块规则）

| 模块 | 频率 | 说明 |
|------|------|------|
| 模块1 | 每轮校准 | 每次PRD版本升级时 |
| 模块2 | 每轮评审 | 设计方案/验收标准/评审文档版本升级时 |
| 模块3 | 每轮对齐 | 对齐文档版本升级时 |
| 模块4 | 每轮验收（≤3次） | 验收失败重试时，第3次不通过转人工 |
| 模块5 | 每天1次全量 + 增量按需 | 全量22:00执行，增量随时触发 |
| 模块6 | 每个提案变更 | 提案审批/生效/回滚时 |

## 三、统一的 version-mapping.md 规范

### 强制要求

所有模块都必须生成 `version-mapping.md`，规范如下：

**位置**：
```
模块1：docs/demand/{需求ID}/version-mapping.md
模块2：docs/demand/{需求ID}/design/version-mapping.md
模块3：docs/demand/{需求ID}/alignment/version-mapping.md
模块4：docs/demand/{需求ID}/acceptance/version-mapping.md
模块5：docs/demand/{需求ID}/inspection/version-mapping.md
模块6：docs/slim/slim-index.md（全局索引）
```

### 统一字段集合

**必含字段（所有模块）**：
- 版本标识（如 V1.0.0 / Round-1 / 20260330-001）
- 生成时间（YYYY-MM-DD HH:mm:ss）
- 对应Commit Hash（7位简写，如 abc1234）
- 前一版本关联（引用路径，便于追溯链条）
- 当前状态（进行中/已完成/已回滚）
- 关键产出物路径

**模块特殊字段**：

模块1：校准报告章节、问题跟踪轮次、提出方（Human/Agent）统计

模块2：设计版本、验收标准版本、平级评审版本、安全/性能/合规评审版本

模块3：关联设计版本、对齐评估分数、通过/未通过标记

模块4：验收标准版本、验收结论、整改次数、Git Tag、是否需标准变更

模块5：巡检类型（全量/增量）、问题统计（P0/P1/P2/P3）、关联需求ID列表

模块6：提案类型（SLIM/ENRICH）、目标指南、提案状态、批准人

### 全局版本映射总表

**新增文件**：`版本映射总表.md`（项目级）

用途：快速查询某需求在全流程中的版本演进

模板：
```markdown
| 需求ID | 模块1(PRD) | 模块2(架构) | 模块3(对齐) | 模块4(验收) | 模块5(巡检) | 最终Commit | 交付日期 |
|--------|-----------|-----------|-----------|-----------|-----------|-----------|--------|
| REQ-2026-001 | V1.0.0 | V1.0.0 | V1.0.0 | V1.0.0 | 报告-20260330-001 | abc1234 | 2026-03-30 |
| REQ-2026-001(修订) | V1.0.1 | V1.0.1 | V1.0.1 | V1.0.1 | 报告-20260331-001 | def5678 | 2026-03-31 |
```

## 四、统一的验收标准检查清单

所有模块的"模块输出验收标准"部分，必须包含以下检查项：

### 必含检查项（所有模块）

- [ ] 产出物版本对应矩阵（version-mapping.md）已生成
- [ ] 版本号格式符合规范（V{主}.{次}.{修} 或模块特例格式）
- [ ] 旧版本文件已保存至 history/ 目录，未删除或覆盖
- [ ] 历史版本与当前版本的完整链条可追溯（通过version-mapping表）
- [ ] 每个版本对应的Commit信息已记录（可通过git log查证）

### 条件检查项（若存在多轮迭代）

- [ ] 多轮迭代的所有轮次均有独立版本文件
- [ ] 版本号逐次递增，无跳号或重复
- [ ] version-mapping表记录了轮次间的关联关系
- [ ] 每轮迭代的变更原因/理由已记录

### 模块特殊检查项

**模块1**：
- [ ] 校准问题跟踪表包含完整轮次记录（Round-1/2/3...）
- [ ] 问题清单中"提出方"字段已填写（Human / Agent-0）
- [ ] 跟踪表顶部"轮次总览"与实际轮次节内容一致

**模块2**：
- [ ] 设计版本、验收标准版本、各评审文档版本号同步递进
- [ ] 所有评审文档均已归档至 review/ 目录

**模块3**：
- [ ] 对齐文档版本与关联的设计方案版本号保持一致
- [ ] COMMIT_MAPPING.md 记录了所有Commit与版本的对应关系
- [ ] 多轮对齐重做的历史版本已保存

**模块4**：
- [ ] 验收报告版本与验收标准版本保持关联
- [ ] 整改次数在version-mapping表中明确记录（≤3次）
- [ ] 若存在验收标准变更，criteria-changelog.md已记录

**模块5**：
- [ ] 巡检报告按规范格式独立归档（报告-YYYYMMDD-{类型}-{序号}.md）
- [ ] 多轮整改场景中，每次巡检报告标注了关联的上次报告路径
- [ ] 例外审批记录已归档至 docs/demand/inspection/exceptions/

**模块6**：
- [ ] 所有提案文件（SLIM-/ENRICH-）已归档于 docs/slim/proposals/
- [ ] 驳回的提案文件已保留，审批记录标注"驳回"
- [ ] 指南文档末尾包含变更历史章节
- [ ] slim-index.md通过了所有校验规则（S-001～S-006）

## 五、统一的回滚/恢复操作规范

所有模块支持回滚机制（以下流程）：

### 回滚触发条件

满足任一即可启动流程：
1. 验收报告发现致命问题（需在24小时内回滚）
2. 架构评审发现严重风险，设计方案需回滚
3. 代码实现与设计严重偏离，对齐文档需回滚重新评估
4. 规约指南生效后出现负作用（模块6）

### 回滚流程（5步）

1. **提案发起**：提交"回滚申请文档"，说明原因、影响范围、目标版本
2. **人类审批**：技术负责人在24小时内审批
3. **执行回滚**：批准后通过Git revert执行，生成新Commit
4. **记录留痕**：在产出物文件末尾追加"回滚记录"章节
5. **后续验证**：重新执行该版本的后续流程

### 版本号处理（一致约定）

**原则**：版本号单调递增，不退回旧版本号

示例：
- V1.0.1 出现缺陷 → 回滚到 V1.0.0
- 修复后重新生成版本号为 V1.0.2（而非重新标记为V1.0.1）

## 六、Commit Hash 统一绑定规范

所有产出物文件头部必须包含标准YAML Frontmatter：

```yaml
---
需求ID: REQ-YYYY-NNN
版本号: V1.0.0
关键Commit: abc1234
基准Commit: def5678（代码提交后填写）
生成时间: YYYY-MM-DD HH:mm:ss
---
```

同时在version-mapping表中记录完整Commit Hash（40位），支持git log追溯。

## 八、版本管理 Git Hook 校验规则

pre-commit 阶段自动检查：
- [ ] 每个版本化产出物更新后，对应的version-mapping.md是否同步更新
- [ ] version-mapping表中版本号是否与文件版本号一致
- [ ] 新增版本的Commit Hash是否与git log中记录一致

不通过则阻断提交，提示"版本映射不一致，请同步更新version-mapping.md"。
