# 模块3：开发对齐与代码实现

## 1. 目标定位

本模块核心解决「架构方案→可交付代码」的落地问题，通过开发前的方案对齐确认消除模糊点，通过门禁机制保障代码质量，确保最终交付的代码与设计方案完全一致。主要解决以下问题：

- 开发前未明确设计方案中的模糊点/争议点，导致实现偏离；
- 代码质量缺乏自动化保障，单测覆盖率不足；
- 代码与设计方案存在偏差，发现晚、整改成本高；
- 产出物归档不规范，无法追溯。

## 2. 涉及的 Agent

| Agent | 角色 | 职责 |
|-------|------|------|
| Agent-3（高级开发Agent） | 执行主体 | 对齐设计方案，编写代码、预埋单元测试，提交至代码仓库 |
| Agent-2（平级评审Agent） | 偏差复核 | 代码与设计方案存在偏差时，自动介入复核，提出修改建议 |
| 专家Agent（安全/性能/合规，可选） | 代码专项审查 | 代码推送至主干后，按需触发对应专项评审；安全专家Agent审查代码安全漏洞、性能专家Agent评估高耗路径、合规专家Agent校验合规要求 |
| 人类（技术负责人） | 门禁配置 + 兜底决策 | 配置Git Hook门禁规则，处理无法自动修复的偏差；决策是否触发专家Agent代码审查 |

## 3. 触发前置条件

- 模块2（架构设计与多维度评审）完成：架构评审所有结论为"通过"，无致命/严重遗留问题；
- **自动触发**：架构评审通过后，自动触发Agent-3。

### 产出物版本管理与保留规范

本模块产出物遵循**版本联动与追加不覆盖** 原则：

**关键原则**：

1. **版本与上游联动规则**：
   - 对齐文档版本号应与关联的设计方案版本号保持一致（如设计方案 `architecture-V1.0.1.md` 对应 `alignment-V1.0.1.md`）
   - 当上游设计文档升级版本（从V1.0.0→V1.0.1）时，下游对齐文档应同步升级版本号
   - 多轮对齐评估失败重做时，对齐文档版本号逐次递增（V1.0.1 → V1.0.2 → V1.0.3...）

2. **代码版本与Commit绑定**：
   - 每个Commit信息必须包含对应的对齐文档版本号，格式：`【{需求ID}】完成{功能描述}，对齐版本V{版本号}，单测通过`
   - 代码库中 `COMMIT_MAPPING.md` 应记录 Commit → 设计版本 → 对齐版本的对应关系
   - 便于后续追溯"某个Commit对应的设计是哪个版本"

3. **历史版本保留**：
   - 对齐评估失败的版本应保存至 `docs/demand/{需求ID}/alignment/history/`
   - 保留完整的"对齐评估→评分不通过→退回补充→重新对齐"的过程记录

4. **版本对应矩阵**（新增必需产出物）：
   - 生成至 `docs/demand/{需求ID}/alignment/version-mapping.md`
   - 记录对齐文档版本、关联设计版本、评估分数、Commit范围、状态

### 与其他模块的标准化对接

- **上游（模块2）**：接收评审通过的架构设计方案和验收标准文档；
- **下游（模块4）**：代码推送至主干后，自动触发Agent-4执行验收；
- **接口约定**：对齐确认文档头部包含需求ID、关联设计文档版本；Commit信息格式统一，绑定对齐文档版本；版本映射矩阵记录完整的版本联动关系。

## 5. 处理流程、校验标准与异常处理

### 5.1 操作步骤（共5步，含门禁卡点）

**第1步（自动触发）**：架构评审通过后，自动触发Agent-3。

**第2步（Agent-3执行前置探索）**：

在执行对齐确认之前，Agent-3必须完成以下项目探索，理解设计方案的背景和约束：

1. **理解项目背景与现有功能**：
   - 阅读 `README.md` 或项目顶层文档，理解项目核心目标和现有功能全貌；
   - 检查 `docs/demand/` 下是否有相关的历史需求与设计方案，了解该功能的演进过程；
   - 梳理本次需求与现有功能的关联与边界。

2. **阅读项目规范文档**：`docs/ARCHITECTURE.md`、`docs/CONVENTIONS.md`、`docs/TESTING.md`，确保编码时严格遵循项目约束；

3. **深度探索现有代码库**，回答以下关键问题：
   - 需求涉及的代码位置在哪些包/模块中？现有实现是否有可复用部分？
   - 新代码的依赖关系是否存在循环风险？
   - 类似功能的已有实现中，有哪些Pattern值得参考？
   - 修改现有接口/类型会影响哪些调用方？

4. 将探索结论记录在《设计方案对齐确认文档》的"背景理解"章节中。

**第3步（Agent-3执行对齐确认）**：

1. 读取评审通过的设计方案、验收标准、《设计方案对齐确认文档编写指南》；
2. 基于前置探索，梳理设计方案中的模糊点/争议点，生成待确认问题清单；
3. 按《方案对齐评估指南》逐维度评估，计算综合得分；
4. 输出《设计方案对齐确认文档》，得分≥80分且无单项<70分方可进入编码；
5. 如有设计文档不完整或需求覆盖不全，退回Agent-1补充。

**第4步（Agent-3执行编码）**：

1. 基于对齐确认后的设计方案，编写全量代码；
2. 预埋单元测试，覆盖率须达到项目基线（≥90%）；
3. 提交代码前，Git Hook自动触发门禁校验。

**第5步（门禁校验 + 归档）**：

1. **pre-commit**：自动校验代码风格（Linter无错误）、单测通过率（≥90%）、编译正确性，通过方可提交；
2. **pre-push**：自动校验代码与设计方案的一致性（接口签名、核心逻辑），通过方可推送至主干；
3. 代码推送成功后，Agent-3自动将代码、单元测试用例、对齐确认文档绑定需求ID、架构版本、Commit信息，完成归档；
4. 自动触发模块4（验收流程）。

**第6步（专家Agent代码专项审查，按需触发）**：

代码推送至主干后，满足以下任意条件时，人类触发对应专家Agent执行代码级专项审查：

| 触发条件 | 触发的专家Agent | 审查重点 |
|----------|----------------|----------|
| 涉及用户鉴权、数据存储、外部接口、敏感数据处理 | @安全评审 | 注入漏洞、越权访问、敏感数据明文、不安全依赖 |
| 涉及高并发路径、批量数据处理、缓存设计、数据库查询 | @性能评审 | N+1查询、锁竞争、内存泄漏、无效缓存 |
| 涉及数据留存、用户隐私、跨境传输、日志脱敏 | @合规评审 | 数据合规要求、隐私协议覆盖、审计日志完整性 |

> 触发语法示例：`@安全评审 请对 REQ-2026-018 的代码实现执行安全专项审查，代码路径：[路径]`
>
> 专家Agent 审查结论输出至 `docs/demand/{需求ID}/review/` 目录，问题分级遵循致命/严重/一般/优化四级标准；致命/严重问题须整改后方可进入验收流程。

### 5.2 校验标准（刚性约束，阻断性）

| 校验维度 | 校验内容 | 阻断规则 |
|----------|----------|----------|
| 代码规范 | Linter校验无错误，严格遵循项目级编码规范 | 有错误则阻断提交 |
| 单测要求 | 单元测试覆盖率≥90%，所有单测用例执行通过 | 覆盖率不达标则阻断提交 |
| 编译正确性 | 代码可正常编译，无语法错误 | 编译失败则阻断提交 |
| 一致性要求 | 代码逻辑、接口设计与架构方案完全一致，无偏离 | 不一致则阻断推送 |
| Commit规范 | Commit信息格式：`【{需求ID}】{描述}，单测通过` | 格式不符则阻断提交 |

### 5.3 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| 门禁校验不通过（代码规范/单测/编译） | Git Hook弹窗提示具体问题，Agent-3修改后重新提交，直至通过 |
| 代码与设计方案存在偏差 | Agent-2自动介入复核，提出修改建议，Agent-3修改后重新校验 |
| 对齐确认文档评分不通过（<80分） | 退回Agent-1补充设计文档或明确边界，补充后重新对齐评估 |
| 设计文档存在歧义导致实现困难 | Agent-3在对齐确认文档中标注歧义点，人类介入明确最终方案 |
| 专家Agent代码审查发现致命/严重问题 | Agent-3在24小时内完成整改，重新触发对应专家Agent复核确认后方可进入验收流程 |
| 专家Agent代码审查发现一般/优化问题 | 纳入下次迭代整改，不阻断当前验收流程，问题记录归档至评审文档 |

## 7. 《设计方案对齐确认文档编写指南》编写规范

本指南为Agent-3在开发前执行设计方案对齐确认时的执行依据，确保所有模糊点/争议点在编码前得到明确。

### 7.1 指南核心目录（必含模块，不可遗漏）

1. **文档基础信息**：需求ID、关联设计文档版本、对齐时间；
2. **待确认疑问清单**：梳理设计方案中的模糊点、争议点，逐条列出；
3. **逐条答疑**：针对疑问，给出明确答案，绑定设计方案对应段落；
4. **方案对齐评估结果**：基于《方案对齐评估指南》的打分与结论；
5. **风险提示与改进建议**：识别潜在风险，提出优化方向；
6. **对齐确认签名**：Agent-3签名、人类复核人签名、确认时间。

## 8. 《方案对齐评估指南》编写规范

本指南为Agent-3执行设计方案对齐确认时的评估依据，确保对齐过程标准化与可量化。

### 8.1 评估维度与权重

| 评估维度 | 权重 | 评估内容 |
|----------|------|----------|
| 需求一致性 | 25% | 设计方案是否完整覆盖需求，是否存在超出需求的过度设计 |
| 技术可行性 | 20% | 技术选型是否合理，是否存在技术风险或依赖风险 |
| 架构合理性 | 20% | 模块划分是否清晰，职责边界是否明确，扩展性是否充足 |
| 实现完整性 | 15% | 是否涵盖核心数据结构、接口契约、异常处理等必要设计 |
| 风险预案充分性 | 10% | 是否识别潜在风险并提供应对措施 |
| 文档规范性 | 10% | 是否符合《方案设计文档编写指南》的结构与格式要求 |

### 8.2 评分规则

- 单项评分：每项满分100分：
  - 90-100分：优秀，无问题或仅有优化建议；
  - 80-89分：良好，存在少量一般问题，不影响落地；
  - 70-79分：合格，存在一定问题，需整改后落地；
  - 60-69分：待改进，存在较多问题，需重新审视；
  - 60分以下：不合格，存在严重缺陷，需重新设计。

- 综合评分公式：
  ```
  综合得分 = Σ(单项得分 × 权重)
  ```

- **通过标准**：综合得分≥80分，且无单项得分低于70分。

### 8.3 评估执行流程

1. **文档完整性检查**：确认设计文档包含《方案设计文档编写指南》要求的全部必含模块；
2. **需求溯源校验**：逐项核对设计方案与需求PRD的对应关系，标注覆盖/超出/缺失项；
3. **技术方案审查**：评估技术选型的合理性，识别潜在技术风险；
4. **架构设计评审**：检查模块划分、职责边界、数据流设计是否合理；
5. **风险预案评估**：确认风险识别是否全面，应对措施是否可行；
6. **评分与结论**：按维度打分，计算综合得分，给出评估结论。

### 8.4 评估输出物

| 输出物 | 说明 |
|--------|------|
| 《设计方案对齐确认文档》 | 包含疑问清单、答疑记录、评估结果、风险提示 |
| 《方案对齐评估打分表》 | 记录各维度得分、问题明细、整改建议 |
| 待确认问题清单 | 需人类或下游Agent确认的问题列表 |

### 8.5 异常处理

| 异常场景 | 处置方式 |
|----------|----------|
| 设计文档不完整 | 退回Agent-1补充，补充后重新评估 |
| 需求覆盖不全 | 标注缺失项，要求Agent-1补充设计或明确边界 |
| 技术风险过高 | 标注风险点，建议调整技术方案或增加风险预案 |
| 评分不通过 | 退回Agent-1整改，整改后重新评估 |

## 9. 模块输出验收标准

本模块完成的验收标准：

- [ ] 《设计方案对齐确认文档》存在于 `docs/demand/{需求ID}/alignment/alignment-V{版本号}.md`；
- [ ] 对齐评估综合得分≥80分，无单项<70分；
- [ ] 代码已推送至代码仓库，绑定需求ID和Commit信息；
- [ ] 单元测试覆盖率≥90%，所有单测用例通过；
- [ ] pre-commit和pre-push门禁均已通过；
- [ ] Commit信息格式符合规范，绑定需求ID和对齐文档版本；
- [ ] 产出物版本对应矩阵已生成至 `docs/demand/{需求ID}/alignment/version-mapping.md`，记录对齐版本、关联设计版本、评估分数、Commit范围；
- [ ] 对齐文档版本号与上游设计方案版本号保持同步（如设计V1.0.1对应对齐V1.0.1）；
- [ ] 若存在多轮对齐重做，旧版本文件已保存至 history/ 目录，且 `COMMIT_MAPPING.md` 记录了所有Commit与版本的对应关系。
