# 方案对齐评估指南

**适用场景**：当需要对已评审通过的架构设计方案执行对齐评估、计算综合得分时，遵循本指南。  
**核心目标**：评估结论客观可量化，得分能够准确反映方案的可落地程度，为「是否允许进入编码」提供客观判断依据。

**适用 Agent**：Agent-3（高级开发 Agent），与《设计方案对齐确认文档编写指南》配套使用。

---

## 一、评估前提条件

执行评估前，必须已完成以下准备（详见《设计方案对齐确认文档编写指南》第一章）：

- [ ] 已读取评审通过的架构设计方案（含需求 PRD）；
- [ ] 已阅读项目规范文档（`ARCHITECTURE.md`、`CONVENTIONS.md`、`TESTING.md`）；
- [ ] 已深度探索现有代码库，记录背景理解摘要；
- [ ] 已完成待确认疑问清单梳理，并完成逐条答疑。

> **强制约束**：未完成代码库探索和疑问答疑就执行评估，评估结论缺乏实际依据，分数无效。

---

## 二、评估维度与权重

综合得分由以下六个维度加权计算，权重之和为 100%：

| 评估维度 | 权重 | 核心评估问题 |
|---------|------|------------|
| 需求一致性 | 25% | 设计方案是否完整覆盖 PRD 所有需求点，是否存在超出需求的过度设计？ |
| 技术可行性 | 20% | 技术选型是否合理，是否与项目规范冲突，是否存在不可接受的技术风险？ |
| 架构合理性 | 20% | 模块划分是否清晰，职责边界是否明确，依赖方向是否符合项目架构约束？ |
| 实现完整性 | 15% | 核心数据结构、接口定义、业务逻辑、错误处理、边界条件是否均已说明？ |
| 风险预案充分性 | 10% | 是否识别了主要潜在风险，并提供了可执行的应对措施？ |
| 文档规范性 | 10% | 是否符合《方案设计文档编写指南》的结构与格式要求？ |

---

## 三、各维度评分细则

### 3.1 需求一致性（满分 100，权重 25%）

**评分依据**：逐项核对设计方案与需求 PRD 的对应关系。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 所有需求点均有设计覆盖，无超出需求的过度设计，需求边界清晰 |
| 80-89 | 主要需求点有覆盖，存在 1-2 个次要需求点未明确，或存在少量过度设计 |
| 70-79 | 核心需求有覆盖，存在较多次要需求点未明确，需 Agent-1 补充 |
| 60-69 | 存在核心需求点未覆盖，或方案与需求存在明显偏差 |
| <60 | 大量需求点缺失，或方案与需求严重不符，须重新设计 |

**必须逐项检查的内容**：
- [ ] 对照 PRD 功能列表，逐条确认设计方案中的覆盖位置（章节+内容）
- [ ] 标注「PRD 有需求，方案无对应设计」的缺失项
- [ ] 标注「方案有设计，PRD 无对应需求」的超出项（过度设计）
- [ ] 确认「不在本次范围内」章节是否明确排除了 PRD 明确不做的内容

**评分记录格式**：
```markdown
### 需求一致性评分

| PRD 需求点 | 方案覆盖位置 | 覆盖状态 |
|-----------|------------|---------|
| {需求点1} | 第{N}章 {章节标题} | 完整覆盖 / 部分覆盖 / 未覆盖 |
| {需求点2} | - | 未覆盖 |

**缺失项**：{列出未覆盖的需求点}  
**超出项**：{列出过度设计的部分，或「无」}  
**得分**：{分数} / 100  
**失分原因**：{说明}
```

---

### 3.2 技术可行性（满分 100，权重 20%）

**评分依据**：评估技术选型的合理性和落地风险。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 技术选型符合项目规范，无技术风险，依赖关系明确无循环 |
| 80-89 | 技术选型基本合理，存在 1-2 个可接受的技术风险，已有应对措施 |
| 70-79 | 技术选型存在部分不合规（如语言/框架/数据库与项目规范不符），需调整 |
| 60-69 | 存在较高技术风险，或依赖关系存在循环，需重新评估 |
| <60 | 技术选型与项目规范严重冲突，或存在不可接受的技术风险 |

**必须逐项检查的内容**：
- [ ] 技术选型（语言版本、框架、存储、中间件）是否符合 `docs/ARCHITECTURE.md` 中的约束
- [ ] 新模块的依赖方向是否产生循环依赖（结合代码库实际依赖图）
- [ ] 第三方依赖是否存在版本冲突或安全风险
- [ ] 技术方案是否依赖不稳定的外部接口或实验性特性

**评分记录格式**：
```markdown
### 技术可行性评分

| 检查项 | 结论 | 说明 |
|--------|------|------|
| 技术选型符合项目规范 | 是 / 否 | {说明} |
| 无循环依赖风险 | 是 / 否 | {说明} |
| 第三方依赖无风险 | 是 / 否 | {说明} |
| 无实验性特性依赖 | 是 / 否 | {说明} |

**得分**：{分数} / 100  
**失分原因**：{说明}
```

---

### 3.3 架构合理性（满分 100，权重 20%）

**评分依据**：评估模块划分、职责边界、数据流设计的合理性。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 模块划分清晰，职责单一，数据流向合理，扩展点已预留，符合项目架构原则 |
| 80-89 | 模块划分基本合理，存在 1-2 个职责边界模糊或可扩展性不足的问题 |
| 70-79 | 存在明显的职责混乱或依赖方向问题，需 Agent-1 调整 |
| 60-69 | 架构设计存在较多问题，影响可维护性和扩展性 |
| <60 | 架构设计存在根本性问题，无法支撑需求实现 |

**必须逐项检查的内容**：
- [ ] 每个新模块/包的职责是否单一，是否与已有模块存在职责重叠
- [ ] 模块间的依赖方向是否符合项目分层约束（见 `docs/ARCHITECTURE.md`）
- [ ] 核心数据流是否清晰（入口→处理→存储的路径是否有歧义）
- [ ] 接口设计是否符合项目现有 Pattern（参照代码库中类似接口）
- [ ] 跨模块共享类型的归属是否合理（不产生循环依赖）

**评分记录格式**：
```markdown
### 架构合理性评分

| 检查项 | 结论 | 说明 |
|--------|------|------|
| 模块职责单一 | 是 / 否 | {说明} |
| 依赖方向符合约束 | 是 / 否 | {说明} |
| 数据流向清晰 | 是 / 否 | {说明} |
| 接口符合现有 Pattern | 是 / 否 | {说明} |
| 共享类型归属合理 | 是 / 否 | {说明} |

**得分**：{分数} / 100  
**失分原因**：{说明}
```

### 3.4 实现完整性（满分 100，权重 15%）

**评分依据**：评估设计方案是否提供了足以支撑无歧义实现的所有细节。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 数据结构、接口签名、核心逻辑、错误处理、边界条件、包结构全部说明，实现者无需猜测 |
| 80-89 | 主体内容完整，存在 1-2 处次要细节未说明（如某个边界条件的处理方式） |
| 70-79 | 核心内容有说明，但存在多处实现者需要自行决策的细节 |
| 60-69 | 关键接口签名或核心逻辑未说明，实现者需大量猜测 |
| <60 | 设计方案严重不完整，实现者无法基于此方案编码 |

**必须逐项检查的内容**（参照《方案设计文档编写指南》章节清单）：
- [ ] 核心数据结构：每个字段有语义说明、统计口径、边界值处理
- [ ] 接口定义：所有接口方法的完整签名、参数语义、返回值约定（含 key 不存在时的行为）
- [ ] 核心逻辑：用伪代码描述执行顺序（非纯文字），含错误处理策略
- [ ] 边界条件：分母为零、空集合、负数钳制等边界已枚举
- [ ] 存储语义：每个写入接口说明 upsert/append/insert，读取接口说明 key 不存在时的行为
- [ ] 包结构与文件规划：每个新文件的职责和位置已列出

**评分记录格式**：
```markdown
### 实现完整性评分

| 检查项 | 完整性 | 缺失内容 |
|--------|--------|---------|
| 数据结构定义 | 完整 / 部分 / 缺失 | {缺失字段或口径说明} |
| 接口签名完整 | 完整 / 部分 / 缺失 | {缺失的接口或参数说明} |
| 核心逻辑伪代码 | 完整 / 部分 / 缺失 | {缺失的逻辑段} |
| 错误处理策略 | 完整 / 部分 / 缺失 | {未说明的错误场景} |
| 边界条件枚举 | 完整 / 部分 / 缺失 | {未枚举的边界} |
| 包结构规划 | 完整 / 部分 / 缺失 | {缺失的文件规划} |

**得分**：{分数} / 100  
**失分原因**：{说明}
```

### 3.5 风险预案充分性（满分 100，权重 10%）

**评分依据**：评估方案是否识别了主要风险，并提供了可执行的应对措施。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 主要技术风险、业务风险均已识别，应对措施具体可执行 |
| 80-89 | 主要风险已识别，应对措施基本可行，个别次要风险未覆盖 |
| 70-79 | 识别了部分风险，但应对措施较为模糊，或遗漏了较明显的风险 |
| 60-69 | 风险识别不足，或应对措施不可执行 |
| <60 | 几乎未识别风险，或方案本身存在不可接受的风险 |

**必须检查的常见风险类型**：
- [ ] 性能风险：高并发场景下的瓶颈点是否已识别（N+1 查询、锁竞争等）
- [ ] 数据一致性风险：分布式场景下的多实例并发是否已处理
- [ ] 兼容性风险：修改现有接口对调用方的影响是否已评估
- [ ] 依赖可用性风险：外部依赖不可用时的 fallback 行为是否已定义
- [ ] 数据迁移风险：若涉及存储结构变更，迁移方案是否已说明

**评分记录格式**：
```markdown
### 风险预案充分性评分

| 风险类型 | 是否识别 | 应对措施可执行性 | 说明 |
|---------|---------|---------------|------|
| 性能风险 | 是 / 否 | 可执行 / 模糊 / 无 | {说明} |
| 数据一致性风险 | 是 / 否 | 可执行 / 模糊 / 无 | {说明} |
| 兼容性风险 | 是 / 否 | 可执行 / 模糊 / 无 | {说明} |
| 依赖可用性风险 | 是 / 否 | 可执行 / 模糊 / 无 | {说明} |

**得分**：{分数} / 100  
**失分原因**：{说明}
```

### 3.6 文档规范性（满分 100，权重 10%）

**评分依据**：评估设计文档是否符合《方案设计文档编写指南》的结构与格式要求。

| 分数段 | 判定标准 |
|--------|---------|
| 90-100 | 所有必含章节齐全，格式规范，锚点信息完整，代码片段合法 |
| 80-89 | 主要章节齐全，存在 1-2 个章节格式不规范或内容不完整 |
| 70-79 | 缺少 1-2 个必含章节，或锚点信息不完整 |
| 60-69 | 缺少多个必含章节，文档结构混乱 |
| <60 | 文档严重不规范，无法作为编码依据 |

**必须逐项检查的内容**（对照《方案设计文档编写指南》第二章章节清单）：
- [ ] 文档基础信息（需求 ID、关联版本、基准 Commit、生效时间）
- [ ] 需求溯源与背景
- [ ] 边界定义（做什么、不做什么）
- [ ] 整体架构（模块位置+数据流）
- [ ] 技术选型
- [ ] 核心数据结构
- [ ] 接口定义
- [ ] 核心计算/业务逻辑
- [ ] 非功能设计
- [ ] 包结构与文件规划
- [ ] 验收标准（AC）
- [ ] 风险预案
- [ ] 不在本次范围内

| 核心业务逻辑 | 是 / 否 | 完整 / 不完整 |

## 五、通过判定规则

满足以下**全部**条件，评估结论为「通过」，可进入编码阶段：

- [ ] 综合得分 ≥ 80 分；
- [ ] 无任何单项维度得分 < 70 分；
- [ ] 疑问清单中无「待确认」状态的疑问；
- [ ] 高风险项均已制定应对措施。

以上任一条件不满足，评估结论为「不通过」，须退回 Agent-1 补充/修改设计文档。

## 六、评估输出物

每次对齐评估须输出以下内容：

| 输出物 | 说明 | 归档位置 |
|--------|------|---------|
| 《设计方案对齐确认文档》 | 含疑问清单、答疑记录、评估结论汇总 | `docs/demand/{需求ID}/alignment/alignment-V{版本号}.md` |
| 《方案对齐评估打分表》 | 各维度详细评分记录（即本指南第三章的输出） | 嵌入对齐确认文档，或作为附件存放同目录 |
| 待确认问题清单 | 需人类或 Agent-1 确认的问题列表（若有） | 嵌入对齐确认文档 |

## 七、异常处理

| 异常场景 | 处置方式 |
|---------|---------|
| 设计文档不完整（缺少必含章节） | 退回 Agent-1 补充，补充后重新评估；不可降低评分标准 |
| 需求覆盖不全（存在未覆盖需求点） | 标注缺失项，要求 Agent-1 补充设计或明确排除原因 |
| 技术选型与项目规范冲突 | 标注冲突项，要求 Agent-1 调整技术方案或获得人类授权 |
| 发现设计方案与代码库存在冲突 | 标注冲突点，退回 Agent-1 与 Agent-2 重新评审 |
| 评估得分处于边界值（79-81分） | 逐项复核得分依据，确保评分客观；边界分歧由人类裁决 |

## 八、评估签名规范

### 8.1 签名角色与责任边界

| 角色 | 签名产物 | 责任范围 |
|------|---------|---------|
| Agent-3（评估执行者） | 评估打分表 | 评估前提条件的完整性、各维度评分依据的客观性、综合得分计算的准确性 |
| 人类技术负责人 | 评估打分表（条件必填） | 边界分歧裁决的合理性、不通过结论退回意见的明确性 |

### 8.2 人类签名触发条件

人类技术负责人签名在以下任一情形下为**必填**，否则为可选：

- 综合得分处于边界值区间（79-81 分），存在通过/不通过分歧；
- 评估结论为「不通过」，退回 Agent-1 时需要人类明确补充/修改方向；
- 任意单项维度得分 < 70 分，对应维度的技术决策需要人类确认方向；
- 技术选型存在与项目规范的冲突，需要人类授权豁免。

### 8.3 签名前的自检清单

Agent-3 在签名前必须逐项确认：

- [ ] 已完成第一章所列全部评估前提条件（含代码库探索和疑问答疑）；
- [ ] 六个评估维度均已按第三章细则逐项检查，无跳过；

### 8.4 签名追溯说明

> 评估打分表签名是对齐评估质量的责任绑定点。若后续编码阶段发现对齐评估存在重大漏评（如遗漏明显的依赖循环风险、未发现关键接口未定义等），将追溯至 Agent-3 评估执行者签名责任。人类技术负责人签名的裁决结论，其决策合理性由人类负责人承担追溯责任。追溯结论将驱动本指南评分细则的反思迭代。
