# 验收报告编写指南

**适用场景**：当需要对一次验收执行过程输出验收报告（`report-V{版本号}.md`）时，遵循本指南。  
**核心目标**：验收报告输出后，问题描述清晰可整改，执行记录可追溯，验收结论客观无歧义，Agent-3 能直接依据整改建议修改代码。

**适用 Agent**：Agent-4（验收 Agent）。

---

## 一、输出前的强制准备步骤

在执行任何验收项之前，必须先完成以下准备，否则验收结果无法客观评定。

### 1.1 确认验收标准文档存在且版本匹配

- 确认 `docs/demand/{需求ID}/acceptance/criteria-V{版本号}.md` 已存在；
- 确认验收标准文档头部的「基准 Commit」与当前被验收代码的 Commit 一致；
- 若验收标准版本与代码版本不匹配，**须先要求 Agent-1 更新验收标准**，不可跳过。

> **强制约束**：用过期的验收标准执行验收，等于在错误的基准线上做判断，验收结论无效。

### 1.2 阅读项目背景与现有功能

在执行验收前，必须建立对被验收功能的完整理解，避免验收执行偏差：

- 阅读 `README.md` 或项目顶层文档，理解项目核心目标和用户场景；
- 检查 `docs/demand/` 下是否有相关历史需求与验收记录，了解该功能的验收历史；
- 阅读 `docs/ARCHITECTURE.md`，理解项目设计约束与验收约束；
- 深度探索被验收的核心模块代码，回答以下问题：
  - 代码实现的核心接口、主要数据流是什么？
  - 已有类似功能的验收方式是什么？（可复用现有验收策略）
  - 代码中是否有性能敏感点、安全敏感点值得重点验收？

> **强制约束**：跳过代码探索直接执行验收，等于只验证了「有没有输出」，无法发现实现与设计意图的偏差。

### 1.3 确认验收环境就绪

- 验收所需的依赖服务（数据库、中间件等）均已启动并可访问；
- 验收数据已准备（前置数据、Mock 配置、环境变量）；
- 验收工具（测试框架、性能测试工具、安全扫描工具）版本与验收标准中声明的一致。

---

## 二、验收报告的必要章节

一份合格的验收报告必须包含以下所有章节，缺少任何一章都可能导致验收结论无法追溯或整改无法执行。

### 章节清单

| 章节 | 作用 | 缺失后果 |
|------|------|---------|
| 文档基础信息 | 需求 ID、关联验收标准版本、验收时间、验收 Agent | 无法溯源，文档孤岛 |
| 验收概况 | 验收范围、验收环境、参与角色、背景理解摘要 | 无法还原验收上下文 |
| 验收执行记录 | 按验收标准逐项执行，记录实际结果（通过/不通过/跳过） | 无法判断哪些项已覆盖，哪些未覆盖 |
| 问题清单 | 致命/严重/一般/优化四级，每条含复现步骤+整改建议 | Agent-3 无法知道改哪里、怎么改 |
| 验收结论 | 通过/有条件通过/不通过，含判定依据 | 无法触发后续流程 |
| 遗留问题跟踪 | 未闭环问题的责任人、整改时限、跟踪计划 | 问题沉没，无法闭环 |
| 验收签名 | 验收 Agent 签名、开发/产品负责人确认 | 验收结论缺乏共识确认 |

## 三、文档基础信息规范

文档头部必须包含以下所有字段，字段缺失不可提交：

```markdown
**需求 ID**：REQ-{需求ID}  
**关联验收标准**：docs/demand/{需求ID}/acceptance/criteria-V{版本号}.md  
**被验收代码 Commit**：{commit hash}  
**验收报告版本**：V{版本号}  
**验收执行时间**：YYYY-MM-DD HH:MM  
**验收 Agent**：Agent-4  
**验收结论**：通过 / 有条件通过 / 不通过（摘要，详见第五章）  
**归档路径**：docs/demand/{需求ID}/acceptance/report-V{版本号}.md  
```

## 四、验收概况规范

### 4.1 验收概况格式

```markdown
## 验收概况

| 字段 | 内容 |
|------|------|
| 验收范围 | {本次验收覆盖的功能点列表} |
| 验收环境 | {环境名称，如「测试环境 v2.1.0」} |
| 数据库版本 | {版本} |
| 依赖服务状态 | {所有依赖服务的运行状态} |
| 验收耗时 | {总耗时，如「23分14秒」} |
| 验收 Agent | Agent-4 |
| 辅助角色 | {如性能专家 Agent、安全专家 Agent，无则填「无」} |
```

### 4.2 背景理解摘要（必填）

在验收概况中必须包含代码探索的结论摘要，作为验收视角的基础：

```markdown
### 背景理解摘要
- **核心模块**：{被验收功能所在的主要包/模块路径}
- **核心接口**：{本次验收涉及的关键接口名称}
- **主要数据流**：{简述数据从入口到存储的流转路径}
- **历史验收记录**：{有/无，若有则注明历史版本和结论}
- **重点验收区域**：{代码探索发现的性能敏感点或安全敏感点}
```

## 五、验收执行记录规范

### 5.1 执行记录的总体要求

- 必须按验收标准文档的章节顺序逐项记录，不可跳项；
- 每项验收结果只能填写：`通过` / `不通过` / `跳过（原因）`；
- 「跳过」必须注明原因，不可无故跳过；
- 实际执行结果与预期结果不一致时，必须记录差异详情。

### 5.2 单条执行记录格式

```markdown
**AC-{大类}.{序号}** {验收项标题}

- **执行状态**：通过 / 不通过 / 跳过
- **执行命令**：`{实际执行的命令}`
- **执行输出**：
  ```
  {实际输出内容，可截取关键部分}
  ```
- **实际结果**：{实际观察到的结果}
- **预期结果**：{验收标准中定义的预期结果}
- **差异说明**：{仅「不通过」时填写，描述实际与预期的具体差异}
- **问题编号**：{仅「不通过」时填写，关联问题清单中的编号，如 BUG-001}
```

### 5.3 各验收维度的执行记录要求

**业务验收执行记录**
- 正向场景、异常场景、边界场景必须分组记录；
- 每条记录需包含实际输出（不可只写「已执行」）。

**接口验收执行记录**
- 记录每个接口在各入参场景下的实际返回值和错误码；
- 若接口返回格式与验收标准定义不一致，必须标注为「不通过」并关联问题编号。

**性能验收执行记录**
- 必须记录实测数值（P50、P99、CPU 峰值、内存峰值）；
- 与阈值的对比必须明确（如「P99 实测 823ms，阈值 500ms，超标 64.6%」）；
- 附上测试工具的原始输出或截图路径。

```markdown
**AC-PERF.1** 接口响应时间 P99 ≤ 500ms

- **执行状态**：不通过
- **执行命令**：`go test -bench=BenchmarkGetMetrics -benchtime=10s ./observer/...`
- **实测结果**：
  | 指标 | 实测值 | 阈值 | 是否达标 |
  |------|--------|------|---------|
  | P50  | 312ms  | -    | -       |
  | P99  | 823ms  | 500ms | 未达标 |
  | CPU 峰值 | 45% | 70% | 达标 |
  | 内存峰值 | 620MB | 800MB | 达标 |
- **差异说明**：P99 超标 64.6%，性能瓶颈位于 `GetCurrentUsagesBatch` 方法（N+1 查询，见 BUG-002）
- **问题编号**：BUG-002
```

**安全验收执行记录**
- 记录每项安全检查的执行方法和结果（工具扫描结果/手动测试结果）；
- 安全漏洞必须记录 CVE 编号（如适用）和影响范围。

**兼容性验收执行记录**
- 记录每个目标环境的实际执行结果；
- 不同环境之间的差异必须说明原因。

## 六、问题清单规范

### 6.1 问题分级标准

| 问题级别 | 判定标准 | 对验收结论的影响 |
|----------|---------|----------------|
| 致命 | 核心功能完全不可用，或存在数据安全漏洞 | 验收结论为「不通过」，必须修复后重新验收 |
| 严重 | 核心功能受损，或性能指标严重未达标 | 验收结论为「不通过」，必须修复后重新验收 |
| 一般 | 边缘场景错误，或影响用户体验但不影响核心功能 | 验收结论可为「有条件通过」，纳入限期整改 |
| 优化 | 有更优实现方案，当前实现可运行 | 不影响验收结论，可选整改 |

### 6.2 每条问题的必要字段

每条问题**必须**包含以下所有字段，缺失字段不可提交：

```markdown
**BUG-{序号}**【{问题级别}】{问题标题（一句话）}

- **关联验收项**：AC-{大类}.{序号}
- **问题描述**：{具体描述发现了什么问题，包含实际观察到的现象}
- **复现步骤**：
  1. {步骤1}
  2. {步骤2}
  3. 实际结果：{...}
  4. 预期结果：{...}
- **影响范围**：{此问题影响哪些功能点或用户场景}
- **根因分析**：{问题根因，精确到代码文件和行号（如能定位）}
- **整改建议**：{Agent-3 可直接执行的修改建议，无模糊表述}
- **整改时限**：{致命/严重：立即；一般：{N}个工作日内}
- **整改负责人**：Agent-3
```

### 6.3 问题描述的写法规范

**错误写法（模糊，无法整改）**：
> BUG-001：性能不达标，需要优化

**正确写法（具体可执行）**：
> **BUG-001**【严重】`GetCurrentUsagesBatch` 存在 N+1 查询，导致 P99 超标 64.6%
>
> - **关联验收项**：AC-PERF.1
> - **问题描述**：并发 100 QPS 下，`GetCurrentUsagesBatch` 对每个 key 单独调用存储层 `Get` 方法，产生 N 次串行查询，实测 P99 为 823ms，阈值为 500ms
> - **复现步骤**：
>   1. 执行 `go test -bench=BenchmarkGetMetrics -benchtime=10s ./observer/...`
>   2. 实际结果：P99 = 823ms
>   3. 预期结果：P99 ≤ 500ms
> - **影响范围**：所有调用 `CollectMetrics` 的路径，高并发场景下性能劣化
> - **根因分析**：`observer/collector.go:142` 在 for 循环内逐个调用 `store.Get(key)`，未使用批量接口
> - **整改建议**：将 `store.Get(key)` 循环替换为 `store.GetBatch(keys)`，参考 `docs/demand/{需求ID}/design/architecture-V1.0.0.md` 第四章 4.2「批量接口必要性」
> - **整改时限**：立即（严重问题）

## 七、验收结论规范

### 7.1 结论格式

```markdown
## 验收结论

**结论**：通过 / 有条件通过 / 不通过

### 验收汇总

| 验收维度 | 总项数 | 通过 | 不通过 | 跳过 |
|---------|--------|------|--------|------|
| 业务验收 | {N} | {N} | {N} | {N} |
| 接口验收 | {N} | {N} | {N} | {N} |
| 性能验收 | {N} | {N} | {N} | {N} |
| 安全验收 | {N} | {N} | {N} | {N} |
| 兼容性验收 | {N} | {N} | {N} | {N} |
| **合计** | {N} | {N} | {N} | {N} |

### 问题汇总

| 问题级别 | 数量 | 对结论的影响 |
|---------|------|------------|
| 致命 | {N} | 不通过（须为 0 才能通过） |
| 严重 | {N} | 不通过（须为 0 才能通过） |
| 一般 | {N} | 有条件通过（须 ≤ 3） |
| 优化 | {N} | 不影响结论 |

### 判定依据
{说明结论是如何得出的，引用验收标准中的通过判定规则}

### 下一步操作
- **通过**：触发 CI 构建，执行版本打标（格式：`V{版本号}-{需求ID}`），完成交付
- **有条件通过**：遗留「一般」问题已录入跟踪表，由 Agent-3 限期整改，整改后无需重新验收
- **不通过**：Agent-3 依据问题清单整改，重新提交验收（本次为第 {N}/3 次重试）
```

### 7.2 结论判定规则

| 结论 | 判定条件 |
|------|---------|
| 通过 | 所有验收项 100% 通过，致命/严重问题为 0，一般问题 ≤ 3 |
| 有条件通过 | 核心功能（业务+接口）100% 通过，无致命/严重问题，一般问题 ≤ 3 |
| 不通过 | 存在任意致命/严重问题，或任意业务/接口验收项不通过 |

> 以上判定规则以验收标准文档中的「验收结果判定标准」章节为准，若两者冲突，以验收标准文档为准，并在报告中注明差异原因。

## 八、遗留问题跟踪规范

所有「一般」及以上级别的未闭环问题，必须录入遗留问题跟踪表：

```markdown
## 遗留问题跟踪

| 问题编号 | 级别 | 问题标题 | 整改责任人 | 整改时限 | 当前状态 |
|---------|------|---------|-----------|---------|---------|
| BUG-003 | 一般 | 空集合返回 nil 而非空切片 | Agent-3 | YYYY-MM-DD | 待整改 |

### 跟踪规则
- 「有条件通过」场景下，遗留一般问题整改完成后，由 Agent-3 在代码 commit message 中标注 `fix({需求ID}): BUG-{序号}`；
- 整改完成后更新本表中的「当前状态」为「已修复（Commit: {hash}）」；
- 不需要重新触发完整验收，但需要 Agent-4 对整改项做 Delta 验证。
```

## 九、验收签名规范

### 9.1 签名表格

验收报告尾部必须包含以下签名表：

```markdown
## 验收签名

| 角色 | 签名 | 确认时间 | 确认内容 |
|------|------|---------|---------|
| 验收执行者 | Agent-4 | YYYY-MM-DD HH:MM | 已完成强制准备步骤（含代码探索），所有验收项均已执行或注明跳过原因，问题描述含复现步骤和整改建议，验收结论客观有据 |
| 开发负责人 | Agent-3 / {人类开发负责人} | YYYY-MM-DD HH:MM | 已知晓验收结论，所有问题的整改计划已制定，整改时限已确认 |
| 产品负责人 | {人类产品负责人} | YYYY-MM-DD HH:MM | 验收结论与需求目标一致，遗留问题的处置决策已确认 |
```

### 9.2 各角色签名触发条件

**验收执行者（Agent-4）**（必填）：完成验收报告后自签，确认已遵循本指南所有强制步骤。

**开发负责人**（必填，区分以下两种情形）：

| 情形 | 签名方 | 说明 |
|------|--------|------|
| 验收结论为「通过」或「有条件通过」 | Agent-3 自签 | Agent-3 确认已知晓结论，遗留问题整改计划已制定 |
| 验收结论为「不通过」，或存在致命/严重问题 | **人类开发负责人**签名 | 需人类确认问题严重性认知和整改优先级决策 |

**产品负责人**（条件必填）：在以下任一情形下为**必填**，否则为可选：

- 验收发现**致命**级别问题（任意数量）；
- 验收发现**严重**级别问题（任意数量）；
- 验收结论为「不通过」；
- 「有条件通过」场景下，遗留「一般」问题影响对外可见的功能行为；
- 验收结论与需求预期存在争议，需产品负责人裁决。

### 9.3 签名前自检清单

验收执行者在签名前必须逐项确认：

- [ ] 已确认验收标准文档存在且版本与被验收代码 Commit 匹配；
- [ ] 已完成代码探索，背景理解摘要已填写在验收概况中；
- [ ] 所有验收项均已执行，跳过项均有原因说明，无无故跳过；
- [ ] 每条执行记录包含：执行命令、实际输出、实际结果与预期结果对比；
- [ ] 每条问题均包含完整字段：关联验收项、复现步骤、影响范围、根因分析（精确到文件/行号）、整改建议；
- [ ] 问题级别判定符合第六章分级标准；
- [ ] 验收结论与第七章判定规则一致，汇总数据无误；
- [ ] 所有「一般」及以上级别问题均已录入遗留问题跟踪表（含整改时限和责任人）。

### 9.4 签名追溯说明

> 验收签名是交付质量的最终把关节点。若验收通过后在生产环境发现遗漏的缺陷，将追溯至验收执行者（Agent-4）签名责任，要求其对本指南的对应检查清单进行反思迭代。人类开发负责人或产品负责人签名的问题处置决策，其合理性由对应人类负责人承担追溯责任。

## 十、重新验收规范

### 10.1 触发条件

以下情况必须重新执行完整验收：

- 验收结论为「不通过」，Agent-3 修改代码后重新提交；
- 验收标准文档版本更新（验收基准变更）；
- 被验收代码 Commit 变更（任何修改均触发）。

### 10.2 重新验收的报告处理

- 重新验收输出新版本报告（版本号递增，如 `report-V1.1.0.md`）；
- 新报告必须引用上次报告版本，并说明本次修复的问题列表：

```markdown
**本次为第 {N}/3 次重试验收**  
**上次报告**：report-V{上次版本号}.md  
**本次修复的问题**：BUG-001（已修复，Commit: {hash}）、BUG-002（已修复，Commit: {hash}）  
**本次新增问题**：{无 / BUG-{序号}（说明）}
```

- Delta 验证：重新验收时，对上次「通过」的项可标注「沿用上次结论（Commit 未变更）」，但涉及修改代码的关联项必须重新执行。

### 10.3 超过最大重试次数

第 3 次重试仍不通过时：

```markdown
**触发人类介入**  
**原因**：已达最大重试次数（3次），验收仍不通过  
**阻塞项**：BUG-{序号}（{级别}）：{一句话说明为何无法自动整改}  
**建议**：{人类介入方向，如「验收标准需调整」或「需人工排查根因」}
```

## 十二、常见报告错误示例

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 执行记录只写结论 | 「AC-1.1：通过」 | 必须记录实际执行命令和输出 |
| 问题描述无复现步骤 | 「性能不好」 | 给出具体命令、实测数值和与阈值的对比 |
| 整改建议模糊 | 「需要优化性能」 | 给出精确到文件/函数级别的修改建议 |
| 验收结论无判定依据 | 只写「通过」 | 必须引用验收标准的通过判定规则 |
| 遗留问题无跟踪信息 | 记录了问题但无责任人和时限 | 所有遗留问题必须录入跟踪表 |
| 跳过项无原因说明 | 「AC-SEC.2：跳过」 | 必须注明跳过原因，如「不适用：本功能无外部接口」 |

## 十三、输出格式与归档要求

- 验收报告以 **Markdown** 格式输出；
- 归档路径：`docs/demand/{需求ID}/acceptance/report-V{版本号}.md`；
- 版本号与对应的验收标准文档版本保持一致（标准 V1.0.0 → 报告 V1.0.0；重新验收则递增为 V1.1.0）；
- 归档时机：验收执行完成后 **24 小时内**归档至代码仓库；
- commit message 格式：`docs({需求ID}): add acceptance report V{版本号}`；
- 同步更新版本映射表，记录验收报告与验收标准的关联关系及 CI 构建记录。
