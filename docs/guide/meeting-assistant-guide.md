# 会议助手 Agent 执行指南

**适用场景**：需要通过腾讯会议进行需求讨论，并将会议纪要自动转化为可直接输入 Agent-0 的标准化需求草稿时，遵循本指南。

**核心目标**：通过会议室自动化集成、会议纪要智能提取、需求草稿结构化输出，打通「会议室 → 企微群 → Agent-0 流程」的全链路，确保会议决策不遗漏，需求信息完整可追溯，后续Agent流程无需人工整理。

---

## 一、会议助手 Agent 的核心职责

会议助手 Agent 在企微群中以数字人员工身份常驻，主要负责三个环节：

| 环节 | 职责 | 触发方式 | 产出物 |
|------|------|---------|--------|
| **会议预约** | 调用腾讯会议 API 创建会议室，自动生成会议链接 | 人类 @会议助手 发起预约指令 | 腾讯会议链接、日程通知 |
| **会议纪要记录** | 接入腾讯会议实时字幕/录音，自动生成结构化纪要 | 会议进行中自动启动 | 结构化会议纪要（含决策项、待办项、遗留问题） |
| **需求草稿转化** | 从会议纪要提取需求相关内容，整理为 Agent-0 输入格式 | 会议结束后自动执行 | 符合 Agent-0 输入规范的需求草稿 |

---

## 二、会议预约规范

### 2.1 预约触发指令格式

人类在企微群中发送以下格式的预约指令：

```
@会议助手 请预约腾讯会议

会议主题：[会议主题]
计划时间：[YYYY-MM-DD HH:MM]
预计时长：[分钟数]
参会人员：[人名1、人名2、...]
会议议题：
  - [议题1]
  - [议题2]
  - [议题3]
```

**示例**：
```
@会议助手 请预约腾讯会议

会议主题：用户登录优化需求评审
计划时间：2026-03-31 14:00
预计时长：60
参会人员：张三、李四、王五、赵六
会议议题：
  - 微信扫码登录功能需求评审
  - 安全合规要求确认
  - 开发实现范围对齐
```

### 2.2 必填字段说明

| 字段 | 说明 | 是否必填 | 格式要求 |
|------|------|---------|---------|
| 会议主题 | 本次会议的核心主题 | **必填** | 8-50字，不含特殊符号 |
| 计划时间 | 会议开始时间 | **必填** | `YYYY-MM-DD HH:MM`，需至少提前1小时 |
| 预计时长 | 会议预计持续时间 | **必填** | 15-240分钟 |
| 参会人员 | 预计参会的人员名单 | **必填** | 逗号/中文顿号分隔的人名列表 |
| 会议议题 | 本次会议的核心议题 | **必填** | 项目符号列表，至少2项 |

### 2.3 会议助手的预约响应

会议助手接收预约指令后，在企微群内应返回如下响应：

```markdown
【会议助手】已创建腾讯会议

📅 会议主题：用户登录优化需求评审
🕐 时间：2026-03-31 14:00-15:00（北京时间）
👥 参会人员：张三、李四、王五、赵六
🔗 会议链接：https://meeting.tencent.com/xxx
📝 会议纪要将在会议结束后30分钟内自动生成
```

**强制要求**：
- **必须**包含会议链接，确保参会人可点击加入
- **必须**标注时区（北京时间）
- **必须**标注纪要生成预期时间（会议结束后30分钟内）
- **必须**保留会议ID供后续追踪

## 三、会议纪要生成规范

### 3.1 纪要自动采集

会议助手与腾讯会议 API 集成，会议中自动启用：

| 采集方式 | 覆盖内容 | 质量要求 |
|---------|---------|---------|
| **实时字幕流** | 会议中所有发言的文字记录 | 覆盖率 ≥ 95%，识别准确率 ≥ 90% |
| **录音/录视频** | 会议完整音视频 | 音质清晰，可用于后续追溯 |
| **发言人识别** | 每条发言对应的发言人姓名 | 识别准确率 ≥ 95% |

### 3.2 结构化纪要的产出格式

**会议助手必须在会议结束后 30 分钟内在企微群发布结构化纪要**，格式如下：

```markdown
【会议助手】会议纪要已生成

📅 会议信息
- 主题：用户登录优化需求评审
- 时间：2026-03-31 14:00-15:00
- 参会人：张三（需求方）、李四（架构师）、王五（开发负责人）、赵六（测试负责人）
```

## 四、需求草稿的转化规范

### 4.1 转化的前置条件

**只有当会议纪要中包含以下信息时，会议助手才能生成需求草稿**：

- [ ] **核心诉求明确**：会议中明确讨论了"新增什么功能"或"改进什么现状"
- [ ] **涉及系统范围确认**：新功能/改进涉及的系统模块已识别
- [ ] **业务规则澄清**：功能的处理规则、异常处理等已讨论
- [ ] **关键决策锁定**：做什么 / 不做什么 的决策已确认（见纪要「决策事项」）

**如果缺少上述任何信息**，会议助手应在企微群中明确标注：

```markdown
【会议助手】需求草稿转化前置检查

⚠️ 以下信息在会议纪要中不完整，暂无法生成标准化需求草稿：
  - 核心诉求不清晰（讨论较发散，缺乏明确功能诉求）
  - 涉及系统模块未确认（需明确"改造哪些模块"）
  - 业务规则未澄清（"异常情况如何处理"尚未讨论）

✅ 建议后续处理：
  1. 召开补充讨论会，确认上述信息
  2. 或由需求方基于纪要手工补齐信息后提交
  3. 完成后重新触发 @会议助手 转化需求草稿
```

### 4.2 需求草稿的输出模板

**当前置条件满足后，会议助手在企微群中发布需求草稿**，格式遵循 Agent-0 的输入要求：

```markdown
【会议助手】需求草稿已生成，可提交 Agent-0 处理

---

## 需求信息

**需求来源**：会议讨论（微信登录优化需求评审，2026-03-31）
**会议纪要链接**：[企微/群文件链接]
**提交时间**：2026-03-31 15:30

---

## 需求背景

用户当前登录体验较差，仅支持账号密码和企业微信两种方式，与竞品相比功能不足。通过新增第三方登录支持（微信扫码），降低用户门槛，提升登录转化率。

**相关背景数据**：
- 当前日均登录失败率：8%
- 竞品的登录方式覆盖：微信、支付宝、抖音等
- 预期通过新功能提升登录成功率至 96%+

---

## 核心诉求

新增**微信扫码登录**功能，与现有账号体系保持独立，不强制用户使用，提供可选的"一键登录"体验。

**需求的 3 个必做事项**（缺一不可）：
1. 支持微信 OAuth2 登录流程集成
2. 支持新账号通过微信登录自动创建
3. 支持已有账号绑定微信账号（可选）

**需求的 3 个必不做事项**（坚决不做）：
1. 不改造现有账号密码登录流程
2. 不强制用户切换登录方式
3. 不支持其他第三方登录（如支付宝、抖音等，后续迭代）

---

## 约束条件

### 技术约束
- 微信第三方应用需通过企业微信审核（预计 1-2 周）
- 现有用户账号体系保持不变，仅新增登录入口
- 需通过企业信息安全审查

### 时间约束
- 微信 OAuth 集成审核：1-2 周
- 功能设计与开发：2-3 周
- 测试与发布：1 周

### 业务约束
- 保持与现有登录体系的向后兼容性
- 不涉及用户数据迁移
```

## 五、常见问题与处理规范

### 5.1 会议纪要缺乏关键信息时

| 缺失信息 | 会议助手的处理方式 | 人类的后续行动 |
|---------|-------------------|--------------|
| 核心诉求模糊不清 | ⚠️ 标注"核心诉求不明确"，不生成需求草稿 | 需求方补充后重新触发 |
| 决策项未确认 | ⚠️ 标注"以下决策项未确认"，列出具体项目 | 人类@相关角色确认后回复 |
| 参会人员未记录 | ⚠️ 标注"部分发言人身份不明确" | 人类补充发言人信息 |
| 行动项无负责人 | ⚠️ 标注"以下行动项缺乏责任人"，列出具体项 | 人类指定责任人后确认 |

### 5.2 会议中需求发散无法聚焦时

**症状**：会议讨论涉及多个不相关的功能或需求，纪要中的"核心诉求"不明确

**会议助手的处理**：
```markdown
【会议助手】会议纪要生成告警

⚠️ 本次会议讨论涉及多个独立需求，无法聚焦：
  - 微信登录（功能1）
  - 账号注销流程优化（功能2）
  - 二次验证增强（功能3）

📋 建议处理方式：
  1. 将上述功能按优先级分拆为 3 个独立需求
  2. 分别为每个需求触发 @需求校准 处理
  3. 或召开补充会议逐个确认后再转化

❌ 暂不生成需求草稿（避免信息混乱）
```

### 5.3 会议中涉及多个角色分工的需求时

**症状**：需求涉及产品、开发、设计、运营等多个角色的工作，边界不清晰

**会议助手的处理**：
```markdown
【会议助手】会议纪要已生成（多角色工作需确认分工）

📌 本需求涉及多个角色的工作，请先确认工作分工后再转化为需求草稿：

| 角色 | 涉及工作 | 确认状态 |
|------|---------|---------|
| 设计 | 新增微信登录界面设计 | ❓ 未明确分配 |
| 开发 | OAuth2 集成与账号绑定逻辑 | ✅ 王五负责 |
| 测试 | 跨浏览器/客户端兼容性测试 | ✅ 赵六负责 |
| 运营 | 用户告知与引导文案 | ❓ 未明确分配 |

💡 建议：
  1. 在会议纪要中明确各角色的分工与责任边界
  2. 确认后重新触发 @会议助手 转化需求草稿
```

### 5.4 会议后临时补充信息时

**症状**：会议生成纪要后，参会人员通过群消息补充了新信息或澄清了歧义

**会议助手的处理**：
```markdown
【会议助手】检测到会议后补充信息

已接收以下补充信息：
  - [用户补充内容]

🔄 已更新会议纪要的相关章节，请确认无误后告知是否需要重新生成需求草稿：
  @会议助手 更新完成，请重新生成需求草稿
```

## 六、与其他 Agent 的协作规范

### 6.1 与 Agent-0（需求校准）的衔接

| 交互阶段 | 会议助手的职责 | Agent-0 的职责 | 衔接点 |
|---------|-------------------|-----------------|--------|
| 会议预约→纪要生成 | 预约会议、采集字幕、生成纪要 | 无 | - |
| 纪要→需求草稿 | 从纪要提取信息、生成初稿 | **必须阅读会议纪要链接**，确认核心诉求、决策、约束条件 | @需求校准 触发前，草稿需包含完整纪要链接 |
| 需求草稿→PRD | 将草稿推送至群，@需求校准 触发 | 基于草稿与纪要，执行标准化处理 | Agent-0 的 PRD 需追溯到会议纪要 |
| PRD Review→确认 | 支撑需求方 Review PRD 内容 | 确保 PRD 与会议决策无偏离 | 若 PRD 与原始决策偏离，会议助手可协助对标回溯 |

### 6.2 与 Workflow-Agent 的协作

当需求进入全流程自动化时，会议助手提供的会议纪要与需求草稿可作为 Workflow-Agent 的输入参考：

```markdown
📎 【会议助手】为 @Workflow-Agent 提供初始化信息

原始会议纪要：[链接]
需求草稿（已通过 Agent-0 校准）：[PRD 文档链接]
关键决策记录：[表格汇总]
参会人员及角色：[列表]

💡 Workflow-Agent 可使用上述信息作为全流程执行的上下文参考
```

### 6.3 与 Agent-7（Backlog 整理）的协作

会议讨论中产生的「待跟进事项」（非当前需求范围但值得后续处理的优化点、技术债务、潜在改进等）应由会议助手在纪要末尾单独列出，并写入全局 Backlog：

| 场景 | 会议助手职责 | Agent-7 职责 |
|------|------------|-------------|
| 会议中出现超出当前需求范围的改进想法 | 在纪要中标注为「待跟进事项」 | 在下次整理时汇总至全局 `docs/backlog/backlog.md` |
| 会议决定搁置某功能点 | 在纪要中记录搁置原因，标注「建议纳入 Backlog」 | 评估优先级后纳入全局 Backlog 排期 |
| 会议产生的技术债务讨论 | 在需求草稿备注中标注「技术债务待跟进」 | 整理为 Backlog 条目，给出优先级建议 |

写入格式：将待跟进事项直接追加至 `docs/backlog/backlog.md`，来源标注为「会议纪要 MID-{会议ID}」，状态设为「待汇总」，等待 Agent-7 下次整理时处理。

### 6.4 与企微群的信息同步

会议助手的所有产出物（纪要、草稿）都应在企微群中**即时推送**，确保：

- **决策可见性**：所有参会人都能看到会议决策
- **信息可追踪**：群消息支持版本管理与修改历史
- **流程透明性**：后续 Agent 流程对所有人可见

```yaml
会议纪要链接: [链接]

```

## 八、会议助手的自我监测与异常处理

### 8.1 纪要质量自检清单

会议助手在生成纪要后，应自动执行以下检查：

| 检查项 | 检查内容 | 是否通过 | 异常处理 |
|--------|---------|---------|---------|
| 字幕覆盖率 | 会议发言的字幕识别覆盖率 ≥ 95% | ☐ 是 ☐ 否 | 若低于 95%，标注异常，建议人工补充 |
| 发言人识别 | 所有发言人身份识别准确率 ≥ 95% | ☐ 是 ☐ 否 | 若存在识别错误，标注并请人工确认 |
| 决策项完整性 | 所有会议中表达的决策都被记录 | ☐ 是 ☐ 否 | 若遗漏，标注缺失项，提示人工补充 |
| 待办项明确性 | 所有待办项都有明确的责任人与 deadline | ☐ 是 ☐ 否 | 若缺失，标注并提示人工补充 |
| 录音链接有效性 | 会议录音可正常访问 | ☐ 是 ☐ 否 | 若失效，标注异常，建议重新上传 |

### 8.2 需求草稿的转化质量检查

| 检查项 | 检查内容 | 是否通过 | 异常处理 |
|--------|---------|---------|---------|
| 前置条件完备 | 见第 4.1 节的前置条件检查清单 | ☐ 是 ☐ 否 | 若不满足，标注并列出缺失项 |
| 必填字段完整 | 见第 4.2 节的必填字段列表 | ☐ 是 ☐ 否 | 若缺失，标注并提示补充 |
| 纪要链接有效 | 需求草稿中包含的会议纪要链接可访问 | ☐ 是 ☐ 否 | 若失效，提示修复链接 |
| 决策项准确映射 | 需求草稿中的决策项与纪要的「决策事项」表一致 | ☐ 是 ☐ 否 | 若不一致，标注差异并重新确认 |

## 九、故障排查与反馈机制

### 9.1 常见问题处理

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 会议链接生成失败 | 腾讯会议 API 调用异常 | 重试预约，若仍失败，提示人类手工预约 |
| 字幕采集中断 | 网络连接丢失 | 自动重连，失败后标注并建议人工补充 |
| 发言人识别错误 | 语音识别模型精度不足 | 标注识别不确定项，提示人工确认 |
| 需求草稿生成失败 | 纪要信息缺失 | 标注缺失的必填信息，提示补充 |

### 9.2 反馈机制

会议助手应支持人类的反馈与修正：

```markdown
@会议助手 更新会议纪要

会议ID: MID-YYYY-NNN
更新内容:
  - 纠正李四的发言（字幕识别有误）
  - 补充遗留问题：[新增遗留问题描述]
  - 新增待办项：[新增行动项]
```

## 十、关键指标与质量目标

会议助手的执行质量应满足以下指标：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 会议链接生成成功率 | 99% | 预约指令下达后，正常生成会议链接 |
| 纪要生成延迟 | ≤ 30 分钟 | 会议结束后 30 分钟内生成结构化纪要 |
| 字幕识别覆盖率 | ≥ 95% | 会议发言的字幕记录完整度 |
| 发言人识别准确率 | ≥ 95% | 发言人身份正确匹配 |
| 需求草稿生成准确度 | ≥ 90% | 草稿内容与会议纪要的一致性 |
| 与 Agent-0 的衔接成功率 | ≥ 95% | 生成的需求草稿能被 Agent-0 直接处理，无需返工 |

## 十一、总结与快速参考

### 11.1 会议助手的三大核心职责

| 职责 | 触发方式 | 产出物 | 质量目标 |
|------|---------|--------|----------|
| 1️⃣ 会议预约 | @会议助手 预约 | 腾讯会议链接 + 日程提醒 | 链接生成成功率 99% |
| 2️⃣ 纪要记录 | 会议自动启动 | 结构化会议纪要 | 生成延迟 ≤ 30 分钟 |
| 3️⃣ 需求转化 | 纪要生成后 | 符合 Agent-0 格式的需求草稿 | 衔接成功率 ≥ 95% |

### 11.2 必填信息速查表

**预约指令的必填字段**：会议主题 | 计划时间 | 预计时长 | 参会人员 | 会议议题

**会议纪要的必填章节**：会议信息 | 讨论要点 | 决策事项 | 原始字幕链接

**需求草稿的必填字段**：需求来源 | 会议纪要链接 | 需求背景 | 核心诉求 | 必做/必不做 | 验收预期 | 会议决策记录

### 11.3 异常处理速查表

| 异常情况 | 处理原则 | 企微群通知内容 |
|---------|---------|--------------|
| 纪要信息不完整 | ⚠️ 标注缺失项，暂不转化需求 | 「以下信息不完整：[列表]」 |
| 需求诉求不清 | ⚠️ 标注模糊项，提示补充讨论 | 「核心诉求不明确，建议[处理建议]」 |
| 决策项未确认 | ⚠️ 标注待确认项，提示人类确认 | 「以下决策项待确认：[列表]」 |
| API 调用失败 | ⚠️ 标注失败原因，提示人工补救 | 「预约失败，原因：[技术原因]，建议[备选方案]」 |
