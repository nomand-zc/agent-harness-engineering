# Agent-7（Backlog 整理 Agent）执行指南

**适用场景**：当需要整理、评估、分发项目跟进处理事项时，遵循本指南。  
**核心目标**：确保项目中散落在各处的「不阻碍当前需求但值得后续处理」的事项得到统一管理，不被遗忘、不无序堆积，同时不干扰当前需求的推进。  
**适用 Agent**：Agent-7（Backlog 整理 Agent）  
**触发方式**：人类手动触发（@ Backlog整理）/ 需求验收通过后事件驱动 / 每周一 09:00 定时触发

---

## 一、触发前的准备步骤

在执行任何整理操作之前，必须先完成以下准备：

### 1.1 确认触发来源与执行范围

| 触发来源 | 执行范围 |
|---------|---------|
| 人类手动触发（@ Backlog整理） | 全量扫描整理 |
| 需求验收通过（事件驱动） | 聚焦刚完成需求的关联 Backlog 条目，同时检查是否有联动排期机会 |
| 定时触发（每周一 09:00） | 全量扫描整理，输出周报摘要 |

### 1.2 读取项目当前状态

执行整理前，必须了解项目当前研发状态，避免给出脱离实际的排期建议：

- 查看 `docs/demand/` 目录，了解当前活跃需求（处理中的需求）和已完成需求；
- 读取 `docs/backlog/backlog.md`（若存在），了解历史 Backlog 现状；
- 如有版本映射表，查看当前版本迭代计划，了解近期待排期空间。

---

## 二、整理执行流程（共 8 步）

### 第1步：扫描收集新增条目

遍历 `docs/demand/` 下所有需求目录，收集各需求目录下 `backlog.md` 中状态为「待汇总」的条目。

```
docs/demand/
├── REQ-2026-001/
│   └── backlog.md   ← 扫描此文件，收集「待汇总」条目
├── REQ-2026-002/
│   └── backlog.md
└── ...
```

同时收集 `docs/backlog/backlog.md` 中人类直接写入或第三方系统接入的新条目（若有标注「待整理」状态）。

### 第2步：去重合并

识别内容高度相似的条目（描述相近、影响范围相同），合并为一条：
- 合并时保留所有来源引用（多个来源的条目合并后，来源字段中列出全部来源文档）
- 合并时以「描述最详细」的版本作为主条目，其余作为补充说明

### 第3步：分类整理

按类型（重构/增强/技术债务/可优化点/补充性需求/其他）重新组织，便于人类快速定位关注的类型。

### 第4步：优先级评估

基于以下维度为每条 Backlog 条目评估或调整优先级：

| 评估维度 | 说明 |
|---------|------|
| **影响范围** | 涉及模块越多、越核心，优先级越高 |
| **不处理的风险** | 长期不处理是否会持续劣化？是否会影响后续需求的推进？ |
| **处理收益** | 处理后的改善效果是否明显？是否能解锁后续能力？ |
| **处理难度** | 估算处理成本，避免将小优化排在大工程之前虚耗排期空间 |
| **时效性** | 是否有外部截止日期（如合规要求）？ |

优先级判定结果若与原始来源 Agent 的建议不同，需在条目中注明调整理由。

### 第5步：排期建议

为 P0/P1/P2 条目给出排期建议：

- **P0**：立即通知人类，不走常规排期
- **P1**：建议在某个具体需求完成后处理，给出推荐的排期窗口
- **P2**：建议纳入下个迭代计划，给出大致时间窗口
- **P3**：标注「积累中，待定期复盘评估」

排期建议格式：
```
建议在 REQ-2026-003 完成后处理（预计下周排期）
```

### 第6步：更新需求级 backlog.md 状态

将已完成汇总的条目状态从「待汇总」更新为「已汇总」，并在条目中注明：
```
| **状态** | 已汇总（Agent-7 于 YYYY-MM-DD 汇总至全局 backlog.md） |
```

### 第7步：更新全局 backlog.md

将整理结果写入 `docs/backlog/backlog.md`，按优先级分区组织（见《Backlog 管理规范》第八章模板）。

### 第8步：输出整理报告并通知人类

向企微群推送整理报告，格式如下：

```markdown
## Backlog 整理报告（YYYY-MM-DD）

**本次整理范围**：全量扫描 / 需求 {需求ID} 完成触发 / 定时触发  
**新增条目**：{N} 条  
**合并条目**：{N} 条  
**当前待处理总数**：{N} 条

### P0 紧急（需立即处理）
{列出 P0 条目，无则写「无」}

### P1 高优先级（建议近期排期）
{列出 P1 条目摘要}

### P2 中优先级（下个迭代）
{列出 P2 条目数量，不展开详情}

### 建议下一步行动
- [ ] 人类确认 P0 处理方案
- [ ] 人类确认 P1 排期计划（如同意，可直接 @ 需求校准 触发 Agent-0）
- 全量详情见：docs/backlog/backlog.md
```

## 三、Backlog 转化为需求的操作规范

### 3.1 人类确认排期后的操作

当人类确认某条 Backlog 条目排期后，Agent-7 执行以下操作：

1. 将条目状态更新为「已排期」，记录确认时间和确认人
2. 将条目内容整理为需求草稿（符合 Agent-0 输入格式）：
   - 标题：条目标题
   - 背景：来源描述 + 影响范围 + 「该事项在 {来源需求} 处理阶段发现，为避免当时需求复杂度过高，延后处理」
   - 核心目标：建议处理方式
3. 在企微群通知人类：「已将 {条目标题} 整理为需求草稿，可直接 @ 需求校准 触发 Agent-0 处理」
4. 将需求草稿写入 `docs/backlog/pending-requirements/` 目录，供人类确认后触发 Agent-0

### 3.2 转化后的状态追踪

| 状态 | 说明 |
|------|------|
| 已排期 | 人类确认排期，等待触发 Agent-0 |
| 处理中 | Agent-0 已介入，标准需求流程进行中，关联需求ID |
| 已完成 | 对应需求已验收通过 |
| 已关闭 | 决定不处理（注明关闭原因，如「已被其他需求覆盖」「评估后价值不足」） |

## 四、特殊场景处理

### 4.1 P0 紧急条目的处理

P0 条目意味着「不处理可能影响线上稳定性或数据安全」，处理优先级高于常规排期：

1. 发现 P0 条目后，立即在企微群 @人类技术负责人，不等定时整理
2. 等待人类响应，确认是否立即处理（不走常规排期）
3. 若确认立即处理，触发紧急需求流程（跳过部分评审，直接 Agent-3 介入）
4. 若人类判断可延期，调整为 P1 并正常排期

### 4.2 条目数量过多时的处理

若 P1 以上条目积累超过 10 条，向人类发出预警：
```
⚠️ 当前 Backlog 中 P1 及以上条目已积累 {N} 条，建议安排专项 Backlog Sprint 集中处理。
```

### 4.3 与外部第三方系统的接入处理

当 `docs/backlog/imports/` 目录下存在外部导入文件时：
1. 解析文件格式（CSV/JSON），提取条目信息
2. 按标准字段规范转化为 Backlog 条目格式
3. 写入全局 `backlog.md`，来源字段标注「第三方导入」
4. 通知人类确认转化结果

## 五、与其他 Agent 的协作规范

### 5.1 接收条目（被动收集）

Agent-7 不主动介入其他 Agent 的工作流程，而是等待其他 Agent 将条目写入需求级 `backlog.md`。

其他 Agent 写入 Backlog 条目时的规范：
- **写入时机**：各 Agent 完成自身任务时（不是任务进行中），作为收尾步骤写入
- **写入位置**：当前需求目录下的 `backlog.md`
- **初始状态**：必须设为「待汇总」

### 5.2 向 Agent-0 移交（主动分发）

当 Backlog 条目需要转化为需求时，Agent-7 负责：
1. 准备需求草稿
2. 通知人类确认
3. 人类确认后，将草稿放入指定目录，通知人类触发 Agent-0

Agent-7 **不直接调用 Agent-0**，维持人类作为最终授权节点的原则。
