# 验收标准文档编写指南

**适用场景**：当需要为一个需求输出验收标准文档（`criteria-V{版本号}.md`）时，遵循本指南。  
**核心目标**：验收标准文档输出后，Agent-4 能够无歧义地逐项执行验收测试，所有验收项均可量化、可自动化，不存在"说不清楚算不算通过"的模糊项。

**适用 Agent**：Agent-1（高级架构师，负责编写验收标准文档）。编写过程中发现的不影响当前验收的优化建议，写入 `docs/demand/{需求ID}/backlog.md`，由 Agent-7（Backlog 整理 Agent）统一排期。

---

## 一、编写前的强制准备步骤

在写任何一条验收项之前，必须先完成以下准备，否则验收标准必然产生遗漏或无法执行。

### 1.1 确认设计方案已完成

验收标准文档必须基于已完成的架构设计方案编写，不可超前或独立编写：

- 确认 `docs/demand/{需求ID}/design/architecture-V{版本号}.md` 已存在且包含完整的接口定义、核心逻辑和边界条件；
- 验收标准中的每条 AC 必须能追溯到设计方案中的具体章节，不可凭空编写。

> **强制约束**：没有设计方案就写验收标准，等于在不知道「做什么」的情况下定义「做对了的标准」，必然产生漏项或与实现不符。

### 1.2 阅读项目测试规范

| 文档 | 必读内容 |
|------|---------|
| `docs/TESTING.md` | 单元测试规范、集成测试规范、基准测试写法、mock 策略 |
| `docs/ARCHITECTURE.md` | 验收约束、性能基线要求 |

**不阅读测试规范直接编写验收标准，等于制定了项目无法执行的验收规则。**

### 1.3 梳理需求的验收维度

在编写具体验收项前，先梳理本次需求涉及哪些验收维度，避免遗漏：

- [ ] 业务场景验收（正向/异常/边界）
- [ ] 接口验收（入参校验、出参格式、错误码）
- [ ] 性能验收（响应时间、并发量、资源占用）
- [ ] 安全验收（权限校验、数据脱敏、防注入）
- [ ] 兼容性验收（环境兼容、版本兼容）
- [ ] 并发安全验收（data race、分布式一致性）

不涉及的维度在文档中明确标注「不适用（原因）」，不可留白。

## 二、验收标准文档的必要章节

一份合格的验收标准文档必须包含以下所有章节，缺少任何一章都可能导致验收执行产生歧义。

### 章节清单

| 章节 | 作用 | 缺失后果 |
|------|------|---------|
| 文档基础信息 | 需求 ID、关联架构版本、基准 Commit、生效时间 | 无法溯源，文档孤岛 |
| 验收总则 | 验收范围、验收工具、验收环境、通过判定规则 | Agent-4 不知道验收边界和通过条件 |
| 业务验收 | 正向场景、异常场景、边界场景，逐条可执行 | 核心业务逻辑未覆盖 |
| 接口验收 | 入参校验、出参格式、错误码、接口稳定性 | 接口行为未约束，实现可能偏差 |
| 性能验收 | 并发量、吞吐量、响应时间、资源占用量化阈值 | 性能要求未量化，无法判断是否达标 |
| 安全验收 | 权限校验、数据脱敏、防注入、漏洞扫描标准 | 安全要求缺失，上线存在安全风险 |
| 兼容性验收 | 环境兼容、版本兼容、依赖兼容 | 多环境部署可能失败 |
| 验收结果判定标准 | 全量通过条件、不通过整改要求、重试规则 | 验收结论无法客观判定 |

## 三、文档基础信息规范

文档头部必须包含以下所有字段，字段缺失不可提交：

```markdown
**需求 ID**：REQ-{需求ID}  
**关联架构版本**：architecture-V{版本号}.md  
**基准 Commit**：{设计方案对应的 commit hash}  
**验收标准版本**：V{版本号}  
**生效时间**：YYYY-MM-DD  
**编写 Agent**：Agent-1  
**归档路径**：docs/demand/{需求ID}/acceptance/criteria-V{版本号}.md  
```

## 四、验收总则规范

### 4.1 验收范围声明

明确本次验收覆盖哪些功能点，以及**不覆盖**哪些内容（不可留白）：

```markdown
## 验收总则

### 验收范围
本次验收覆盖以下功能点（对应架构设计方案中的需求点）：
- {功能点1}（对应设计方案第{N}章）
- {功能点2}（对应设计方案第{N}章）

### 不在验收范围内
- {排除项1}（原因：{为何排除}）
- {排除项2}（原因：{为何排除}）
```

### 4.2 验收工具与环境

```markdown
### 验收工具
| 工具 | 用途 | 版本要求 |
|------|------|---------|
| {测试框架} | 单元测试/集成测试 | {版本} |
| {性能测试工具} | 并发/压测 | {版本} |
| {安全扫描工具} | 漏洞扫描 | {版本} |

### 验收环境
- 执行环境：{开发/测试/预生产}
- 数据库版本：{版本}
- 依赖服务：{列出外部依赖}
```

### 4.3 通过判定规则

通过判定规则必须在文档中明确声明，不可由 Agent-4 自行判断：

```markdown
### 验收通过判定规则
满足以下全部条件，视为验收通过：
- [ ] 所有业务验收项：100% 通过
- [ ] 所有接口验收项：100% 通过
- [ ] 性能验收：全部指标达到量化阈值
- [ ] 安全验收：无致命/严重安全问题
- [ ] 兼容性验收：所有目标环境均通过

以下情况视为不通过，需整改后重新验收：
- 任意业务验收项未通过
- 性能指标未达阈值
- 存在致命/严重安全问题
```

## 五、业务验收规范

### 5.1 每条业务验收项的结构

每条验收项**必须**包含以下字段，缺失字段不可提交：

```markdown
**AC-{大类}.{序号}** {验收项标题（一句话）}

- **验收条件**：{什么情况下算通过，一句话，包含可量化的判断标准}
- **前置条件**：{执行此验收项前需要准备的数据或状态}
- **执行步骤**：
  1. {步骤1}
  2. {步骤2}
- **预期结果**：{具体的输出值、状态变化、返回内容}
- **测试文件**：`{测试文件路径}`
- **测试函数**：`{TestXxx_Yyy}`
- **执行命令**：`{go test -run TestXxx_Yyy ./path/...}`
```

### 5.2 必须覆盖的场景类型

| 场景类型 | 说明 | 不可遗漏的典型用例 |
|---------|------|-----------------|
| 正向场景 | 典型输入下的预期正常输出 | 至少 1 个完整业务流程的端到端用例 |
| 异常场景 | 输入非法、依赖服务故障、数据不存在 | 存储层返回 error、输入为 nil/空、key 不存在 |
| 边界场景 | 极值、临界值、特殊组合 | 零值、最大值、空集合、分母为零、负数钳制 |

### 5.3 业务验收写法示例

**错误写法（模糊，无法执行）**：
> AC-1.1：指标计算正确

**正确写法（具体可执行）**：
> **AC-1.1** `EffectiveAvailable` 在账号全部可用时返回总数
>
> - **验收条件**：当所有账号 `Status == Available` 且无限流规则触发时，`EffectiveAvailable` 等于 `TotalAccounts`
> - **前置条件**：初始化 3 个账号，Status 均为 Available，UsageRule.Total 均为 0（无限额）
> - **执行步骤**：
>   1. 调用 `observer.CollectMetrics(ctx)`
>   2. 读取返回的 `Metrics.EffectiveAvailable`
> - **预期结果**：`EffectiveAvailable == 3`，`TotalAccounts == 3`
> - **测试文件**：`observer/observer_test.go`
> - **测试函数**：`TestCollector_EffectiveAvailable_AllAvailable`
> - **执行命令**：`go test -run TestCollector_EffectiveAvailable ./observer/...`

## 六、接口验收规范

### 6.1 入参校验验收

对每个对外接口，必须验收以下入参场景：

| 场景 | 必须包含的用例 |
|------|-------------|
| 合法入参 | 典型合法输入，返回预期结果 |
| 非法入参 | nil、空字符串、负数、超长字符串，返回明确错误码 |
| 边界入参 | 最大值、最小值、特殊字符 |

### 6.2 出参格式验收

```markdown
**AC-{大类}.{序号}** `{接口名}` 出参格式符合规范

- **验收条件**：返回值包含所有必要字段，字段类型与接口定义一致，不含多余字段
- **预期结果**：
  - 成功时：`{预期 JSON/结构体格式}`
  - 失败时：返回 `{错误类型}`，错误信息包含 `{关键词}`
```

### 6.3 错误码验收

所有错误码必须在验收标准中明确列出，不可由实现者自行决定：

```markdown
| 错误场景 | 错误码 | 错误信息 |
|---------|--------|---------|
| key 不存在 | ErrNotFound | "not found: {key}" |
| 存储层故障 | ErrStorageUnavailable | "storage unavailable" |
| 入参为空 | ErrInvalidInput | "invalid input: {field} is required" |
```

## 七、性能验收规范

### 7.1 性能指标必须量化

所有性能验收项必须有明确的数值阈值，禁止使用「尽量快」「不能太慢」等模糊表述：

**错误写法**：
> 接口响应时间应该比较快

**正确写法**：
> 接口响应时间 P99 ≤ 500ms（并发 100 QPS 下）

### 7.2 性能验收项模板

```markdown
**AC-PERF.{序号}** {性能指标名称}

- **验收条件**：{量化阈值，含并发量、数据量等前提条件}
- **测试场景**：
  - 并发数：{N} goroutine
  - 数据量：{N} 条记录
  - 持续时长：{N} 秒
- **通过标准**：
  - P50 ≤ {N}ms
  - P99 ≤ {N}ms
  - CPU 占用峰值 ≤ {N}%
  - 内存占用峰值 ≤ {N}MB
- **测试文件**：`{测试文件路径}`
- **测试函数**：`Benchmark{Xxx}`
- **执行命令**：`go test -bench=Benchmark{Xxx} -benchtime=10s ./path/...`
```

### 7.3 必须覆盖的性能场景

| 场景 | 说明 |
|------|------|
| 正常负载 | 典型 QPS 下的响应时间和资源占用 |
| 峰值负载 | 最大设计并发量下的表现 |
| 长时间运行 | 持续运行下的内存增长（检查内存泄漏） |
| 冷启动 | 首次请求（无缓存）的响应时间 |

## 八、安全验收规范

### 8.1 必须覆盖的安全场景

| 安全类型 | 必须验收的场景 |
|---------|-------------|
| 权限校验 | 无权限访问返回 401/403，有权限访问正常 |
| 数据脱敏 | 敏感字段（手机号、密码、Token）在日志/响应中不明文输出 |
| 输入校验 | SQL 注入、XSS、命令注入等恶意输入被拦截 |
| 密钥安全 | 密钥/Token 不存储在代码中，不出现在日志中 |

### 8.2 安全验收项模板

```markdown
**AC-SEC.{序号}** {安全验收项标题}

- **验收条件**：{安全要求的通过标准}
- **测试方法**：{手动测试 / 自动化扫描工具名称}
- **测试步骤**：
  1. {步骤1}
  2. {步骤2}
- **预期结果**：{安全处理后的预期行为}
- **不通过条件**：{明确说明什么情况判定为不通过}
```

## 九、兼容性验收规范

### 9.1 环境兼容性

```markdown
**AC-COMPAT.{序号}** 多环境部署兼容

- **验收条件**：在所有目标环境中部署并运行，核心功能无异常
- **目标环境**：
  - [ ] 开发环境（{配置说明}）
  - [ ] 测试环境（{配置说明}）
  - [ ] 预生产环境（{配置说明}）
- **验证方法**：在每个环境执行业务验收的正向场景用例
- **通过标准**：所有目标环境的正向场景 100% 通过
```

### 9.2 版本兼容性

若本次需求修改了对外接口，必须验收向下兼容性：

```markdown
**AC-COMPAT.{序号}** 接口向下兼容

- **验收条件**：旧版本客户端调用新版本接口，不产生破坏性错误
- **验证方法**：使用旧版本入参格式调用新接口
- **通过标准**：旧入参格式正常处理，不返回 4xx/5xx 错误
- **不适用条件**：本次需求为全新接口（无历史版本），标注「不适用」
```

## 十、并发安全验收规范

凡是涉及并发、后台 goroutine 或跨实例共享状态的功能，必须包含以下验收项：

### 10.1 data race 检测

```markdown
**AC-CONC.1** 并发读写无 data race

- **验收条件**：并发场景下使用 `-race` 标志运行测试，无 data race 报告
- **执行命令**：`go test -race ./path/...`
- **通过标准**：测试输出中无 `DATA RACE` 字样
```

### 10.2 goroutine 生命周期

```markdown
**AC-CONC.2** goroutine 正常退出，无泄漏

- **验收条件**：调用 `Stop()` 后，所有后台 goroutine 在 {N}ms 内退出
- **验证方法**：调用 Stop() 前后对比 `runtime.NumGoroutine()` 数量
- **通过标准**：Stop() 后 goroutine 数量恢复到初始水平（允许 ±{N} 波动）
```

### 10.3 多实例一致性（集成测试，适用时）

```markdown
**AC-CONC.3** 多实例写入同一存储时结果一致

- **验收条件**：N 个实例同时写入，最终存储中数据不丢失、不重复（按语义定义）
- **前置条件**：启动 {N} 个实例，共享同一存储后端
- **执行步骤**：并发写入 {N} 条数据，等待所有写入完成后读取
- **通过标准**：读取结果符合存储语义定义（upsert/append/insert）
```

## 十一、验收结果判定标准规范

### 11.1 通过分级

```markdown
## 验收结果判定标准

| 结论 | 判定条件 |
|------|---------|
| 通过 | 所有验收维度 100% 达标，无致命/严重问题 |
| 有条件通过 | 核心功能 100% 达标，存在「一般」级问题，整改后无需重新验收 |
| 不通过 | 任意业务/接口/性能/安全验收项未达标，或存在致命/严重问题 |
```

### 11.2 重试规则

```markdown
### 不通过整改规则
- 验收不通过时，Agent-3 根据验收报告中的整改建议修改代码，重新提交验收；
- 最多重试 3 次；第 3 次仍不通过，触发人类介入；
- 每次重试必须针对上次验收报告中的所有不通过项进行修改，不可遗漏；
- 若不通过原因为「验收标准不合理」，由人类修改验收标准文档后重新触发验收，修改记录绑定需求 ID。
```

## 十二、常见编写错误示例

以下是验收标准文档中容易出现的无效写法，须避免：

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 验收项模糊 | 「功能正常运行」 | 给出具体输入、操作、预期输出 |
| 指标未量化 | 「性能良好」 | 给出具体阈值，如「P99 ≤ 500ms」 |
| 缺少执行命令 | 只写测试场景，无执行命令 | 每组 AC 必须给出可直接执行的命令 |
| 遗漏错误路径 | 只写正向场景 | 必须覆盖异常场景和边界场景 |
| 遗漏并发验收 | 涉及并发的功能无 data race 检测 | 必须包含 `-race` 验收项 |
| 验收项无法追溯 | 验收项与设计方案无关联 | 每条 AC 必须对应设计方案中的具体章节 |

## 十三、验收标准签名规范

验收标准文档完成后，必须由相关责任方进行签名确认，作为「验收标准可交付 Agent-4 执行」的前置门禁。

### 13.1 签名表格

在验收标准文档末尾添加以下签名表：

```markdown
## 验收标准签名

| 角色 | 签名 | 确认时间 | 确认内容 |
|------|------|---------|---------|
| 编写者 | Agent-1 | YYYY-MM-DD HH:MM | 已完成强制准备步骤，所有验收维度均已覆盖，每条 AC 均可无歧义执行 |
| 产品/技术负责人 | {人类负责人} | YYYY-MM-DD HH:MM | 验收范围与需求目标一致，性能/安全等关键阈值已确认 |
```

> **签名规则**：
> - **编写者签名**（必填）：Agent-1 完成验收标准文档后自签，确认已遵循本指南所有强制步骤；
> - **产品/技术负责人签名**（条件必填）：当验收标准涉及以下任一情形时，人类负责人签名为**必填**，否则为可选：
>   - 包含性能验收阈值（如 P99、QPS、内存上限等量化指标）；
>   - 包含安全验收要求（权限、脱敏、漏洞扫描标准）；
>   - 涉及接口向下兼容性约束；
>   - 验收通过判定规则存在争议性设定（如「有条件通过」的范围）。

### 13.2 签名前的自检清单

编写者在签名前必须逐项确认：

- [ ] 已确认关联的架构设计方案已完成，每条 AC 均可追溯到设计方案具体章节；
- [ ] 已阅读 `docs/TESTING.md`，所有执行命令符合项目测试规范；
- [ ] 不在验收范围内的维度已明确标注「不适用（原因）」，无留白；
- [ ] 每条 AC 包含完整字段：验收条件、前置条件、执行步骤、预期结果、测试文件、测试函数、执行命令；
- [ ] 业务验收已覆盖正向、异常、边界三类场景；
- [ ] 涉及并发的功能已包含 `-race` 检测和 goroutine 生命周期验收项；
- [ ] 所有性能指标均有量化阈值，无「尽量快」等模糊表述；
- [ ] 验收结果判定标准（通过/有条件通过/不通过）已明确声明。

### 13.3 签名追溯说明

> 验收标准签名是验收质量的责任起点。若后续 Agent-4 执行验收时发现验收项存在重大遗漏、无法执行、或与实际需求不符，将追溯至编写者签名责任，要求编写者对本指南进行反思迭代。涉及人类负责人签名的性能/安全阈值，其合理性由人类负责人承担追溯责任。
