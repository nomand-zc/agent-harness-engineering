# 设计方案评审文档编写指南

**适用场景**：当需要对一份架构设计方案输出评审文档时，遵循本指南。  
**核心目标**：评审文档输出后，问题描述清晰可执行，修改方向明确无歧义，不产生「评审了但改不动」的情况。

**适用 Agent**：Agent-2（平级评审）、安全专家 Agent、性能专家 Agent、合规专家 Agent。

---

## 一、评审前的强制准备步骤

在写任何一条评审意见之前，必须先完成以下准备，否则评审意见必然产生误判或遗漏。

### 1.1 绑定被评审的设计文档版本

- 确认被评审的设计方案文档路径与版本号（如 `docs/demand/{需求ID}/design/architecture-V1.0.0.md`）；
- 确认文档头部包含：需求 ID、关联架构版本、基准 Commit；
- 若文档头部缺失上述锚点信息，**评审前须先要求 Agent-1 补充**，不可跳过。

> **强制约束**：针对没有版本锚点的文档进行评审，等于在无法溯源的基础上讨论问题，后续修改无法验证是否已覆盖所有评审意见。

### 1.2 阅读项目规范文档（评审基线）

评审意见必须有据可依，不能凭感觉判断。评审前必须阅读：

| 文档 | 用途 |
|------|------|
| `docs/ARCHITECTURE.md` | 判断技术选型是否符合项目约束、模块边界是否合理 |
| `docs/CONVENTIONS.md` | 判断命名规范、接口风格是否符合项目规范 |
| `docs/TESTING.md` | 判断验收标准是否可自动化执行 |
| `docs/demand/{需求ID}/PRD.md` | 判断设计方案是否覆盖所有需求点 |

**不阅读规范直接评审，等于用个人偏好代替项目标准，产生无效评审意见。**

### 1.3 理解需求背景与设计意图

- 阅读需求 PRD，明确需求的核心目标和边界；
- 重点理解设计方案的「需求溯源」和「边界定义」章节，避免评审超出需求范围的内容；
- 检查 `docs/demand/` 目录下是否有历史设计文档，评估本次设计与历史方案的一致性。

### 1.4 探索现有代码库（不可跳过）

评审的核心价值之一是发现「方案与现有代码库的冲突」。若不探索代码库，评审意见只能停留在文档层面，无法发现真实的接口冲突、循环依赖或重复实现。

必须完成以下探索：

**理解项目现状**
- 阅读 `README.md` 或项目顶层文档，了解项目核心目标、使用场景；
- 梳理当前已实现的功能模块，确认新需求在整体功能版图中的位置是否与设计方案描述一致。

**验证方案中的依赖关系**
- 检查设计方案涉及的包/模块在代码库中的实际位置；
- 验证依赖方向：方案声称的依赖关系在代码中是否可行，是否存在潜在循环依赖；
- 检查方案中的接口/类型是否与现有代码中的定义冲突或重复。

**核实方案的可落地性**
- 检查方案中引用的现有接口签名是否与代码库中一致；
- 确认方案声称「参照现有 Pattern」的部分，现有 Pattern 是否确实存在且与方案描述匹配；
- 评估方案对现有接口的修改会影响哪些调用方，判断影响范围是否在方案中已声明。

> **强制约束**：跳过代码库探索直接评审，等于只做「文档合规性检查」，无法发现方案与现有代码的真实冲突，评审价值大打折扣。

## 二、评审文档的必要章节

一份合格的评审文档必须包含以下所有章节，缺少任何一章都可能导致评审意见无法落地。

### 章节清单

| 章节 | 作用 | 缺失后果 |
|------|------|---------|
| 文档基础信息 | 需求 ID、评审文档版本、被评审文档版本、评审时间、评审 Agent | 无法溯源，文档孤岛 |
| 评审基线 | 绑定被评审文档版本、评审范围、评审标准来源 | 评审意见缺乏依据 |
| 评审维度打分 | 各维度量化评分（满分 100，80 分及以上通过） | 无法判断是否通过，无法横向比较 |
| 问题分级明细 | 致命/严重/一般/优化四级，每条含原文引用+风险说明+修改建议 | 修改者不知道改哪里、怎么改 |
| 评审结论 | 通过/不通过，不通过须注明核心原因与阻塞项 | 无法触发后续流程 |
| 评审追溯 | 评审 Agent 签名、复核人、归档路径 | 评审记录不可追溯 |

## 三、文档基础信息规范

文档头部必须包含以下所有字段，字段缺失不可提交：

```markdown
**需求 ID**：REQ-{需求ID}  
**评审类型**：平级评审 / 安全专项 / 性能专项 / 合规专项  
**被评审文档**：docs/demand/{需求ID}/design/architecture-V{版本号}.md  
**被评审文档基准 Commit**：{commit hash}  
**评审文档版本**：V1.0.0  
**评审时间**：YYYY-MM-DD  
**评审 Agent**：Agent-2 / 安全专家 Agent / 性能专家 Agent / 合规专家 Agent  
**归档路径**：docs/demand/{需求ID}/review/{review-type}-review-V1.0.0.md  
```

## 四、评审基线声明规范

评审基线章节必须明确以下三项，确保评审有据可依：

### 4.1 绑定被评审文档版本

```markdown
本次评审基于以下文档版本：
- 设计方案：architecture-V1.0.0.md（Commit: {hash}）
- 验收标准：criteria-V1.0.0.md（Commit: {hash}）
```

### 4.2 声明评审范围

评审范围必须与评审类型对应，不可超范围或遗漏：

| 评审类型 | 评审范围 |
|----------|---------|
| 平级评审（Agent-2） | 架构合理性、可扩展性、模块边界、接口设计、验收标准完整性 |
| 安全专项 | 认证鉴权、数据加密、输入校验、权限边界、密钥管理、日志脱敏 |
| 性能专项 | 并发模型、存储读写路径、缓存策略、N+1 查询、资源占用上限 |
| 合规专项 | 数据合规、隐私保护、审计日志、保留期限、跨境传输合规 |

### 4.3 声明评审标准来源

```markdown
评审依据：
- docs/ARCHITECTURE.md（设计约束与模块边界）
- docs/CONVENTIONS.md（命名规范与接口风格）
- docs/TESTING.md（测试规范与验收标准要求）
- 项目级安全/性能/合规基线（如适用）
```

## 五、评审维度打分规范

### 5.1 打分维度

满分 100 分，**80 分及以上为通过**，低于 80 分为不通过，须注明失分项。

| 评审类型 | 打分维度 | 权重说明 |
|----------|---------|---------|
| 平级评审 | 架构合理性（25）、可扩展性（25）、可落地性（25）、验收标准完整性（25） | 四维均等权重 |
| 安全专项 | 认证鉴权（30）、数据安全（30）、输入校验（20）、审计与监控（20） | 偏重认证与数据安全 |
| 性能专项 | 并发模型（30）、存储访问效率（30）、资源占用可控性（20）、性能基线可量化（20） | 偏重并发与存储 |
| 合规专项 | 数据合规（40）、隐私保护（30）、审计日志（30） | 偏重数据合规 |

### 5.2 打分格式

```markdown
## 评审维度打分

| 评审维度 | 得分（满分25） | 失分原因摘要 |
|---------|--------------|------------|
| 架构合理性 | 20 | 模块间依赖方向存在潜在循环（见问题 P-001） |
| 可扩展性 | 23 | 扩展点已预留，接口设计合理 |
| 可落地性 | 18 | 多处接口签名不完整，实现者需猜测参数语义（见 P-002、P-003） |
| 验收标准完整性 | 15 | 缺少并发安全和错误路径的测试场景（见 P-004） |

**总分：76 / 100（不通过）**
```

## 六、问题分级明细规范

### 6.1 问题分级标准

| 问题级别 | 判定标准 | 处置要求 |
|----------|---------|---------|
| 致命 | 架构根本无法满足需求，或存在数据安全风险，继续开发必然失败 | 必须修改，修改前不可进入开发 |
| 严重 | 严重影响核心功能或性能指标，可能导致线上故障或数据错误 | 必须修改，限期整改（48 小时内） |
| 一般 | 影响代码可维护性或边缘场景处理，不影响核心功能 | 建议修改，纳入当前版本整改 |
| 优化 | 有更优实现方案，但当前方案可落地，不影响核心功能 | 可选修改，不强制 |

### 6.2 每条问题的必要字段

每条评审问题**必须**包含以下所有字段，缺失字段不可提交：

```markdown
**P-{序号}**【{问题级别}】{问题标题（一句话）}

- **原文引用**：> {被评审文档中的原文，含章节定位，如「第三章 3.3 - 跨模块共享类型的归属原则」}
- **风险说明**：{此问题如不修改，会导致什么具体后果？必须说明影响范围}
- **修改建议**：{具体的修改方向，Agent-1 可直接执行，无模糊表述}
- **参考依据**：{来自哪条规范或哪个先例，如「docs/ARCHITECTURE.md 第2章禁止 storage 包依赖业务层包」}
```

### 6.3 问题描述的写法规范

**错误写法（模糊，无法执行）**：
> 接口设计不够清晰，建议改进。

**正确写法（具体可执行）**：
> **P-002**【严重】`MetricStore.GetCurrentUsages` 接口未说明 key 不存在时的行为
>
> - **原文引用**：> 第五章「接口定义」- `GetCurrentUsages(ctx, keys []string) ([]*Metrics, error)`
> - **风险说明**：当 keys 中存在未初始化的账号时，实现者无法判断应返回 `ErrNotFound`、零值 `Metrics` 还是跳过，导致不同后端实现行为不一致，调用方可能出现 nil pointer panic
> - **修改建议**：补充接口注释：「若某个 key 不存在，对应位置返回零值 `Metrics{}` 而非 error；仅存储层故障时返回 error」
> - **参考依据**：`docs/guide/solution-design-guide.md` 第四章 4.3「存储语义必须明确」

### 6.4 问题数量的合规边界

| 问题级别 | 评审通过的最大允许数量 |
|----------|---------------------|
| 致命 | 0 |
| 严重 | 0 |
| 一般 | ≤ 3 |
| 优化 | 不限 |

超出上述上限，评审结论必须为「不通过」。

## 七、评审结论规范

### 7.1 结论格式

```markdown
## 评审结论

**结论**：通过 / 不通过

**总分**：{分数} / 100

**核心阻塞项**（仅不通过时填写）：
- P-{序号}（{级别}）：{一句话说明为何阻塞}

**通过条件**（仅不通过时填写）：
- [ ] P-001 已修改并重新提交评审
- [ ] P-002 已修改并重新提交评审

**本次评审有效期**：本结论基于 architecture-V{版本号}.md（Commit: {hash}），设计文档更新后须重新评审。
```

### 7.2 通过判定规则

满足以下**全部**条件，方可给出「通过」结论：

- [ ] 总分 ≥ 80；
- [ ] 致命问题：0 条；
- [ ] 严重问题：0 条；
- [ ] 一般问题：≤ 3 条；
- [ ] 设计方案覆盖 PRD 所有需求点；
- [ ] 验收标准所有指标均已量化，无模糊表述。

以上任一条件不满足，结论为「不通过」，须注明阻塞项。

## 八、各评审类型的专项检查清单

### 8.1 平级评审（Agent-2）检查清单

**架构合理性**
- [ ] 模块边界清晰，职责单一，无越权访问
- [ ] 依赖方向符合项目架构分层约束（`docs/ARCHITECTURE.md`）
- [ ] 无循环依赖（新模块的依赖方向已分析）
- [ ] 跨模块共享类型的归属已明确
- [ ] 接口设计遵循项目现有 Pattern，无重复定义

**可落地性**
- [ ] 所有接口签名完整（含参数类型完整包路径、注释）
- [ ] 核心逻辑以伪代码或流程图描述，非纯文字
- [ ] 错误处理策略逐一说明（失败时跳过/终止/返回 error）
- [ ] 边界条件已枚举（零值、空集合、分母为零等）
- [ ] 存储语义明确（upsert/append/insert）

**验收标准完整性**
- [ ] 所有指标可量化（如「接口响应时间 ≤ 500ms」），无模糊表述
- [ ] 覆盖正常路径、边界条件、错误路径、并发安全
- [ ] 每组 AC 指定了测试文件路径和执行命令
- [ ] 测试用例有具体的输入和预期输出

### 8.2 安全专项检查清单

**认证与鉴权**
- [ ] 所有外部接口有明确的认证机制（API Key / Token / OAuth 等）
- [ ] 权限边界已定义，防止越权访问
- [ ] 敏感操作有二次鉴权或审批机制

**数据安全**
- [ ] 敏感数据（密码、密钥、Token）存储加密，传输加密
- [ ] 日志中无敏感信息明文输出（脱敏处理）
- [ ] 数据访问路径有最小权限原则保障

**输入校验**
- [ ] 所有外部输入有明确的校验规则（格式、长度、类型）
- [ ] 防 SQL 注入、XSS、命令注入等常见漏洞已在方案中声明

**审计与监控**
- [ ] 关键操作（增删改）有审计日志
- [ ] 安全事件有告警机制

### 8.3 性能专项检查清单

**并发模型**
- [ ] goroutine 生命周期已明确（何时创建、何时退出）
- [ ] 并发安全机制已声明（mutex/channel/原子操作）
- [ ] 分布式场景下的多实例并发策略已定义（Leader 选举、分布式锁）

**存储访问效率**
- [ ] 无 N+1 查询问题（批量接口已评估）
- [ ] 热点路径有缓存策略，缓存失效机制已定义
- [ ] 存储写入频率已评估，无不必要的高频写入

**资源占用**
- [ ] 内存占用上限已评估（大对象、切片扩容）
- [ ] CPU 密集计算有限流或异步处理策略
- [ ] 连接池、goroutine 池有上限设计

**性能基线**
- [ ] 关键接口有可量化的性能目标（ns/op、QPS、延迟 P99）
- [ ] 性能目标对应具体的基准测试用例

### 8.4 合规专项检查清单

**数据合规**
- [ ] 数据分类已明确（业务数据/个人信息/敏感个人信息）
- [ ] 数据保留期限符合法规要求
- [ ] 数据删除/脱敏机制已设计

**隐私保护**
- [ ] 个人信息的收集有最小必要原则声明
- [ ] 跨境数据传输合规性已评估

**审计日志**
- [ ] 日志内容、格式、保留期限符合合规要求
- [ ] 日志不可篡改机制已设计（如适用）

## 九、评审签名规范

### 9.1 签名表格

评审文档尾部必须包含以下签名表：

```markdown
## 评审签名

| 角色 | 签名 | 确认时间 | 确认内容 |
|------|------|---------|---------|
| 评审执行者 | {Agent-2 / 安全专家 Agent / 性能专家 Agent / 合规专家 Agent} | YYYY-MM-DD HH:MM | 已完成强制准备步骤（含代码库探索），所有问题均有原文引用和风险说明，评审结论客观有据 |
| 人类技术负责人 | {人类技术负责人} | YYYY-MM-DD HH:MM | 致命/严重问题已知晓，评审结论符合预期，后续处置方向已确认 |

**被评审文档**：architecture-V{版本号}.md（Commit: {hash}）  
**本评审文档归档路径**：docs/demand/{需求ID}/review/{type}-review-V{版本号}.md  
**下一步操作**：评审通过 → 触发 Agent-3 对齐确认；不通过 → Agent-1 修改后重新提交评审
```

### 9.2 人类技术负责人签名触发条件

人类技术负责人签名在以下任一情形下为**必填**，否则为可选：

- 评审发现**致命**级别问题（任意数量）；
- 评审发现**严重**级别问题（任意数量）；
- 评审结论为「不通过」且退回 Agent-1 修改；
- 技术选型与项目规范存在冲突、需要人类授权豁免；
- 评审总分处于边界值（79-81 分），存在通过/不通过分歧。

### 9.3 签名前自检清单

评审执行者在签名前必须逐项确认：

- [ ] 已绑定被评审文档的版本号和基准 Commit；
- [ ] 已阅读 `docs/ARCHITECTURE.md`、`docs/CONVENTIONS.md`、`docs/TESTING.md` 及需求 PRD；
- [ ] 已完成代码库探索（验证依赖关系、接口冲突、可落地性），非仅文档层面评审；
- [ ] 所有评审问题均包含完整字段：原文引用、风险说明、修改建议、参考依据；
- [ ] 问题级别判定符合第六章分级标准，无主观降级或升级；
- [ ] 评审结论与通过判定规则（第七章）一致，总分计算无误；
- [ ] 评审文档头部基础信息齐全（需求 ID、评审类型、版本号、归档路径）。

### 9.4 签名追溯说明

> 评审签名是设计质量的把关节点。若后续对齐确认或编码阶段发现评审存在重大漏评（如遗漏了显而易见的循环依赖、未识别出接口签名冲突等），将追溯至评审执行者签名责任，要求其对本指南的对应检查清单进行反思迭代。人类技术负责人签名的致命/严重问题处置方向，其决策合理性由人类负责人承担追溯责任。

## 十、重新评审触发规范

### 10.1 何时需要重新评审

以下情况必须触发重新评审，不可沿用上次评审结论：

- 设计文档版本更新（任何修改均触发）；
- 评审结论为「不通过」，Agent-1 修改后提交；
- 人类兜底裁决改变了设计方向。

### 10.2 重新评审的简化处理

若上次评审结论为「不通过」且设计文档仅修改了评审指出的问题：

- 可只针对修改部分进行 Delta 评审；
- Delta 评审文档需引用上次评审文档版本；
- 原有问题已修复的，在问题列表中标注「已修复（Commit: {hash}）」；
- 新引入问题按正常流程处理。

```markdown
**P-001**【严重】~~`MetricStore.GetCurrentUsages` 接口未说明 key 不存在时的行为~~

- **状态**：已修复（Commit: abc1234）
- **验证结果**：接口注释已补充缺失 key 的返回行为，符合预期
```

## 十一、常见评审错误示例

以下是评审中容易出现的无效意见，须避免：

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 主观判断无依据 | 「这个设计不够优雅」 | 引用规范文档中的具体约束条目 |
| 建议模糊无法执行 | 「建议优化存储层」 | 说明具体修改点和预期结果 |
| 超出评审范围 | 安全评审中指出接口命名不规范 | 安全评审只评审安全维度，其他维度留给平级评审 |
| 无原文引用 | 直接写结论，不引用被评审文档原文 | 每条问题必须引用原文，含章节定位 |
| 缺少风险说明 | 「此处有问题，建议修改」 | 必须说明「如不修改会导致什么后果」 |
| 评审意见无法溯源 | 评审文档无版本锚点 | 必须绑定被评审文档的具体版本和 Commit |

## 十二、输出格式与归档要求

- 评审文档以 **Markdown** 格式输出；
- 归档路径：`docs/demand/{需求ID}/review/{type}-review-V{版本号}.md`，其中 `{type}` 为 `peer`、`security`、`performance`、`compliance` 之一；
- 文件名版本号与被评审的设计文档版本保持对应（设计文档 V1.0.0 → 评审文档 V1.0.0）；
- 评审文档提交至代码仓库，commit message 格式：`review({需求ID}): {type} review for architecture-V{版本号}`；
- 评审文档须在签名完成后（即第九章签名表填写完毕）方可提交，未签名的评审文档不得作为后续流程的输入。
