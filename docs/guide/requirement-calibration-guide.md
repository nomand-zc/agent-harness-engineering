# 需求校准指南

**适用场景**：当收到口语化需求、需要输出标准化PRD文档时，遵循本指南。  
**核心目标**：通过系统性的仓库探索与需求校验，确保所有需求完整、清晰、无歧义，且与现有系统无冲突，可直接驱动后续架构设计与代码实现。

**适用 Agent**：Agent-0（需求校准 Agent）。校准过程中发现的、不影响当前需求推进的优化点写入 `docs/demand/{需求ID}/backlog.md`，由 Agent-7（Backlog 整理 Agent）统一汇总排期。

---

## 一、校准前的强制仓库探索步骤

**在执行任何校准动作之前，必须先完成以下仓库探索。** 跳过此步骤将导致校准结论失真——无法判断需求是否已有现成实现、是否与现有功能冲突、是否可从现有代码中复用能力。

### 1.1 阅读项目规范与架构文档

| 文档 | 必读内容 |
|------|---------|
| `AGENTS.md` | 项目整体结构、模块划分、核心文档索引 |
| `docs/overview.md` | 全局架构图、多Agent角色分工、核心思想 |
| `docs/ARCHITECTURE.md` | 系统架构、组件依赖关系、禁止事项 |
| `docs/CONVENTIONS.md` | 命名规范、编码约束、技术栈说明 |
| `docs/ENVIRONMENT.md` | 开发环境、依赖约束、技术能力边界 |

**不阅读架构文档就做校准，等于在不了解系统能力边界的情况下判断需求是否可行。**

### 1.2 探索现有需求与功能实现

探索完成后，必须能回答以下问题：

**现有需求覆盖情况**

- `docs/demand/` 目录下已有哪些需求ID？每个需求的核心功能是什么？
- 当前新需求是否与已有某个需求存在**功能重叠**（完全相同或高度相似）？
- 如果存在重叠，重叠部分是否已经实现？实现文件在哪里？

**功能边界与交集识别**

- 新需求涉及的业务实体（数据结构、接口、服务）在现有代码中是否已存在？
- 现有功能是否已经部分满足新需求？如果是，差距在哪里（扩展 vs. 新增）？
- 新需求是否需要修改现有接口或数据结构？如果是，会影响哪些调用方？

**冲突识别**

- 新需求的业务规则是否与现有已实现功能存在**逻辑冲突**（如同一场景下两个功能的处理结果矛盾）？
- 新需求的非功能指标（性能、安全、兼容性）是否与现有系统约束冲突？
- 新需求是否依赖当前技术栈尚不支持的能力？

**可复用资产识别**

- 现有代码中是否有可直接复用的模块、工具函数、公共组件？
- 现有指南文档（`docs/guide/`）是否已有对应的执行规范可供参考或扩展？

### 1.3 探索动作清单（按序执行）

```
1. 读取 AGENTS.md → 获取项目整体索引，确定探索范围
2. 读取 docs/overview.md → 理解全局架构与模块职责边界
3. 读取 docs/ARCHITECTURE.md + docs/CONVENTIONS.md → 获取技术约束
4. 列出 docs/demand/ 目录 → 获取已有需求ID列表
5. 对每个已有需求ID，读取对应 PRD.md → 提取核心功能摘要
6. 搜索代码库中与新需求关键词相关的实现文件（按业务实体/功能模块搜索）
7. 汇总探索结论：填写"仓库现状快照"（见第二节模板）
```

## 八、校准结论

基于完整性校验、合理性校验和仓库探索结论，给出以下三选一结论：

- **通过**：需求完整、清晰、与现有系统无冲突，可直接进入设计阶段；
- **需补充**：需求主体清晰但存在缺失项或模糊表述，补充后进入设计阶段；
- **需重构**：需求存在重大缺陷（逻辑矛盾 / 与现有功能严重冲突 / 技术不可行），需重新梳理。

## 十二、校准问题跟踪表格式规范

### 12.1 文档定位与双重作用

校准问题跟踪表（`calibration-issues.md`）承担两个核心作用：

1. **驱动PRD迭代**：每轮校准发现的问题驱动下一版PRD的修改，问题闭环后标注解决结论；
2. **模块自我迭代依据**：积累多个需求的跨轮次问题模式，为《需求校准指南》与 `requirements-standardization.md` 的迭代优化提供真实数据。

> **强制约束**：每次更新PRD文档时必须同步更新本跟踪表，追加新一轮的问题记录。**绝不覆盖历史轮次数据**，历史轮次的问题记录须永久保留。
>
> **问题来源全覆盖**：无论是 Agent-0 在校准过程中发现的问题，还是人类在审核 PRD 时提出的意见，均须记录至本跟踪表对应轮次，通过"提出方"字段（`Human` / `Agent-0`）加以区分，不得遗漏任何一方提出的问题。

### 12.3 问题编号规范

- 格式：`R{轮次}-Q{序号}`，如 `R1-Q001`、`R2-Q003`；

- 序号三位补零，从 `001` 起始。

### 12.4 严重程度定义

| 等级 | 含义 | 处置要求 |
|------|------|---------|
| P0 | 必须解决：影响PRD可落地性或存在重大歧义 | 当轮必须闭环，否则不能输出PRD |
| P1 | 建议解决：影响PRD清晰度或完整性 | 当轮优先解决，若无法闭环需在遗留问题中说明 |
| P2 | 记录备案：细节模糊或暂无影响的观察性问题 | 记录即可，不阻断PRD输出 |

### 12.5 跨轮次操作规则

1. **新增轮次**：每次触发新一轮校准时，在文档末尾追加新的 `Round-N` 节，**不修改已有轮次内容**；
2. **更新轮次总览**：每次追加轮次后，同步更新文档顶部的"轮次总览"表格；
3. **闭环更新**：问题闭环时，在对应轮次的问题清单中填写"闭环结论"和"闭环时间"；
4. **全局闭环**：所有问题闭环、人类确认PRD通过后，填写"全局闭环声明"，将文档顶部"跟踪状态"改为"已闭环"。

## 十四、校准签名追溯规范

### 14.1 签名责任边界

需求校准涉及多轮次、多产物，各签名人的责任边界如下：

| 签名人 | 签名产物 | 责任范围 |
|--------|---------|---------|
| Agent-0（校准执行者） | 每轮校准报告 + 最终 PRD + 全局闭环声明 | 仓库探索的完整性、校验维度的覆盖性、问题清单的准确性、校准结论的客观性 |
| 人类需求确认人 | 每轮「通过」结论 + 最终 PRD + 全局闭环声明 | PRD 内容与需求意图的一致性；校准通过结论的业务合理性 |
| 人类产品负责人 | PRD（外部用户产品功能时） | 业务目标、功能范围的产品合理性 |

### 14.2 签名时机

| 产物 | 签名时机 |
|------|---------|
| 每轮校准报告中的「本轮校准签名」 | Agent-0 完成当轮校准后立即签名；人类在校准结论为「通过」时签名 |
| PRD 中的「PRD 签名」 | Agent-0 输出 PRD 草稿后自签；人类在审阅 PRD 内容后签名 |
| 校准问题跟踪表中的「全局闭环签名」 | 所有轮次问题均已闭环、PRD 最终版本确定后，双方签名 |

### 14.3 签名缺失的阻断规则

| 缺失场景 | 阻断效果 |
|---------|---------|
| Agent-0 未签名的校准报告 | 不得提交至代码仓库，不得作为后续工作的输入 |
| 人类未签名的「校准通过」结论 | 不得触发后续架构设计流程（Agent-1 不得启动设计） |
| PRD 缺少需求确认人签名 | 不得作为方案设计的输入基准 |
| 全局闭环声明缺少任一签名 | 需求不得标记为「已通过校准」，版本映射矩阵不得标记为「✅ 完整」 |

### 14.4 签名追溯说明

> 需求校准签名是整个需求→设计→实现链条的质量起点。若后续架构设计或实现阶段发现 PRD 存在重大歧义、遗漏或与需求意图不符，将追溯至对应签名轮次的 Agent-0（仓库探索与校验完整性）和人类需求确认人（需求意图确认）。追溯结论将驱动本指南和相关规范文档的反思迭代。
