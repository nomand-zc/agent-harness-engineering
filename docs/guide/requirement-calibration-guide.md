# 需求校准指南

**适用场景**：当收到口语化需求、需要输出标准化PRD文档时，遵循本指南。  
**核心目标**：通过系统性的仓库探索与需求校验，确保所有需求完整、清晰、无歧义，且与现有系统无冲突，可直接驱动后续架构设计与代码实现。

---

## 一、校准前的强制仓库探索步骤

**在执行任何校准动作之前，必须先完成以下仓库探索。** 跳过此步骤将导致校准结论失真——无法判断需求是否已有现成实现、是否与现有功能冲突、是否可从现有代码中复用能力。

### 1.1 阅读项目规范与架构文档

| 文档 | 必读内容 |
|------|---------|
| `AGENTS.md` | 项目整体结构、模块划分、核心文档索引 |
| `docs/overview.md` | 全局架构图、多Agent角色分工、核心思想 |
| `docs/ARCHITECTURE.md` | 系统架构、组件依赖关系、禁止事项 |
| `docs/CONVENTIONS.md` | 命名规范、编码约束、技术栈说明 |
| `docs/ENVIRONMENT.md` | 开发环境、依赖约束、技术能力边界 |

**不阅读架构文档就做校准，等于在不了解系统能力边界的情况下判断需求是否可行。**

### 1.2 探索现有需求与功能实现

探索完成后，必须能回答以下问题：

**现有需求覆盖情况**

- `docs/demand/` 目录下已有哪些需求ID？每个需求的核心功能是什么？
- 当前新需求是否与已有某个需求存在**功能重叠**（完全相同或高度相似）？
- 如果存在重叠，重叠部分是否已经实现？实现文件在哪里？

**功能边界与交集识别**

- 新需求涉及的业务实体（数据结构、接口、服务）在现有代码中是否已存在？
- 现有功能是否已经部分满足新需求？如果是，差距在哪里（扩展 vs. 新增）？
- 新需求是否需要修改现有接口或数据结构？如果是，会影响哪些调用方？

**冲突识别**

- 新需求的业务规则是否与现有已实现功能存在**逻辑冲突**（如同一场景下两个功能的处理结果矛盾）？
- 新需求的非功能指标（性能、安全、兼容性）是否与现有系统约束冲突？
- 新需求是否依赖当前技术栈尚不支持的能力？

**可复用资产识别**

- 现有代码中是否有可直接复用的模块、工具函数、公共组件？
- 现有指南文档（`docs/guide/`）是否已有对应的执行规范可供参考或扩展？

### 1.3 探索动作清单（按序执行）

```
1. 读取 AGENTS.md → 获取项目整体索引，确定探索范围
2. 读取 docs/overview.md → 理解全局架构与模块职责边界
3. 读取 docs/ARCHITECTURE.md + docs/CONVENTIONS.md → 获取技术约束
4. 列出 docs/demand/ 目录 → 获取已有需求ID列表
5. 对每个已有需求ID，读取对应 PRD.md → 提取核心功能摘要
6. 搜索代码库中与新需求关键词相关的实现文件（按业务实体/功能模块搜索）
7. 汇总探索结论：填写"仓库现状快照"（见第二节模板）
```

## 二、仓库现状快照（校准报告必填章节）

完成第一节的探索后，必须在《需求校准报告》中输出以下快照，作为校准结论的依据：

```markdown
## 仓库现状快照

### 已有需求概览
| 需求ID | 核心功能摘要 | 实现状态 |
|--------|------------|---------|
| REQ-XXXX-XXX | （一句话描述） | 已实现 / 设计中 / 待开发 |

### 与本次新需求的关系分析
| 维度 | 结论 | 说明 |
|------|------|------|
| 功能重叠 | 无 / 部分重叠 / 完全重叠 | （具体说明重叠内容及文件路径） |
| 逻辑冲突 | 无 / 存在冲突 | （具体说明冲突点） |
| 可复用资产 | 无 / 存在 | （具体说明可复用的模块/函数/文档） |
| 技术可行性 | 可行 / 存在风险 | （说明风险点） |

### 探索结论
（一段话总结：新需求在当前仓库中的位置——是全新功能、现有功能扩展、还是与现有功能存在冲突/重复，以及对校准工作的影响。）
```

---

## 三、文档基础信息

每份需求校准报告必须包含以下基础信息：

- **需求ID**：格式 `REQ-年份-序号`（如 `REQ-2026-001`），由人类在确认通过后绑定；
- **需求来源**：提出人姓名或角色；
- **校准时间**：格式 `YYYY-MM-DD`；
- **校准Agent**：`Agent-0`。

---

## 四、原始需求摘要

- 用户原始需求描述（**完整引用，不可摘要**）；
- 业务背景（为什么提出这个需求）；
- 期望目标（需求方希望达成的结果）。

---

## 五、需求完整性校验

检查需求是否涵盖以下要素，缺失项需标注并在"需求澄清问题清单"中列出：

| 要素 | 是否覆盖 | 缺失说明 |
|------|---------|---------|
| 业务目标与成功指标 | ☐ | |
| 用户场景与使用流程 | ☐ | |
| 功能边界（做什么 / 不做什么） | ☐ | |
| 非功能需求（性能、安全、兼容性） | ☐ | |
| 依赖与约束条件 | ☐ | |
| 验收标准雏形 | ☐ | |
| **与现有功能的关系说明**（来自第一节探索结论） | ☐ | |

---

## 六、需求合理性校验

结合第一节的仓库探索结论，检查以下问题：

| 校验项 | 结论 | 说明 |
|--------|------|------|
| 逻辑矛盾（需求内部冲突表述） | 无 / 存在 | |
| 与现有功能逻辑冲突 | 无 / 存在 | （引用快照中冲突点） |
| 功能重复（已有实现可直接满足） | 无 / 存在 | （引用快照中重叠内容） |
| 技术可行性风险 | 无 / 存在 | （引用快照中技术评估） |
| 资源匹配风险（人力/时间/预算） | 无 / 存在 | |
| 优先级冲突（与现有需求规划冲突） | 无 / 存在 | |

---

## 七、需求澄清问题清单

针对完整性校验、合理性校验中发现的模糊、缺失、矛盾点，生成待确认问题列表：

```markdown
| 编号 | 问题描述 | 来源（完整性/合理性/仓库探索） | 期望答复方向 |
|------|---------|--------------------------|------------|
| Q-001 | （问题描述） | （来源） | （期望答复方向） |
```

---

## 八、校准结论

基于完整性校验、合理性校验和仓库探索结论，给出以下三选一结论：

- **通过**：需求完整、清晰、与现有系统无冲突，可直接进入设计阶段；
- **需补充**：需求主体清晰但存在缺失项或模糊表述，补充后进入设计阶段；
- **需重构**：需求存在重大缺陷（逻辑矛盾 / 与现有功能严重冲突 / 技术不可行），需重新梳理。

---

## 九、标准化PRD输出模板

校准结论为"通过"或"需补充（已补充）"后，输出标准化PRD文档到 `docs/demand/{需求ID}/PRD.md`：

```markdown
---
需求ID: REQ-YYYY-NNN
版本号: V1.0.0
生效时间: YYYY-MM-DD
基准Commit: （代码提交后填写）
上版迭代号: 无
---

# 标准化PRD：{需求标题}

## 文档基础信息
- 需求ID：REQ-YYYY-NNN
- 版本号：V1.0.0
- 生效时间：YYYY-MM-DD
- 校准Agent：Agent-0

## 业务背景与目标
（业务背景描述，以及本需求需要达成的业务目标）

## 与现有功能的关系
（基于仓库探索结论：是全新功能、扩展现有功能、还是替换现有功能；
 明确与哪些现有需求ID / 模块存在交集，以及交集的处理策略）

## 用户故事与场景描述
（用 "作为...，我希望...，以便..." 格式描述）

## 功能需求清单
（按模块/优先级组织，每条需求有唯一编号）

## 非功能需求
- 性能：（如"每秒并发≥100次，接口响应时间≤500ms"）
- 安全：（如"验证码有效期10分钟，支持防暴力破解"）
- 兼容性：（如"兼容iOS 14+、Android 10+"）
- 可观测性：（如"接口调用量、错误率须接入监控"）

## 边界定义
（明确不做的内容；明确与现有功能的边界：哪些由现有功能负责、哪些由本需求负责）

## 依赖与约束
（外部依赖、技术约束、时间约束；
 明确依赖哪些现有模块/接口，以及版本要求）

## 验收标准摘要
（核心验收条件，后续由Agent-1细化为完整验收标准）

## 附录
- 仓库探索快照：（引用校准报告中"仓库现状快照"章节的结论）
- 术语表
- 参考文档
```

## 十、校准触发时机

| 触发场景 | 执行要求 |
|---------|---------|
| 新需求接入时 | 必须完整执行本指南全部步骤，包括仓库探索 |
| 需求变更（变更幅度 ≥ 20%） | 重新执行完整校准流程，仓库探索聚焦变更影响范围 |
| 需求变更（变更幅度 < 20%） | 人类直接修改PRD，绑定新版本号，同步更新关联文档 |
| 定期巡检发现偏差 | 触发校准，仓库探索聚焦偏差模块 |

| 校准问题跟踪表 | `docs/demand/{需求ID}/calibration-issues.md` | 待补充/待澄清问题的跟踪与闭环记录 |
