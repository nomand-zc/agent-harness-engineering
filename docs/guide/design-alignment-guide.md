# 设计方案对齐确认文档编写指南

**适用场景**：当需要在编码开始前，对已评审通过的架构设计方案进行对齐确认时，遵循本指南。  
**核心目标**：通过对齐确认，将设计方案中所有模糊点、争议点在编码前明确，确保实现者对方案的理解与设计者一致，避免「代码写完才发现理解偏差」。

**适用 Agent**：Agent-3（高级开发 Agent）。

---

## 一、对齐确认前的强制准备步骤

在执行任何对齐评估之前，必须先完成以下准备，否则对齐确认必然产生遗漏或误判。

### 1.1 确认上游输入就绪

- 确认 `docs/demand/{需求ID}/design/architecture-V{版本号}.md` 已存在，且评审结论为「通过」（无致命/严重遗留问题）；
- 确认 `docs/demand/{需求ID}/acceptance/criteria-V{版本号}.md` 已存在；
- 确认 `docs/demand/{需求ID}/PRD.md` 已存在（用于需求溯源）。

> **强制约束**：在评审未通过的设计方案上执行对齐确认，等于在有缺陷的基础上启动编码，风险极高。

### 1.2 阅读项目规范文档

对齐确认的核心之一是「发现设计方案与项目规范的冲突」，必须先读懂规范再评估方案：

| 文档 | 必读内容 |
|------|---------|
| `docs/ARCHITECTURE.md` | 设计约束、模块职责边界、禁止事项 |
| `docs/CONVENTIONS.md` | 命名规范、接口风格、架构原则 |
| `docs/TESTING.md` | 单元测试规范、mock 策略、覆盖率要求 |

### 1.3 深度探索现有代码库

对齐确认必须基于对现有代码库的真实理解，不可只读设计文档：

**理解项目背景与现有功能**
- 阅读 `README.md` 或项目顶层文档，理解项目核心目标和现有功能全貌；
- 检查 `docs/demand/` 下是否有相关历史需求与设计方案，了解该功能的演进过程；
- 梳理本次需求与现有功能的关联与边界。

**回答以下关键问题（探索结论记录在对齐文档的「背景理解」章节）**
- 需求涉及的代码位于哪些包/模块？现有实现是否有可复用部分？
- 新代码的依赖关系是否存在循环风险？
- 类似功能的已有实现中有哪些 Pattern 值得参考？
- 修改现有接口/类型会影响哪些调用方？

> **强制约束**：跳过代码库探索直接进行对齐确认，等于用文档理解代替实际代码认知，容易漏掉隐藏的实现障碍。

---

## 二、对齐确认文档的必要章节

一份合格的对齐确认文档必须包含以下所有章节，缺少任何一章都可能导致编码时产生歧义或偏差。

### 章节清单

| 章节 | 作用 | 缺失后果 |
|------|------|---------|
| 文档基础信息 | 需求 ID、关联设计文档版本、对齐时间 | 无法溯源，文档孤岛 |
| 背景理解摘要 | 代码库探索结论，建立编码视角 | 对齐结论缺乏实际代码支撑 |
| 待确认疑问清单 | 梳理设计方案中所有模糊点、争议点 | 编码时遇到模糊点无据可查 |
| 逐条答疑记录 | 针对每条疑问给出明确答案，绑定方案原文 | 答疑结论无法追溯，容易反复讨论 |
| 方案对齐评估结果 | 基于《方案对齐评估指南》的打分与结论 | 无法客观判断能否进入编码 |
| 风险提示与改进建议 | 识别潜在风险，提出优化方向 | 编码过程中遭遇可预见的风险 |
| 对齐确认签名 | Agent-3 签名、人类复核人签名、确认时间 | 对齐结论缺乏共识确认 |

---

## 三、文档基础信息规范

文档头部必须包含以下所有字段，字段缺失不可提交：

```markdown
**需求 ID**：REQ-{需求ID}  
**关联设计文档**：docs/demand/{需求ID}/design/architecture-V{版本号}.md  
**关联验收标准**：docs/demand/{需求ID}/acceptance/criteria-V{版本号}.md  
**对齐文档版本**：V{版本号}  
**对齐确认时间**：YYYY-MM-DD  
**执行 Agent**：Agent-3  
**归档路径**：docs/demand/{需求ID}/alignment/alignment-V{版本号}.md  
```

## 四、背景理解摘要规范

背景理解摘要是对代码库探索结论的记录，必须基于真实探索，不可凭文档推断：

```markdown
## 背景理解摘要

- **核心模块位置**：{需求涉及的主要包/模块路径，如 `observer/`, `storage/`}
- **可复用的现有实现**：{现有代码中可直接复用的部分，或「无」}
- **需要修改的现有接口**：{本次需求需要修改的现有接口列表，及其影响的调用方}
- **依赖关系风险**：{是否存在循环依赖风险，如「新模块依赖 storage，storage 已依赖 account，无循环风险」}
- **参考 Pattern**：{现有代码中类似实现的路径，如「参照 health/ 中的 LeaderElector 接口设计」}
- **历史相关需求**：{有/无，若有则注明历史需求 ID 和关联关系}
```

## 五、待确认疑问清单规范

### 5.1 疑问来源

疑问必须来自以下几类真实发现，不可臆造：

| 疑问来源 | 说明 |
|---------|------|
| 设计方案表述模糊 | 方案中存在「视情况而定」「参考现有实现」等模糊描述 |
| 设计与代码库不一致 | 方案中引用的接口/类型与代码库实际定义不符 |
| 边界条件未覆盖 | 方案未说明某种输入或状态下的行为 |
| 依赖关系存疑 | 方案中的模块依赖在代码库中可能产生循环 |
| 验收标准与方案不一致 | 验收标准描述的行为与方案设计存在矛盾 |
| 实现路径不明确 | 方案说明了「做什么」，但没有说明「在哪里做」 |

### 5.2 每条疑问的格式

```markdown
**Q-{序号}**【{疑问类型}】{疑问标题（一句话）}

- **方案原文**：> {设计文档中的原文，含章节定位}
- **疑问描述**：{具体描述什么地方不清楚，或与代码库有何冲突}
- **影响范围**：{此疑问若无法明确，会导致哪些实现决策无法做出}
- **待确认方**：Agent-1 / 人类技术负责人
```

### 5.3 疑问列表示例

**错误写法（泛泛而问，无法得到有效答案）**：
> Q-1：接口设计是否合理？

**正确写法（具体可答，有影响范围说明）**：
> **Q-001**【边界条件未覆盖】`MetricStore.GetCurrentUsages` 在 keys 为空切片时的行为未定义
>
> - **方案原文**：> 第五章「接口定义」- `GetCurrentUsages(ctx, keys []string) ([]*Metrics, error)`
> - **疑问描述**：方案未说明当 keys 为空切片（`[]string{}`）时应返回空切片还是 error。代码库中类似的 `storage.GetBatch` 在空输入时返回空切片，但方案未声明与此对齐
> - **影响范围**：`CollectMetrics` 在无账号场景下会调用此接口，若行为未定义，实现者可能做出不同选择导致上层 panic
> - **待确认方**：Agent-1

## 六、逐条答疑记录规范

每条疑问必须有对应的答疑记录，答疑结论必须绑定设计方案原文或明确的决策来源：

```markdown
**A-{序号}**（对应 Q-{序号}）

- **答疑结论**：{明确的、可直接指导编码的结论，不能有「视情况」「由实现者决定」等模糊表述}
- **决策依据**：{来自设计文档哪个章节 / 人类裁决 / 与 Agent-1 确认}
- **设计文档是否需要更新**：是 / 否
  - 若「是」：由 Agent-1 在设计文档中补充对应内容，补充后标注「已更新（Commit: {hash}）」
- **确认方**：Agent-1 / 人类技术负责人 / Agent-3 自行判断（需说明判断依据）
```

> **强制约束**：答疑结论不可由 Agent-3 单方面自行决定并记录为「已确认」，必须有 Agent-1 或人类的明确确认。若无法获得确认，须升级为风险项，由人类裁决。

## 七、方案对齐评估结果规范

对齐确认文档必须包含基于《方案对齐评估指南》的完整评估结果：

```markdown
## 方案对齐评估结果

详细评估过程见《方案对齐评估打分表》（同目录）。

| 评估维度 | 权重 | 得分（满分100） | 加权得分 |
|---------|------|--------------|---------|
| 需求一致性 | 25% | {分数} | {分数×0.25} |
| 技术可行性 | 20% | {分数} | {分数×0.20} |
| 架构合理性 | 20% | {分数} | {分数×0.20} |
| 实现完整性 | 15% | {分数} | {分数×0.15} |
| 风险预案充分性 | 10% | {分数} | {分数×0.10} |
| 文档规范性 | 10% | {分数} | {分数×0.10} |
| **综合得分** | 100% | - | **{总分}** |

**评估结论**：通过（≥80分，无单项<70分）/ 不通过（{原因}）

**不通过时的阻塞项**（仅不通过时填写）：
- 维度：{维度名}，得分：{分数}，阻塞原因：{一句话说明}
- 需要 Agent-1 补充/修改：{具体内容}
```

## 八、风险提示与改进建议规范

对齐确认过程中发现的潜在风险必须在此章节明确记录，不可遗漏：

```markdown
## 风险提示与改进建议

| 风险编号 | 风险描述 | 影响等级 | 应对措施 | 责任方 |
|---------|---------|---------|---------|--------|
| RISK-001 | {风险描述} | 高/中/低 | {具体应对措施} | Agent-3 / Agent-1 / 人类 |

### 改进建议（不影响编码启动，但值得记录）
- {建议1}：{具体说明，标注「不影响本次实现，纳入后续优化」}
```

风险等级判定标准：

| 等级 | 判定标准 | 处置要求 |
|------|---------|---------|
| 高 | 可能导致编码失败或上线故障 | 必须在编码前明确应对措施，必要时升级人类介入 |
| 中 | 可能影响代码质量或维护性 | 记录在案，编码时重点关注 |
| 低 | 对本次实现影响较小 | 记录在案，纳入后续优化 |

## 九、对齐确认签名规范

```markdown
## 对齐确认签名

| 角色 | 签名 | 确认时间 | 确认内容 |
|------|------|---------|---------|
| 执行 Agent | Agent-3 | YYYY-MM-DD HH:MM | 疑问清单已完整梳理，答疑结论已确认，评估得分≥80分 |
| 设计方 | Agent-1 | YYYY-MM-DD HH:MM | 所有疑问已答复，设计文档已同步更新（若需要） |
| 人类复核 | {人类技术负责人} | YYYY-MM-DD HH:MM | 高风险项已知晓，对齐结论符合预期 |

```

## 十、对齐结论与后续流程

### 10.1 对齐通过的条件

满足以下**全部**条件，方可给出「通过」结论，进入编码阶段：

- [ ] 所有待确认疑问均已得到明确答复（无「待确认」状态的疑问）；
- [ ] 方案对齐评估综合得分 ≥ 80 分；
- [ ] 无单项评估维度得分 < 70 分；
- [ ] 所有高风险项均已制定应对措施；
- [ ] 需要更新的设计文档已由 Agent-1 补充完成。

### 10.2 对齐不通过的处置

| 不通过原因 | 处置方式 |
|-----------|---------|
| 评分不通过（<80分 或单项<70分） | 退回 Agent-1 补充设计文档，补充后重新执行对齐确认 |
| 疑问清单中存在无法答复的问题 | 升级人类介入，人类裁决后记录结论，重新执行对齐确认 |
| 发现高风险项无应对措施 | 标注风险，要求 Agent-1 或人类明确应对措施后方可编码 |
| 设计文档与代码库存在冲突 | 退回 Agent-1 修正设计文档，修正后重新评审和对齐确认 |

---

## 十一、输出格式与归档要求

- 对齐确认文档以 **Markdown** 格式输出；
- 归档路径：`docs/demand/{需求ID}/alignment/alignment-V{版本号}.md`；
- 版本号与关联的架构设计方案版本保持一致；
- 文档提交至代码仓库，commit message 格式：`docs({需求ID}): add design alignment V{版本号}`；
- 对齐确认文档变更时（如答疑补充），版本号递增，旧版本归档保留。
